### Q1. 什么是 Agent 的"幻觉-行动放大"问题？请举例说明并给出防御方案。

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团
**答：**

这是 2025-2026 年 Agent 面试的新热点。与纯对话场景的幻觉不同，Agent 具备工具调用能力，幻觉会导致**实际的系统操作**，危害远大于文本幻觉。

**什么是幻觉-行动放大？**

当 Agent 基于幻觉做出错误推理后，会通过工具调用将错误"执行"到真实系统中。纯 LLM 幻觉只产生错误文本，Agent 幻觉可能删除文件、发送错误订单、修改数据库。

**典型案例：**

```
用户："帮我查询订单 #12345 的物流状态"
Agent 幻觉：编造了一个不存在的物流单号 "SF9999999999"
错误行动：调用退款 API，将该"单号"标记为"物流异常并退款"
结果：用户收到一笔莫名其妙的退款，公司资金损失
```

**四层防御体系：**

**第 1 层：事实锚定（Grounding）**
```python
# 所有工具调用前，强制校验输入参数是否来自可信来源
def validate_tool_input(tool_name, params):
    for key, value in params.items():
        if not is_from_user_input(value) and not is_from_db(value):
            raise HallucinationError(f"参数 {key}={value} 来源不可信，疑似幻觉")
```

**第 2 层：操作分级审批**

| 操作类型 | 审批策略 | 示例 |
|---------|---------|------|
| 只读查询 | 自动执行 | 查询订单状态 |
| 写入操作 | 需用户确认 | 修改地址、取消订单 |
| 资金操作 | 人工审核 | 退款、转账 |
| 删除操作 | 二次确认 + 日志 | 删除数据 |

**第 3 层：输出校验**
- 工具返回结果与 Agent 推理是否一致
- 关键数值范围检查（如退款金额不能超过订单金额）
- 身份校验（操作对象必须是当前用户的资源）

**第 4 层：事后审计 + 回滚**
- 所有工具调用记录完整的 trace
- 高风险操作支持一键回滚
- 异常模式自动告警

### Q2. 请设计一个智能客服 Agent 系统，要求支持多轮对话、敏感操作审批和自动转人工。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 美团、阿里、京东
**答：**

这是美团、阿里的高频真题，考察的是 Context Engineering 思维和系统设计能力。

**整体架构：**

```

                                ├→ 酒店 Agent（预订、改签）
                                └→ 通用 Agent（闲聊、FAQ）
                                    ↓
                            主 Agent 统一输出
```

**核心设计要点：**

**1. 分层容忍度设计（关键！）**

RISK_LEVELS = {

def handle_action(action_type, params):

class ConversationState:

**3. 自动转人工策略**

### Q4. LangGraph 在多 Agent 协作中扮演什么角色？请对比 LangGraph 与其他多 Agent 框架。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、LangChain 生态公司
**答：**

LangGraph 是 LangChain 团队推出的低级编排框架，专为构建有状态、多步骤的 Agent 工作流设计。

**LangGraph 核心特性：**

```python
from langgraph.graph import StateGraph, END

# 定义状态
class AgentState(TypedDict):
    messages: list
    current_agent: str
    task_status: str

# 构建图
workflow = StateGraph(AgentState)

# 添加节点（每个节点是一个 Agent 或处理步骤）
workflow.add_node("researcher", research_agent)
workflow.add_node("coder", code_agent)
workflow.add_node("reviewer", review_agent)

# 添加边（定义流转逻辑）
workflow.add_edge("researcher", "coder")
workflow.add_conditional_edges(
    "coder",
    quality_check,  # 条件路由函数
    {"pass": "reviewer", "fail": "coder"}  # 通过则审核，不通过则重写
)
workflow.add_edge("reviewer", END)

# 设置入口
workflow.set_entry_point("researcher")

# 编译运行
app = workflow.compile()
result = app.invoke({"messages": [user_query]})
```

**与其他框架对比：**

| 维度 | LangGraph | CrewAI | AutoGen | OpenClaw |
|------|-----------|--------|---------|----------|
| 定位 | 低级编排框架 | 多角色协作 | 对话式多 Agent | 本地优先 Agent |
| 状态管理 | 内置图状态机 | 隐式 | 基于对话历史 | 文件系统 |
| 循环支持 | ✅ 原生支持 | ❌ 线性为主 | ✅ | ✅ |
| 可视化 | LangGraph Studio | ❌ | ❌ | ❌ |
| 适合场景 | 复杂工作流 | 团队式任务 | 研究探索 | 个人助手 |
| 生产就绪 | 高（LangChain 生态） | 中 | 低 | 中 |

**LangGraph 的优势：**
- 图结构天然支持循环、分支、并行
- 内置 checkpoint 机制，支持长时间运行任务的断点恢复
- 与 LangSmith 集成，可观测性好
- 支持 human-in-the-loop 审批节点

**LangGraph 的局限：**
- 学习曲线较陡，需要理解图编程范式
- 简单场景用它有点杀鸡用牛刀
- 与 LangChain 生态绑定较深

### Q5. 你的 RAG 系统需要处理多种类型的文档（PDF、Markdown、代码、表格），如何设计自适应的 Chunking 策略？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐⭐ | 阿里云、腾讯、字节跳动
**答：**

2026 年多项 benchmark（FloTorch、NVIDIA、Vectara NAACL）表明，Chunking 策略对 RAG 质量的影响**不亚于 Embedding 模型的选择**。

**核心发现：**
- FloTorch 2026 benchmark：Recursive 512 tokens 得分 69%，Semantic 仅 54%（因为平均 chunk 仅 43 tokens）
- NVIDIA benchmark：金融文档用 1024 tokens 效果最好
- Chroma：Semantic chunking 召回率 91.9%，但端到端准确率差

**自适应 Chunking 方案：**

```python
from langchain_text_splitters import (
    RecursiveCharacterTextSplitter,
    MarkdownHeaderTextSplitter,
    PythonCodeTextSplitter,
)

def adaptive_chunk(file_path: str, content: str) -> list[str]:
    file_type = detect_file_type(file_path)
    
    if file_type == "markdown":
        # Markdown：先按标题拆分，再在每个 section 内递归拆分
        header_splitter = MarkdownHeaderTextSplitter(
            headers_to_split_on=[("#", "h1"), ("##", "h2"), ("###", "h3")]
        )
        sections = header_splitter.split_text(content)
        recursive = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50)
        return [chunk for section in sections 
                for chunk in recursive.split_text(section.page_content)]
    
    elif file_type == "code":
        # 代码：按函数/类拆分
        return PythonCodeTextSplitter(chunk_size=1024).split_text(content)
    
    elif file_type == "pdf_financial":
        # 金融 PDF：按页拆分，保留表格完整性
        return page_level_split(content, min_chunk_size=512)
    
    elif file_type == "table":
        # 表格：整表作为一个 chunk，附加表头描述
        return [format_table_with_header(content)]
    
    else:
        # 通用：递归字符拆分（benchmark 验证的最优默认策略）
        return RecursiveCharacterTextSplitter(
            chunk_size=512,
            chunk_overlap=50,  # 10% overlap
            separators=["\n\n", "\n", " ", ""]
        ).split_text(content)
```

**各文档类型的最优策略：**

| 文档类型 | 推荐策略 | Chunk Size | 关键考量 |
|---------|---------|------------|---------|
| 通用文本 | Recursive | 512 tokens | benchmark 最优默认值 |
| Markdown/HTML | Header-based + Recursive | 512/section | 保留文档结构 |
| 金融 PDF | Page-level | 1024 tokens | 保持表格完整性 |
| 代码仓库 | Language-aware | 函数/类级别 | 不切断函数逻辑 |
| 短文档/FAQ | 不拆分 | 全文档 | 拆分反而损害质量 |
| 多主题文档 | Semantic（设 min_size） | 256-512 | 必须设 min_chunk_size≥200 |

**关键提醒：Semantic Chunking 的陷阱**

```python
# ❌ 错误：不设最小 chunk size
chunker = SemanticChunker(similarity_threshold=0.5)
# 平均 chunk 仅 43 tokens，端到端准确率 54%

# ✅ 正确：必须设 min_chunk_size
chunker = SemanticChunker(
    similarity_threshold=0.5,
    min_chunk_size=200,  # 防止碎片化
    chunk_size=512
)
```

### Q6. Agent 系统中如何实现有效的上下文管理？当对话超出 Context Window 时怎么处理？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 百度、小红书、快手
**答：**

上下文管理是 Agent 工程化的核心挑战。Context Window 有限，但 Agent 需要积累大量历史信息才能做好决策。

**四种上下文溢出处理策略：**

**策略 1：滑动窗口 + 摘要压缩**
```python
class SlidingWindowMemory:
    def __init__(self, max_tokens=8000):
        self.max_tokens = max_tokens
        self.messages = []
        self.summary = ""
    
    def add_message(self, msg):
        self.messages.append(msg)
        if self.count_tokens() > self.max_tokens:
            # 将旧消息压缩为摘要
            old_messages = self.messages[:len(self.messages)//2]
            self.summary = llm_summarize(self.summary + "\n" + format_messages(old_messages))
            self.messages = self.messages[len(self.messages)//2:]
    
    def get_context(self):
        return f"历史摘要：{self.summary}\n\n最近对话：{format_messages(self.messages)}"
```

**策略 2：分层记忆架构**

| 记忆层 | 内容 | 加载时机 | 存储方式 |
|--------|------|---------|---------|
| 工作记忆 | 当前任务上下文 | 始终在窗口内 | 内存 |
| 短期记忆 | 本次会话历史 | 按需加载最近 N 轮 | 内存/Redis |
| 长期记忆 | 用户偏好、关键事实 | 按语义检索加载 | 向量数据库 |
| 情景记忆 | 过往任务经验 | 相似任务触发 | 结构化存储 |

**策略 3：RAG 式记忆检索**
```python
def retrieve_relevant_memory(query: str, top_k: int = 5) -> str:
    # 将历史对话存入向量数据库
    query_embedding = embed(query)
    relevant_chunks = vector_db.search(query_embedding, top_k=top_k)
    return "\n".join([chunk.text for chunk in relevant_chunks])
```

**策略 4：关键信息提取（Context Engineering）**
```python
def extract_key_info(conversation_history: list) -> dict:
    """用 LLM 从对话历史中提取结构化关键信息"""
    prompt = """从以下对话中提取关键信息，输出 JSON：
    - user_intent: 用户最终目标
    - collected_info: 已收集的信息
    - pending_tasks: 待完成的任务
    - decisions_made: 已做出的决定
    """
    return llm_extract(prompt, conversation_history)
```

**生产环境最佳实践：**
- 优先保留最近 3 轮对话 + 关键实体信息
- 用结构化提取替代简单截断
- 长期记忆用向量检索，不全量塞入 prompt
- 定期清理过期的短期记忆

### Q7. 如何设计 Agent 的 A/B 测试和灰度发布方案？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、快手、拼多多
**答：**

Agent 系统的 A/B 测试比传统 Web 应用更复杂，因为 LLM 的非确定性意味着同样的输入可能产生不同输出。

**Agent A/B 测试的特殊挑战：**

| 挑战 | 传统 A/B | Agent A/B |
|------|---------|-----------|
| 输出确定性 | 同输入 → 同输出 | 同输入 → 可能不同输出 |
| 评估指标 | 点击率、转化率 | 任务完成率、用户满意度 |
| 交互深度 | 单次请求-响应 | 多轮对话、多步骤执行 |
| 影响范围 | UI 变化 | 可能影响实际操作 |

**灰度发布方案：**

```python
class AgentRouter:
    def __init__(self):
        self.experiments = {
            "prompt_v2": {
                "traffic": 0.1,  # 10% 流量
                "config": {"system_prompt": "v2_prompt.txt", "model": "gpt-4o"}
            },
            "tool_routing_v2": {
                "traffic": 0.05,  # 5% 流量
                "config": {"tool_selector": "embedding_based"}
            }
        }
    
    def route(self, user_id: str, request: AgentRequest) -> AgentConfig:
        # 基于 user_id 的确定性哈希，保证同一用户始终进入同一实验组
        hash_val = hashlib.md5(user_id.encode()).hexdigest()
        hash_fraction = int(hash_val[:8], 16) / 0xFFFFFFFF
        
        cumulative = 0
        for exp_name, exp_config in self.experiments.items():
            cumulative += exp_config["traffic"]
            if hash_fraction < cumulative:
                return exp_config["config"]
        
        return self.default_config  # 默认配置
```

**评估指标设计：**

```python
eval_metrics = {
    # 自动化指标
    "task_completion_rate": "Agent 是否成功完成用户任务",
    "avg_turns": "平均对话轮数（越少越好）",
    "tool_call_accuracy": "工具调用准确率",
    "hallucination_rate": "幻觉检测率",
    "avg_latency_ms": "平均响应延迟",
    "avg_cost_per_session": "每次会话平均成本",
    
    # 人工评估指标
    "user_satisfaction": "用户满意度评分（1-5）",
    "response_quality": "回答质量评分",
    "safety_score": "安全性评分",
}
```

**发布流程：**
1. **Shadow Mode**：新版本并行运行，不影响用户，对比输出质量
2. **Canary 1%**：小流量验证，监控异常指标
3. **灰度 10%**：扩大流量，收集统计显著性数据
4. **全量发布**：确认无问题后全量切换

**关键：同一用户的多次请求要进入同一实验组**，避免用户体验不一致。

### Q8. 请解释 Agent 中的 "Lost in the Middle" 问题，并说明如何缓解。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 腾讯、微软、Anthropic 生态公司
**答：**

"Lost in the Middle" 是 Stanford 和 UC Berkeley 在 2023 年提出的经典发现，指 LLM 在处理长上下文时，对中间位置的信息关注度显著低于开头和结尾。

**问题本质：**

```
[开头信息] ← 关注度高 ✓
[中间信息 1] ← 容易被忽略 ✗
[中间信息 2] ← 容易被忽略 ✗
[中间信息 3] ← 容易被忽略 ✗
[中间信息 N] ← 容易被忽略 ✗
[结尾信息] ← 关注度高 ✓
```

**对 Agent 系统的影响：**
- RAG 检索了 10 个 chunk，真正有用的在第 5 个，被 LLM 忽略
- 多轮对话中，中间轮次的关键信息被遗忘
- 工具返回的长结果中，关键数据在中间位置

**缓解方案：**

**1. 重排序（Reranking）——把重要信息放前面**
```python
# 检索后用 cross-encoder 重排序
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-v2-m3")
query_doc_pairs = [(query, doc.text) for doc in retrieved_docs]
scores = reranker.predict(query_doc_pairs)
# 按分数降序排列，最重要的放最前面
ranked_docs = sorted(zip(retrieved_docs, scores), key=lambda x: x[1], reverse=True)
```

**2. 减少 context 长度**
- 只传最相关的 top-3 而不是 top-10
- 对每个 chunk 做摘要压缩
- 用 map-reduce 策略：先分别处理每个 chunk，再汇总

**3. 结构化组织 context**
```xml
<context>
<most_relevant>
<!-- 最重要的信息放在开头 -->
{top_1_chunk}
</most_relevant>
<supporting>
{top_2_chunk}
{top_3_chunk}
</supporting>
<reference>
<!-- 其他信息放在最后 -->
{other_chunks}
</reference>
</context>
```

**4. 分步处理**
- 不要一次性把所有 context 塞给 LLM
- 先用 LLM 筛选相关 chunk，再基于筛选结果生成答案
- 两步走：Retrieve → Filter → Generate

**面试加分点**：提到"这个问题在 128K+ 长上下文模型中依然存在，上下文越长越严重，所以 RAG 不会被长上下文模型完全替代"。

### Q9. 你的 Agent 系统上线后发现 LLM 输出格式不稳定（有时 JSON、有时纯文本），如何工程化解决？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、阿里、字节跳动
**答：**

输出格式不稳定是 Agent 系统最常见的工程问题之一。Function Calling 依赖严格的 JSON 格式，一旦 LLM 输出不合规，整个调用链就断了。

**问题根因：**
- LLM 本质是概率模型，不保证输出格式
- 不同模型的指令遵循能力差异大
- temperature > 0 时，格式不稳定概率更高
- 复杂 prompt 中，格式指令可能被"忘记"

**五层解决方案（从轻到重）：**

**第 1 层：Prompt 强约束**
```python
system_prompt = """你必须严格按照以下 JSON Schema 输出，不要输出任何其他内容：
{
  "action": "tool_name",
  "parameters": {"key": "value"}
}
注意：
- 必须以 { 开头，以 } 结尾
- 不要包含 markdown 代码块标记
- 不要包含解释文字
"""
```

**第 2 层：结构化输出 API（推荐）**
```python
# OpenAI Structured Outputs（强制 JSON Schema）
response = client.chat.completions.create(
    model="gpt-4o",
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "tool_call",
            "schema": {
                "type": "object",
                "properties": {
                    "action": {"type": "string"},
                    "parameters": {"type": "object"}
                },
                "required": ["action", "parameters"]
            }
        }
    },
    messages=messages
)
# 100% 保证输出符合 schema
```

**第 3 层：输出解析 + 重试**
```python
import json
import re

def parse_llm_output(raw_output: str, max_retries: int = 3) -> dict:
    for attempt in range(max_retries):
        try:
            # 尝试直接解析
            return json.loads(raw_output)
        except json.JSONDecodeError:
            # 尝试提取 JSON 块
            json_match = re.search(r'\{.*\}', raw_output, re.DOTALL)
            if json_match:
                try:
                    return json.loads(json_match.group())
                except json.JSONDecodeError:
                    pass
            
            # 重试，加入更强的格式提示
            raw_output = llm_call(
                f"以下内容不是合法 JSON，请修正后只输出 JSON：\n{raw_output}"
            )
    
    raise FormatError(f"无法解析 LLM 输出为 JSON：{raw_output[:200]}")
```

**第 4 层：Constrained Decoding（受限解码）**
- 使用 Outlines / Guidance 等库，在 token 生成层面强制输出格式
- 保证 100% 格式合规，但可能影响生成质量

**第 5 层：模型选择**
- 格式要求严格的场景用 Claude（指令遵循能力强）
- 简单格式任务用小模型 + Structured Outputs
- 关键路径用 Function Calling（模型原生支持）

### Q10. 如何为 Agent 系统设计有效的回归测试（Regression Testing）方案？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 微软、美团、快手
**答：**

Agent 系统的回归测试比传统软件更难，因为 LLM 的非确定性——同样的输入可能产生不同但都正确的输出。

**核心挑战：**
- LLM 输出不确定，不能用 exact match 判断
- 评估需要理解语义，不是简单的字符串比较
- 工具调用链路长，端到端测试复杂

**三层回归测试体系：**

**第 1 层：Golden Dataset + LLM-as-Judge**
```python
# 维护标准测试集
golden_dataset = [
    {
        "query": "帮我查一下订单 #12345 的物流状态",
        "expected_behavior": "调用物流查询工具，返回订单状态",
        "expected_tool": "query_logistics",
        "expected_params": {"order_id": "12345"},
        "forbidden_actions": ["refund", "cancel_order"],
    },
    # ... 200-500 条
]

def run_regression_test(dataset):
    results = []
    for case in dataset:
        agent_output = agent.run(case["query"])
        
        # 工具调用正确性检查（确定性）
        tool_check = verify_tool_calls(agent_output, case["expected_tool"], case["expected_params"])
        
        # 输出质量检查（用 LLM 判断）
        quality_check = llm_judge(
            query=case["query"],
            response=agent_output.response,
            criteria=case.get("quality_criteria", "准确、有帮助、安全")
        )
        
        results.append({"case": case, "tool": tool_check, "quality": quality_check})
    
    return generate_report(results)
```

**第 2 层：关键行为断言**
```python
def test_no_hallucination():
    """确保 Agent 不会编造不存在的工具"""
    result = agent.run("帮我转账 100 元")
    assert "transfer_money" not in [call.tool for call in result.tool_calls], \
        "Agent 不应调用不存在的转账工具"

def test_safety_guard():
    """确保高风险操作需要用户确认"""
    result = agent.run("删除所有订单记录")
    assert result.requires_confirmation == True, \
        "删除操作应触发用户确认"

def test_cost_bound():
    """确保单次调用成本在预算内"""
    result = agent.run("分析过去一年的销售趋势")
    assert result.token_usage < 10000, \
        f"Token 消耗 {result.token_usage} 超出预期"
```

**第 3 层：端到端 Trace 对比**
```python
def test_agent_trace():
    """验证 Agent 的执行路径符合预期"""
    result = agent.run("查询北京天气并推荐穿搭")
    
    trace = result.execution_trace
    assert trace[0].type == "intent_classification"
    assert trace[1].type == "tool_call" and trace[1].tool == "weather_api"
    assert trace[2].type == "llm_generation"
    # 不检查具体内容，只检查执行路径
```

**测试频率：**
- 每次 Prompt 修改后 → 跑 full regression
- 每次模型升级后 → 跑 full regression
- 每次工具变更后 → 跑 affected test cases
- 每日定时 → 跑 smoke test（核心场景子集）

**面试加分点**：提到"Golden Dataset 要定期更新，因为用户 query 的分布会随时间变化，旧的测试用例可能不再有代表性"。
