### Q1. 在 ReAct 框架中，Agent 陷入「幻觉循环」的根本原因是什么？有哪些经过验证的设计模式可以防止这种循环？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、Google、微软
**答：**

CoPrep 2026 面试指南中特别指出，「防止 Agent 在 ReAct 循环中陷入幻觉循环」已成为区分候选人的关键问题。

**根本原因：** 幻觉循环发生于 Agent 在没有足够的环境反馈（ground truth）支撑判断时，持续生成不存在的工具返回值或自我验证步骤。典型流程：Agent 调用一个不存在的工具 → LLM 「虚构」返回结果 → 基于虚构结果继续推理 → 进入死循环。

**经过验证的实战策略：**

**1. 严格的工具 Schema 拦截：** Agent 框架必须对 LLM 生成的工具调用做严格的 Schema 验证。如果 LLM 试图调用不存在的工具（如 `order_food()`）或传入非法参数，框架应在执行前拒绝。参考 Anthropic 的 Poka-yoke 原则——修改工具参数设计使错误更难发生。

**2. 观测质量是关键：** CoPrep 强调「Agent 的好坏取决于其工具返回的反馈质量」。返回 `API_ERROR` 的含糊错误信息对 LLM 毫无帮助；而 `ERROR: Invalid parameter 'location'. Must be latitude/longitude` 这样具体的错误信息能让 LLM 自我纠正。

**3. 循环检测器（Loop Detector）：** 在 ReAct 主循环外，增加一个独立的 Monitor 模块，持续监控「Agent 是否在重复相同的 Thought-Action-Observation 序列」。如果检测到 3 次以上重复模式，自动中断并强制 Planner 重新生成新计划。

**4. 最大迭代硬限制：** 设置绝对上限（如 15-20 步），超过后自动进入「安全模式」——Agent 必须向人类用户汇报当前状态并请求指导。

**5. 结果验证层：** 在每个工具返回后增加「验证节点」——检查返回结果是否与预期 Schema 一致。如果返回的是虚构数据，验证层触发回退。

**面试加分：** 「最容易被忽视的一点是：Agent 的幻觉循环往往不是 LLM 的问题，而是**工具设计问题**。如果工具的返回值格式不稳定或者错误信息太模糊，LLM 就会『编造』缺失的信息。好的工具设计应该像 REST API 一样有稳定的返回值 Schema 和详细的错误码，这才是从源头切断幻觉循环的方法。」

### Q2. Agentic RAG、Adaptive RAG、Corrective RAG（CRAG）三种高级 RAG 模式各有什么特点？在什么场景下应该选择哪一种？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | DataCamp、所有 RAG 团队
**答：**

DataCamp 2026 年 RAG 面试指南对这几种模式做了深入对比，这是今年面试中极具区分度的问题。

**Agentic RAG：** 引入检索代理（Retrieval Agents）——LLM 自主决定是否需要检索、从哪个数据源检索、以及检索到什么程度才足够。特点：Agent 拥有自主判断权，可以在推理过程中多次、有策略地检索。适合：需要灵活应对不同复杂度的查询场景。

**Adaptive RAG：** 系统根据查询的复杂度动态调整检索策略。它可以选择不检索（直接回答简单问题）、单次 RAG（常规查询）或迭代 RAG（复杂多跳查询）。特点：通过查询分类器（Query Classifier）提前判断难度级别。适合：查询类型多样、需要响应速度的场景。

**Corrective RAG（CRAG）：** 在检索后增加一个「相关性评估」步骤——LLM 判断检索到的文档是否与查询相关。仅将判定为「相关」的文档送入生成器；如果判定为「不相关」，触发纠正机制（重新检索或调用其他源）。适合：对输出准确性要求极高的场景，如医疗、金融报告。

**模式选择决策树：**
- 查询类型高度可预测且简单 → 常规 RAG 就够
- 查询类型多样，但每个都不复杂 → Adaptive RAG
- 查询边界不明确，需要 Agent 自主探索 → Agentic RAG
- 数据噪声大，检索结果中常混入不相关内容 → CRAG
- 最贵的场景 → Agentic RAG + CRAG 组合：Agent 自主检索 + 每步纠正验证

**面试加分：** 「2026 年还有一个趋势叫 **Self-RAG**——不仅评估检索到的文档，还对模型自身的生成结果做质量评估，确保输出和检索结果的一致性。Self-RAG 是最能体现『Agentic』本质的 RAG 变体，因为它在检索和生成两个环节都保持了自我反思能力。」

### Q3. 在 Multi-Agent 系统中，Agent 之间的通信协议有哪些设计模式？各自的优缺点是什么？

**类型：** 系统设计
**考察频率：** ⭐⭐⭐⭐ | LangChain、CallSphere、Anthropic
**答：**

LangChain 文档和 CallSphere 2026 面试指南都指出，Agent 间的通信机制设计是 Multi-Agent 系统中最关键也最容易踩坑的地方。

**三种主流通信模式：**

**模式 1 — 共享黑板（Shared Blackboard）：**
所有 Agent 在一个共享状态空间中读写。实现简单，Agent 之间无需直接通信。缺点：状态冲突难以控制——两个 Agent 可能同时写入冲突数据；「信息噪声」污染——一个 Agent 的大量中间输出可能干扰其他 Agent 的推理。
适合：协作写文档、迭代优化的场景（如 Evaluator-Optimizer 模式）。

**模式 2 — 消息传递（Message Passing）：**
Agent 之间通过结构化消息直接通信。消息有明确的 sender、receiver、payload schema。优点：通信路径明确，容易追溯和审计。缺点：系统复杂度随 Agent 数量指数级增长。
适合：需要严格权限控制的企业环境、每个 Agent 有独立职责的场景。

**模式 3 — 层级通信（Hierarchical）：**
所有通信经过一个中央 Head Agent（Orchestrator）中转。Orchestrator 接收 Worker Agent 的输出，整理后再传递给下一个。优点：控制流程清晰、信息传递可控。缺点：Orchestrator 成为瓶颈——信息丢失或误解读的风险集中在中央节点。
适合：Anthropic 的 Orchestrator-Workers 模式——Worker 不需要互相理解，只需与 Orchestrator 通信。

**2026 年生产实践：** 大多数生产级 Multi-Agent 系统采用**混合模式**——Orchestrator 作为主通信中枢（层级模式），但允许 Worker 在 Orchestrator 授权下通过消息传递机制直接交换数据。这种方式既保留了控制流程的清晰度，也避免了信息通过中间层时被过度简化或丢失。

**面试加分：** 「另一个容易被忽略的设计决策是：**Agent 之间传递什么信息**。最常见的错误是把 Agent A 的完整对话历史传递给 Agent B——这不仅浪费 Token，还会引入无关噪声干扰 Agent B 的判断。最佳实践是让 Orchestrator 对每个 Worker 的输出做摘要后再传递，只保留当前子任务真正相关的上下文。」

### Q4. Agent 如何决定何时停止执行一个任务？请系统性地说明 Agent 停止策略的四种核心模式。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | novelvista、CallSphere、所有 Agent 工程师
**答：**

novelvista 2026 面试指南中特别指出，停止策略（Stopping Strategy）是 Agent 设计中最容易被忽视但生产中最关键的环节。一个没有良好停止策略的 Agent 可能进入死循环、浪费大量 Token、或在不恰当的时候过早停止。

**四种核心停止策略：**

**1. 目标完成检查（Goal Completion Check）：** Agent 持续检查任务目标是否已满足。例如，当 `booking_confirmed = true` 时停止。这是最直观的方法，但需要目标状态可以被程序化检测。失效场景：目标条件可能被误满足（如 Agent 把「确认过」当作「已完成」）。

**2. 置信度阈值（Confidence Threshold）：** Agent 在每一步评估自己的置信度。当连续 N 步的中间结果置信度低于阈值时，Agent 应寻求人类帮助而非继续尝试。适合：需要在不确定时及时刹车以减少资源浪费的场景。

**3. 步数限制（Step Limit / Max Iterations）：** 设置绝对的最大工具调用次数（如 25 次）。这是最后的防线——即使其他策略都失效了，步数限制也能防止无限循环。CallSphere 建议同时设置 Token 预算和时间预算（硬拒绝超时）。

**4. 状态停滞检测（Stagnation Detection）：** 监控 Agent 的「进度信号」——如果 Agent 连续几轮没有产生有意义的进展（工具调用模式重复、中间结果不变），自动触发停止或降级策略。novelvista 称之为「No meaningful progress detected」策略。

**面试加分：** 「生产系统应该组合使用多种停止策略。泥梨推荐的组合是：**步数限制作为硬性保护 + 目标完成作为主停止条件 + 状态停滞检测作为早期预警**。另外有一种 2026 年正在兴起的策略叫做**渐进式释放（Progressive Release）**——先给 Agent 少量步数预算，如果它表现出色（连续正确调用、持续接近目标），逐步释放更多步数。这种方式既能防止循环，又不会过早切断高价值探索。」

### Q5. 当 Agent 在循环中重复调用同一个工具但每次返回不同的结果时（例如轮询状态 API），如何设计循环检测与断环机制？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | Anthropic、OpenAI、所有 Agent 平台团队
**答：**

这是 CallSphere 2026 面试指南和 CoPrep 文章中都重点讨论的场景——看似「合理」的工具调用重复实际上可能是 Agent 陷入死循环的表现。

**循环检测的三种方法：**

**方法 1 — 语义模式匹配（Semantic Pattern Matching）：**
检测以下模式是否重复出现：

```
Thought: "I need to check the status again"
Action: call_api(check_status)
Observation: "Status: PROCESSING"
Thought: "Let me check again..."
Action: call_api(check_status)
Observation: "Status: PROCESSING"
```
如果同一工具名 + 同一参数序列出现 3 次以上且 Observation 没有意义的变化，触发断环。

**方法 2 — 信息增益检测（Information Gain Detection）：**
每一次工具调用后，评估返回结果对当前状态的「信息增量」。如果连续 K 次调用的信息增量低于阈值（输出与上次基本一致），说明 Agent 在空转。

**方法 3 — 规划偏离检测（Plan Deviation Detection）：**
Agent 在执行计划时持续偏离原始目标路径。例如，原始计划是「1. 查订单 → 2. 查物流 → 3. 返回」，但 Agent 卡在「查订单」步骤反复调用。

**断环机制：**
1. **自动降级：** 中断循环后，Agent 进入「保守模式」——限制可调用的工具集合，只允许只读操作
2. **强制休息：** Agent 暂停 N 秒后重试（某些状态变化不需要频繁轮询）
3. **目标重置：** 中断当前计划，让 Planner 完全重新生成新计划
4. **人类升级：** 如果以上全部失败，向人类发送「我卡住了，帮我一把」消息

**面试加分：** 「2026 年有一个很好的实践：**把工具调用视为一次『探索』而非『执行』**。每次工具调用应该回答一个明确的『信息假设』——Agent 在调用前应该能回答『我调用这个工具是为了验证什么假设？』。如果 Agent 无法明确表达调用目标，就是一个早期的循环预警信号。这本质上是一种**元推理监测**的思路。」

### Q6. 设计一个生产级 AI Agent Evaluation 框架，需要包含哪些维度的评估指标？为什么说「Trajectory Metrics」和「Outcome Metrics」同等重要？

**类型：** 系统设计
**考察频率：** ⭐⭐⭐⭐⭐ | Galileo、Thinking Company、Anthropic
**答：**

Galileo 的《Agent Evaluation Framework》指南和 Thinking Company 的《AI Agent Evaluation in Production》都对这个问题给出了系统性的回答。2026 年，Agent 评估已从「测不测」进入「怎么测才科学」的阶段。

**多维评估指标体系：**

**一、结果指标（Outcome Metrics）——回答了「Agent 是否完成了任务」：**
1. **任务完成率（Task Completion Rate）：** Agent 是否成功达成了用户目标？二进制成功/失败判断
2. **部分完成得分：** 3/5 子任务完成 = 60% 完成度
3. **人工干预率：** Thinking Company 强调——人类需要多少次干预才能让 Agent 完成任务

**二、轨迹指标（Trajectory Metrics）——回答了「Agent 是怎么完成任务的」：**
Galileo 的核心论点是：结果指标告诉你 Agent 是否工作，轨迹指标告诉你为什么它会失败。
1. **工具调用效率：** 完成任务需要多少次工具调用？最优路径应该是几步？
2. **工具选择正确率：** Agent 是否选择了正确的工具，还是靠试错碰对了？
3. **参数准确性：** 工具调用的参数是否正确？语法准确率和语义准确率
4. **中间冗余度：** Agent 走了多少弯路？回头路占比多少？

**三、效率指标：**
1. **消耗 Token 数（总成本）**
2. **运行时长（延迟）**
3. **每任务平均模型调用次数**

**四、安全合规指标：**
1. **越权操作次数：** Agent 是否尝试过未被授权的操作
2. **模糊指令处理：** Agent 在不确定时选择澄清还是猜测
3. **失败安全行为：** 发生错误时 Agent 表现是否安全

**生产化评估流水线：**
Thinking Company 建议三级评估策略：
- **确定性测试（Deterministic）：** 精确预期结果的测试（如「预订 3 月 30 日下午 2 点」→ 检查创建时间是否正确）
- **LLM-as-Judge 测试：** 质量评估（如「总结近期 AI 安全论文」→ LLM 评分相关性、完整性、准确性）
- **人工评估：** 随机抽样真实用户交互，按有用性、安全性、效率评分

**面试加分：** 「Galileo 还有一个很棒的洞见：**轨迹指标比结果指标更容易发现回归问题**。任务完成率可能一周内都保持 85%，但工具调用次数却从 5 次悄悄增长到了 8 次——这意味着 Agent 正在慢慢『变笨』。跟踪轨迹指标的时序变化可以比结果指标更早地发现 Agent 退化。」

### Q7. RAG 系统中，当检索到的文档与 LLM 的内部知识冲突时，系统应该如何处理？有哪些经过验证的冲突解决策略？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | DataCamp、所有 RAG 团队
**答：**

DataCamp 2026 年 30 题指南中明确指出，检索信息与 LLM 内部知识的冲突是 RAG 系统在生产中最常遇到的挑战之一，也是面试高频题。

**主要冲突类型：**
1. **时间冲突：** 检索到的文档更新，但 LLM 训练数据旧（如 2026 年的新政策 vs 模型在 2025 年的训练数据）
2. **粒度冲突：** 检索到的文档提供具体细节，LLM 内部知识只有概览
3. **事实冲突：** 检索文档与 LLM 知识存在事实不一致（如不同来源的统计口径不同）

**解决策略（从弱到强）：**

**策略 1 — 提示工程强制约束：** 在 System Prompt 中明确写入「请仅基于提供的上下文回答，不要依赖内部知识」。这是最简单的方案，但在 LLM 的「惯性」面前并不完全可靠。

**策略 2 — 上下文优先排序（Context Priority）：** 设计 prompt 策略让检索到的上下文「覆盖」模型的内部知识。DataCamp 建议使用类似指令：「Use the following context to answer. If the context is silent on a topic, you may use your own knowledge, but clearly indicate which part comes from context and which is your knowledge.」

**策略 3 — 显式冲突标记与用户透明：** 当 Agent 检测到检索文档与内部知识不一致时，将两种信息都呈现给用户，并标注来源置信度。例如：「根据最新文档（2026-05-15 更新），政策为 A。而根据我的训练数据（截至 2025-12），政策为 B。建议您确认官方最新文件。」

**策略 4 — 交叉验证（CRAG 方式）：** 使用 Corrective RAG 流程——先让 LLM 评估检索文档的相关性和可信度，如果文档可信但结果与内部知识矛盾，选择信任检索结果（因为时效性更高）；如果文档本身不可信（质量低、来源不明），回退到内部知识。

**面试加分：** 「2026 年还有一个很有前景的方案：**显式知识版本管理（Versioned Knowledge）**。给检索的文档打时间戳和来源标签，让 LLM 能够明确感知『这个信息来自 2026-05 更新的官方文档』。配合 LLM 对自己知识截止日期的认知，通过时间戳比较来做出判断。这本质上是把人类的『谁更可信』判断逻辑——时效性 > 权威性 > 模型记忆——程序化地注入 Agent 的决策过程。」

### Q8. 如何为 Agentic AI 系统设计成本控制策略？在多 Agent 系统中，模型路由（Model Routing）的最佳实践是什么？

**类型：** 产品思维
**考察频率：** ⭐⭐⭐⭐ | CallSphere、novelvista、所有生产 Agent 团队
**答：**

CallSphere 面试指南明确警告：Multi-Agent 系统可能比单 Agent 贵 5-10 倍。novelvista 的 ROI 测量部分也强调了成本控制是 2026 年面试的必考点。

**Agent 成本控制策略分三个层次：**

**层次 1 — 每步优化：**
- **缓存复用：** 对相同输入的检索结果、工具返回值做缓存（Cache-Augmented Generation / CAG）
- **减少无效调用：** 检索门控——只在需要外部信息时才调用检索工具，而非每次查询都检索
- **Token 预算限制：** 设置每个任务的 Token 上限，超出后自动进入缩减模式

**层次 2 — 任务级优化：**
- **渐进式模型路由：** 先让小型模型（如 Claude Haiku 4.5）处理常规请求，只有当任务复杂度超出小模型能力时，才升级到大型模型（如 Claude Sonnet/Opus）
- **预分类路由：** 用路由器（Router）在入口处判断请求类型，简单问题走低成本路径，复杂问题走高成本路径。Anthropic 的 Routing 模式正是为此设计

**层次 3 — 系统级优化：**
- **并行执行：** 在可能的子任务间使用并行化（Sectioning 模式），降低总运行时间（从而降低集群占用成本）
- **热度追踪：** 高频重复的请求可以预计算并缓存完整结果

**模型路由最佳实践：**
CallSphere 建议的决策框架：

| 子任务类型 | 推荐模型 | 理由 |
|---|---|---|
| 简单路由/分类 | Haiku / GPT-4o-mini | 便宜、足够准确 |
| 单步工具调用 | Sonnet / GPT-4o | 平衡能力和成本 |
| 复杂推理/规划 | Opus / GPT-4 | 需要深度思考能力 |
| 代码生成 | Sonnet / Claude Code | 代码能力最优 |
| 检索后摘要 | Haiku + 结构化输出 | 只需对检索结果做摘要 |

**面试加分：** 「2026 年成本控制的最大杠杆不是模型本身，而是**任务分解粒度**。如果你把一个『写报告』的任务不加分解直接丢给 Opus，成本极高；但如果你把它分解为『搜资料→摘要→组织大纲→写正文→润色』五个步骤，并且前四步用 Haiku 或 Sonnet，最后一步用 Opus——总成本可以降低 60-70% 而质量几乎不降。这需要对任务有深刻理解，这种能力正是面试官想看到的。」

### Q9. 设计一个 Agent 可观测性（Observability）框架，需要追踪哪些关键数据点？如何设计 Agent 的可观测性三大支柱（Logs、Metrics、Traces）？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | Thinking Company、Galileo、所有 Agent 团队
**答：**

Thinking Company 2026 指南特别指出，传统监控对 Agent 完全不够用——因为你无法像追踪常规 API 调用那样追踪一个自主决策链。

**Agent 可观测性的三大支柱：**

**支柱 1 — 结构化日志（Logs）：**
每一轮 Agent 推理循环都生成结构化日志，包含：
- Agent 的 Thought（推理过程）：LLM 在每一步思考了什么
- Action（工具调用）：调用了哪个工具、传入了什么参数
- Observation（工具返回）：返回结果全文
- 决策依据：Agent 为什么选择了这个工具而不是另一个
- 性能元数据：每次 LLM 调用耗时、Token 消耗

Thinking Company 强调：**日志的结构化程度决定了调试效率**——非结构化的自然语言日志虽然直观，但无法自动化分析。

**支柱 2 — 聚合指标（Metrics）：**
- **成功率：** 任务完成率（按请求类型、模型、工具分解）
- **延迟分布：** P50/P95/P99 的每步决策时间
- **工具调用次数分布：** 平均/最大/最小调用次数
- **Token 消耗：** 输入/输出 Token 比、每日 Token 成本
- **人工干预率：** Thinking Company 定义的「每当人类必须介入修正」的比率
- **循环检测触发率：** 触发断环机制的次数

**支柱 3 — 分布式追踪（Traces）：**
Agent 的一个请求可能经历多次 LLM 调用和工具调用。完整追踪包括：
- 跨 LLM 调用的 trace ID（将一次 Agent 任务中的所有 LLM 调用关联起来）
- 每次工具调用的 span（完整输入输出 + 耗时 + 状态码）
- Agent 状态转换图（从初始化 → 推理 → 工具调用 → 结果评估的完整路径）

**关键监控事件：**
- Tool call repeated with same params（参数完全相同的重复调用）
- No new information after N steps（N 步后无新信息）
- Confidence dropping（置信度持续下降）
- Context window approaching limit（上下文窗口接近满）
- Token budget exceeded（Token 预算超限）

**面试加分：** 「生产 Agent 的调试和传统后端调试有本质区别——你无法像回放数据库查询那样『重放』一次 agent 决策。所以 **Trace ID 的端到端关联** 就至关重要：把用户请求 ID、LLM 调用 ID、工具调用 ID 三者关联起来，才能从『用户投诉』回溯到『哪个 Agent 决策出了问题』。另一个好实践是**每一步的 Checkpointing**——保存每一步的状态快照，支持任意步的回滚重试。」

### Q10. Agent 系统在检索和生成过程中如何检测和纠正偏见？如何确保 Agent 输出的公平性和客观性？

**类型：** 安全伦理
**考察频率：** ⭐⭐⭐ | DataCamp、CallSphere、novelvista
**答：**

DataCamp RAG 面试指南和 CallSphere Agent 安全部分都涉及了这个问题。随着 Agent 在招聘、金融、医疗等敏感领域的部署加速，偏见检测已成为面试中的增长考点。

**偏见的来源：**

1. **检索偏见（Retrieval Bias）：** 知识库本身的内容倾斜——例如招聘 Agent 的训练数据中多数是男性工程师的简历，导致 Agent 推荐候选人时偏向男性
2. **索引偏见（Indexing Bias）：** 嵌入模型的训练数据倾斜导致某些群体的描述被「语义压缩」（semantically compressed），检索召回率偏低
3. **生成偏见（Generation Bias）：** LLM 的训练数据固有偏见在生成答案时被放大

**系统性的偏见检测与纠正方法：**

**方法 1 — 知识库清洗（Source Curation）：**
在知识库构建阶段过滤偏见内容。DataCamp 指出，第一步永远是在源头确保数据尽可能客观。建立多来源的交叉验证机制——如果一个信息只在单一来源出现且带有主观判断，标记为「低可信度」。

**方法 2 — 检索多样性增强（Diverse Retrieval）：**
在检索阶段主动增加多样性——引入聚类方法，从不同类别的文档中选取代表性样本，而非只返回 Top-K 个相似度最高的结果。DataCamp 提到需要平衡相关性和多样性。

**方法 3 — 专用偏见检查 Agent：**
CallSphere 和 DataCamp 都建议引入一个专门的 Checker Agent 来评估生成的输出是否存在潜在偏见。这个 Agent 的职责不是生成内容，而是审查内容。

**方法 4 — 输出后处理与标记：**
在 Agent 输出给用户前，增加偏见检测层。检测到可能的偏见输出时，触发三种策略：
- **拒绝输出：** 直接阻止输出
- **标记透明：** 输出内容并附上偏见警告
- **重新生成：** 用修正后的 prompt 重新生成

**面试加分：** 「2026 年 Agent 偏见检测最前沿的实践是**差分公平性测试（Differential Fairness Testing）**——对同一查询换不同的群体属性（如男/女、不同文化背景）测试输出的一致性。例如对『推荐一位项目经理』的查询，如果替换 Candidate 的性别/民族后 Agent 的推荐排序发生变化，就检测到了偏见。这种方法比事后检查更可靠，因为它从**系统的行为一致性**角度定义偏见，而不是依赖对『什么算偏见』的主观判断。」
