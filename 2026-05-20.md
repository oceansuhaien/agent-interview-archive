### Q1. LangChain 的 Subagent（子代理）模式与 Supervisor（监督者）模式在设计目标和适用场景上有何本质区别？2026 年更推荐哪种？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | LangChain、Anthropic、字节跳动、高频考题
**答：**

LangChain 2026 多 Agent 文档和 Anthropic《Building Effective Agents》指南都深入讨论了 Agent 编排模式的选择。这是 2026 年面试中区分架构设计能力的关键题。

**Subagent 模式（主代理将子代理作为工具）：**
主 Agent 维护完整的决策控制权，子 Agent 以「工具」的形式注册。主 Agent 决定何时调用哪个子 Agent、传入什么上下文、如何整合返回结果。本质是「Agent as a tool」——主 Agent 仍然拥有最终决策权。LangChain 文档称其为「Skill-based」模式。

**Supervisor 模式（监督者协调）：**
Supervisor Agent 负责任务路由和 Worker Agent 的协调，但 Worker 本身也是自主决策者——它们可以在自己的任务域内自由调用工具和做决策，只需要向 Supervisor 汇报结果。LangChain 文档称其为「Orchestrator」模式。

**本质区别：**

| 维度 | Subagent 模式 | Supervisor 模式 |
|------|--------------|-----------------|
| 控制粒度 | 主 Agent 完全控制 | Worker 有自主权 |
| 通信模型 | 函数式调用 | 任务式派发 |
| 复杂度 | 低，适合 5 个以下子 Agent | 高，适合大规模团队 |
| 可扩展性 | 有限，主 Agent 易成瓶颈 | 好，可水平扩展 |

**2026 年推荐：** Anthropic 推荐从 Subagent 模式开始，只有在子 Agent 数量超过 5-10 个或需要大量自主决策时才升级到 Supervisor。原因是 Subagent 模式保持了 Single Agent 的可调试性，而 Supervisor 模式引入了中央协调 Agent 的额外复杂度和成本。

**面试加分：** 「一个实用的决策标准是：**子 Agent 是否需要自己的记忆和状态**。如果子 Agent 只是执行一次性任务（查天气、算数学），Subagent 模式就够了。如果子 Agent 需要维护自己的持久化记忆（如客服 Agent 需要对每个用户保持长期对话上下文），就应该用 Supervisor 模式让每个 Worker 独立维护状态。」

### Q2. AI Agent 在生产中使用结构化输出（Structured Output / JSON Mode）时有哪些常见的陷阱？如何设计可靠的结构化输出系统？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | OpenAI、Anthropic、所有 Agent 平台团队
**答：**

Anthropic《Building Effective Agents》指南和 CoPrep 面试指南都强调了结构化输出在 Agent 系统中的核心地位。但结构化输出在生产中有很多隐藏的陷阱。

**常见陷阱：**

**陷阱 1 — Schema 过于复杂：** 嵌套过深（3 层以上）或字段过多的 JSON Schema 会显著降低 LLM 的生成稳定性。Anthropic 建议：「保持格式接近模型在互联网上见过的自然格式。」Field 数超过 20 个时错误率急剧上升。

**陷阱 2 — 枚举值匹配失败：** LLM 可能会输出 `"status": "PENDING"` 而 Schema 要求的是 `"status": "pending"`。字符串大小写、空格、特殊字符的精确匹配是高频问题。

**陷阱 3 — 空值与缺失字段：** 当 LLM 无法推断某个字段时，它可能会输出 `null`、空字符串、或直接跳过字段——不同行为导致不同的下游错误。

**陷阱 4 — 工具调用的参数类型错误：** Agent 调用的工具参数应有严格的类型约束。Anthropic 指出「工具描述中写入错误处理指引」——参数类型错误时，Agent 应该得到清晰的结构化错误反馈。

**可靠设计策略：**
1. **Schema 扁平化：** 保持 JSON 深度 ≤ 2 层，字段数 ≤ 15
2. **约束解码（Constrained Decoding）：** 使用 Outlines、JSON-mode、Guidance 等工具在生成阶段强制 Schema 合规，而非事后校验
3. **自动纠错回退：** 当结构化输出校验失败时，自动重试（附加「上次输出格式有误，请按如下格式重新输出」）
4. **类型转换适配器：** 在 Agent 框架和下游工具之间加一个 Adapter 层，处理字符串与枚举的模糊匹配

**面试加分：** 「讲出 **JSON Repair** 机制很加分——当 LLM 输出不完整 JSON 时，用一个轻量的修复模型或启发式规则（补全缺失括号、修复字符串引号）而不是直接抛错给 LLM 重做。OpenAI 的 Structured Output 模式 + 修正循环可以覆盖 99% 的格式错误，但代价是额外的延迟。在延迟敏感的场景中，Constrained Decoding 的方案更优。」

### Q3. 除了 ReAct 循环之外，AI Agent 的系统性幻觉还有哪些根本原因？生产环境中有哪些经过验证的缓解策略？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、CoPrep、OpenAI、Google、高频考题
**答：**

CoPrep 2026 面试指南和 Anthropic 的 Agent 构建指南都强调：幻觉问题在 Agent 环境下比在纯对话环境下更复杂也更危险。

**Agent 系统的幻觉来源（不仅限于循环）：**

**来源 1 — 工具返回值「自动补全」：** LLM 在 Observation 不理想时倾向于「创造」它希望看到的工具返回值。例如天气 API 返回 500，Agent 可能编造「今天天气晴朗」来继续执行。

**来源 2 — 上下文稀释（Context Dilution）：** Agent 每次推理都会将之前的 Thought + Action + Observation 全部塞入上下文。随着上下文增长，关键信息被稀释，Agent 开始依赖「模糊记忆」而非精确的 Observation。

**来源 3 — 工具 Schema 的隐含假设：** 工具的 description 写得不好导致 Agent 误解其功能。例如 `search_database(query: str)` 的描述没写清「只支持 SELECT 查询」，Agent 可能用它来执行 INSERT 并编造「插入成功」。

**来源 4 — 决策疲劳（Decision Fatigue）：** 长序列推理中，Agent 在后期倾向于「尽快结束」而忽略验证步骤，导致 hallucinated 结论。

**系统性缓解策略：**
1. **工具返回值 Schema 校验：** 每次工具返回后校验其是否符合预期 Schema。如果不符合，生成结构化的错误提示而不是模糊的「Error」
2. **上下文周期压缩（Context Compression）：** 每 N 步对历史做一次摘要压缩，减少上下文稀释。使用关键信息提取（critical information extraction）而非简单摘要
3. **自一致性检查（Self-Consistency）：** 在最终输出前让 Agent 交叉验证自己的结论是否与所有 Observation 一致
4. **分阶段验证：** 将长任务拆分为有明确检查点的短任务，每个检查点验证中间输出的可靠性

**面试加分：** 「2026 年最前沿的实践是 **Oracle Check**——在 Agent 的步骤之间插入一个独立的验证模型（可以是一个小模型或规则引擎），专门检查 Agent 的推理是否有逻辑跳跃。Oracle Check 不参与决策，只做事实性一致性验证。CoPrep 指出，这是唯一能从『系统级别』而非『模型级别』解决幻觉的方法。」

### Q4. 在 RAG 系统中，Dense Embedding（如 text-embedding-3-large）和 Sparse Embedding（如 SPLADE/BM25）各自适合什么检索场景？混合检索（Hybrid Search）的最佳实践是什么？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | DataCamp、字节跳动、微软、所有 RAG 团队、高频考题
**答：**

DataCamp 2026 RAG 面试指南深入对比了不同检索策略，这是 RAG 工程师面试中的核心区分题。

**Dense Embedding（稠密检索）：**
利用神经网络将文本映射到高维连续向量空间，通过语义相似度检索。

- ✅ 优点：理解语义——「苹果」和「iPhone」在向量空间中相近，能处理同义词和改写
- ❌ 缺点：训练数据偏差——在罕见或专业术语上效果差；「语义压缩」问题——对稀有实体的表示分辨率低
- 适合：开放域问答、语义匹配、需要跨语言理解

**Sparse Embedding（稀疏检索/关键词匹配）：**
基于词频-逆向文档频率（BM25）或学习型稀疏向量（SPLADE）。

- ✅ 优点：精确匹配——「ID-123456」这样的精确实体查询效果好；无训练偏差
- ❌ 缺点：缺乏语义理解——「快速行驶的汽车」和「高速公路上飞驰的车辆」不匹配
- 适合：精确实体查询（订单号、人名）、法律/医疗文档的关键术语匹配

**混合检索最佳实践：**
1. **并行检索 + 加权融合（RRF）：** 同时运行 Dense 和 Sparse 检索，用 Reciprocal Rank Fusion（RRF）合并结果。RRF 会给两个排名靠前的结果加分，防止单一检索策略的遗漏
2. **查询分析器路由：** 用一个分类器判断查询类型——实体查询走 Sparse，语义查询走 Dense，混合查询走 Hybrid
3. **多阶段检索（Multi-stage）：** 第一轮用 Sparse 做粗筛（快速排除无关文档），第二轮用 Dense 做精排（语义排序）
4. **动态权重调整：** 根据查询的意图类型动态调整 Dense vs Sparse 的权重。例如关键词型查询（「2024 年财报」）给 Sparse 更高权重，开放式查询（「公司的成长潜力如何」）给 Dense 更高权重

**面试加分：** 「讲出 **late interaction** 机制（如 ColBERT）很加分——它在 Dense 和 Sparse 之间取得了平衡，保留了 Token 级别的交互信号。另外 DataCamp 提到，2026 年的趋势是**多向量检索（Multi-Vector Retrieval）**——一个文档被编码为多个向量，每个向量代表一个方面，检索时匹配最相关的方面维度。」

### Q5. Plan-then-Execute（先规划后执行）、ReAct（边推理边行动）、Plan-and-Solve（规划与解决分离）三种 Agent 任务规划模式各有什么优劣？生产场景下应该如何选择？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Anthropic、OpenAI、Google DeepMind、字节跳动
**答：**

CoPrep 2026 面试指南将任务规划策略列为 Agent 工程师面试的核心考点，Anthropic 的 Agent 模式分类也深入讨论了不同的规划方法。

**Plan-then-Execute（PTE）：**
Agent 先在「规划阶段」生成完整的执行计划（包含所有步骤和工具调用），然后一次性执行。类似写代码再运行。

- ✅ 优点：每次执行的成本可控（规划阶段消耗推理 Token，执行阶段只需较少的 LLM 调用）；计划可审核
- ❌ 缺点：灵活性差——规划时不知道实际执行会遇到什么意外；一旦计划与实际不符，整个计划作废
- 适用：高度确定性的任务（按流程办事、ETL 流水线）

**ReAct（Reason + Act）：**
Agent 在每一步交替执行推理和行动，没有前期规划。每一步都是一个独立的「看到当前状态 → 决定下一步」。

- ✅ 优点：高度灵活——每一步都可以根据实时的 Observation 调整；适应性强
- ❌ 缺点：Token 消耗高（每一步都需要 LLM 推理）；缺乏大局观——容易陷入局部最优（无意义地重复某一步）
- 适用：不确定性强、需要频繁调整的任务（客服对话、信息搜集）

**Plan-and-Solve（PS）：**
Plan-then-Execute 和 ReAct 的混合。Agent 先生成初始计划（Plan），然后在执行过程中根据 Observation 动态调整计划（Solve）。Plan 不是一次性固定的，Agent 可以在执行中「局部重规划」。

- ✅ 优点：兼具计划的前瞻性和执行的灵活性；执行中可修正计划偏差
- ❌ 缺点：实现复杂；需要额外的「计划修正」机制
- 适用：复杂多步骤任务（代码开发、文档生成、多工具编排）

**生产选择指南：**
| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| 流水线式 ETL | PTE | 步骤确定，无需调整 |
| 开放域问答/搜索 | ReAct | 需要根据搜索结果实时调整 |
| 代码开发/报告撰写 | PS | 需初始计划但执行中可能变更 |
| 客服对话 | ReAct | 高度不确定，每一步依赖用户输入 |

**面试加分：** 「2026 年的生产实践倾向于采用 **Plan-then-ReAct 组合**——先用 PTE 生成一个粗粒度的阶段计划（Phase Plan），然后在每个 Phase 内部使用 ReAct 做细粒度的步骤执行。这种方式既保有了计划的前瞻性，又保留了 ReAct 的灵活性。Anthropic 的实践也支持这种分层规划的思路——高层规划用更强模型，低层执行用更便宜模型。」

### Q6. 如何为生产环境中的 Agent 系统设计 A/B 测试与渐进式发布（Progressive Rollout）方案？Agent 的 A/B 测试与传统后端的核心区别在哪里？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Thinking Company、Galileo、字节跳动、蚂蚁集团
**答：**

Thinking Company 2026 生产指南强调了 Agent 系统在发布和实验方面的特有挑战。这是将 Agent 投入生产时不可避免的问题。

**Agent A/B 测试与传统后端的核心区别：**

| 维度 | 传统后端 A/B 测试 | Agent A/B 测试 |
|------|------------------|----------------|
| 变量控制 | 可精确控制（代码路径确定） | 变量多且相互耦合（LLM 温度、prompt、工具集、检索策略） |
| 评估指标 | 明确的业务指标（CTR、转化率） | 多维度（质量 + 成本 + 安全） |
| 回滚难度 | 低（切回旧版本即可） | 高（Agent 行为因记忆和状态而难以回溯） |
| 测试周期 | 短（分钟级即可收集数据） | 长（需收集完整的推理轨迹） |

**渐进式发布方案：**

**阶段 1 — 离线评估（Offline Eval）：**
在历史数据集上回放 Agent 的行为。用「模拟用户」生成测试请求，比较新旧 Agent 的推理轨迹质量。关键指标：任务完成率变化、工具调用效率变化、成本变化。

**阶段 2 — 影子发布（Shadow Mode）：**
新 Agent 与旧 Agent 并行运行。旧 Agent 处理真实用户请求并产生效果，新 Agent 也接收同样的输入但结果不展示给用户。对比两个 Agent 的推理轨迹和输出质量。这是 Thinking Company 推荐的最安全的测试模式。

**阶段 3 — Canary 发布（Canary Release）：**
将 1-5% 的真实流量导向新 Agent。与旧 Agent 共享同一个数据库/缓存状态（确保测试公平性）。监控关键指标：人工干预率、错误率、延迟、用户反馈。

**阶段 4 — 逐步放量：**
Canary 稳定后逐步放量到 10% → 25% → 50% → 100%。每个阶段至少持续 24 小时收集完整的使用周期数据。如果出现**工具调用错误率上升 5%** 或**人工干预率翻倍**等预警信号，立即暂停放量并回滚。

**面试加分：** 「Agent A/B 测试特有的一个实践是 **Prompt 版本管理**——不仅仅管理代码版本，还要管理 Prompt 版本和 Tool Schema 版本。Galileo 建议使用 Trajectory Diff 工具对比新旧 Agent 在相同输入上的决策路径差异 —— 如果决策路径完全不同但最终结果相似，说明新版本的鲁棒性可能不及旧版本。另外，**成本指标**在 Agent A/B 测试中扮演核心角色 —— 一个质量提高 2% 但成本翻了 3 倍的 Agent 通常不可接受。」

### Q7. Agent 系统如何管理有限的上下文窗口（Context Window）？Agent 在长对话中的记忆压缩（Memory Compression）策略有哪些？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | OpenAI、Anthropic、字节跳动、所有 Agent 平台团队、高频考题
**答：**

CallSphere 2026 面试指南指出，上下文窗口管理是 Agent 在生产中必须面对的核心工程挑战。随着 Agent 逐步推理，历史记录无限增长，必须设计高效的压缩策略。

**四种主流记忆压缩策略：**

**策略 1 — 滑动窗口（Sliding Window）：**
只保留最近 N 轮的 Thought-Action-Observation，丢弃较早的历史。是最简单的策略，但可能丢失重要上下文。

- 实现：当总轮数超过 K 时，删除最早的一半
- 优点：实现简单、Token 开销可控
- 缺点：可能丢失关键信息（用户早期的指令、重要决策依据）

**策略 2 — 摘要压缩（Summarization）：**
每 M 轮对之前的对话历史做一次摘要压缩。将压缩后的摘要与最近几轮的原始记录拼接。

- 实现：用 LLM 将过去的 N 轮生成为 200 字摘要，保留最近 3 轮原始记录
- 优点：保留关键信息、显著降低 Token 消耗
- 缺点：摘要本身可能丢失细节或引入摘要幻觉

**策略 3 — 关键信息提取（Critical Information Extraction）：**
不是对所有内容做通用摘要，而是有选择性地提取「对后续任务有用的信息」。例如提取用户的偏好、已确认的事实、未完成的任务。

- 实现：用结构化格式（如 JSON）提取关键字段，丢弃无关对话内容
- 优点：信息密度高、结构化存储方便检索
- 缺点：需要预先定义「什么是关键信息」的 Schema

**策略 4 — 分层记忆架构（Hierarchical Memory）：**
结合以上三种策略，形成多级记忆体系：

```
短期记忆（最近 3-5 轮原始记录）
    ↓ 每轮写入
中期记忆（摘要压缩，保留关键决策轨迹）
    ↓ 每 5-10 轮压缩
长期记忆（结构化提取，持久化存储）
```

**面试加分：** 「2026 年最推荐的生产实践是 **Token 预算驱动的自适应压缩**——给每个 Agent 任务分配一个 Token 预算（如 32K tokens），当上下文接近阈值时自动触发压缩操作。压缩时优先保留**工具调用返回值**（Observation 中的事实性数据），优先丢弃**Thought**（推理过程可以重建但工具返回不能重建）。这种有取舍的压缩比通用摘要更实用。另外，非对称裁剪（asymmetric pruning）也很重要——系统 prompt 和工具定义优先保留，对话历史优先压缩。」

### Q8. 在 Tool-Use 系统中，Agent 如何准确推断工具参数？当参数类型复杂（嵌套对象、联合类型、可选字段）时有哪些验证和纠错机制？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Anthropic、OpenAI、LangChain
**答：**

Anthropic《Building Effective Agents》指南和 CoPrep 面试指南都强调了 Tool 调用中参数准确性的重要性。Anthropic 的 Poka-yoke 原则明确指出：「工具描述应该包含何时不应使用此工具的指引。」

**Agent 参数推断的挑战：**

1. **多义词参数：** `date` 参数可能接受 `"2026-05-20"`、`"05/20/2026"`、`"May 20, 2026"` 多种格式
2. **隐式类型约束：** `amount` 参数在 Schema 中标记为 `number`，但实际只能接受正整数
3. **可选字段的默认值冲突：** 当 Agent 未传入某个可选字段时，后端使用默认值可能产生意外行为
4. **联合类型（Union Type）：** 参数可以是 `string | number`，Agent 可能生成错误的类型

**验证和纠错机制：**

**机制 1 — Schema 预验证（Pre-Validation）：**
在工具调用执行前，使用 JSON Schema validator 检查参数：
- 类型检查：是否类型匹配（`number` 不能传 `"abc"`）
- 格式检查：日期格式、邮箱格式等
- 枚举检查：是否在允许值范围内

**机制 2 — 智能错误反馈（Structured Error Feedback）：**
当参数校验失败时，Anthropic 建议返回精确的错误信息，而非含糊的「Invalid params」。例如：

```
❌ "Invalid parameters"
✅ "Parameter 'date' received 'abc', expected ISO format (YYYY-MM-DD)"
```

**机制 3 — 参数推断增强（Enhanced Parameter Inference）：**
在 Tool 的描述中，对每个参数提供：
- 精确的格式要求（「接受的格式：YYYY-MM-DD」）
- 参数之间关系的说明（「如果传入 start_date，end_date 也必须传入」）
- 典型值示例（「例如：50000」）

**机制 4 — 自动类型转换 Adapter：**
在 Agent 框架和下游 API 之间加一个适配层，处理常见类型转换：
- 字符串数字 → 数字（`"500"` → `500`）
- 大小写归一化（`"true"` → `True`）
- 格式归一化（各日期格式 → 统一 ISO 格式）

**面试加分：** 「讲出 **参数渐进式验证（Progressive Validation）** 很加分 —— 先在函数调用前做 Schema 级校验（快速失败），然后在工具执行后再做业务逻辑校验（慢速但更全面）。例如先检查 `amount` 是不是数字，执行后再检查金额是否超过账户余额。Anthropic 推荐在工具描述中明确写『哪些错误 Agent 可以自己修复』和『哪些错误必须上报』。」

### Q9. 在边缘设备（手机、IoT 设备）上部署 AI Agent 与在云端部署相比，有哪些独特的设计考量和权衡？2026 年的最佳实践是什么？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐ | Apple、Google、华为、高通
**答：**

CallSphere 面试指南和 DataCamp RAG 指南都提到了 Agent 在不同部署场景下的不同设计。随着端侧 AI 能力的提升，边缘 Agent 在 2026 年已经成为面试热点。

**核心设计差异：**

| 维度 | 云端 Agent | 边缘 Agent |
|------|-----------|-----------|
| 计算资源 | 无限（弹性伸缩） | 有限（CPU/内存受限制） |
| 模型大小 | 可部署 200B+ 模型 | 通常 7B 以下（量化后） |
| 延迟 | 受网络影响（50-200ms） | 本地推理（10-50ms） |
| 隐私 | 数据需上传 | 数据本地处理 |
| 持久性 | 无状态或状态在远端 | 依赖设备本地存储 |
| 电源 | 无需考虑 | 电量敏感 |

**边缘 Agent 的独特设计考量：**

**考量 1 — 模型选择与量化：** 边缘设备只能运行小模型（如 Gemma 2B、Qwen2.5-1.5B、Phi-3-mini），通常需要 INT4/INT8 量化。模型能力弱意味着 Agent 需要更好的外部工具支持和检索能力来弥补推理能力不足。

**考量 2 — 离线韧性（Offline Resilience）：** 边缘 Agent 必须能优雅地处理网络断开——在离线时降级功能集，在线时同步状态和结果。DataCamp 提到，Edge RAG 需要在设备本地维护知识库的轻量副本。

**考量 3 — 工具执行环境：** 边缘 Agent 的工具集合更有限——无法调用云端的大数据服务，但可以访问设备的本地能力（传感器、文件系统、日历、短信）。

**考量 4 — 功耗效率：** 每次频繁的 LLM 推理会显著消耗电池。边缘 Agent 需要「推理预算」——在简单任务上使用规则引擎，仅在需要时才调用 LLM。

**2026 年最佳实践：**
- **分层推理架构：** 边缘 Agent 处理 80% 的简单请求（规则 + 小模型），剩余 20% 的复杂请求通过 API 路由到云端大模型
- **缓存优先策略：** 高频请求的结果在设备本地缓存，避免重复调用 LLM
- **异步同步：** 推理结果和设备状态在后台静默同步云端，不影响用户体验

**面试加分：** 「2026 年边缘 Agent 的杀手级应用是 **On-Device RAG**——设备本地存储用户的个人数据（联系人、日历、笔记），通过小模型 + 本地向量数据库实现完全离线的个性化 Agent。Apple Intelligence 和 Google 的 Gemini Nano 都在这个方向。讲出 **JIT Model Loading（按需加载模型部分）** 或者说 **模型分片推理**——只在需要特定能力时加载模型对应部分——会非常加分。」

### Q10. 在评估 AI Agent 系统的输出质量时，LLM-as-Judge（用 LLM 打分）、Human Evaluation（人工评估）、Deterministic Metrics（确定性指标）三种方法各有何优劣？如何在生产中选择合适的评估策略组合？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Galileo、Thinking Company、Anthropic、字节跳动
**答：**

Galileo 的《Agent Evaluation Framework》和 Thinking Company 的《AI Agent Evaluation in Production》都系统性地对比了不同的评估方法学。2026 年的面试重点已经从「用什么指标」转向了「如何组合指标」。

**三种评估方法的深度对比：**

**LLM-as-Judge（AI 评估）：**
用另一个 LLM（如 GPT-4o、Claude Sonnet）对 Agent 的输出进行评分。
- ✅ 优点：自动化程度高、可扩展、评估维度灵活（准确性、相关性、安全性等）
- ❌ 缺点：Judge 模型自身有偏见（偏向长输出、偏好自己的风格）；一致性差（同一输入两次评分可能不同）；成本高
- 适用：开放式任务的质量评估（报告、答案、对话质量）

**Human Evaluation（人工评估）：**
让人类标注员直接评估 Agent 输出的质量。
- ✅ 优点：最可靠的标准（gold standard）、能捕捉语义层面的细微差别
- ❌ 缺点：成本极高、速度慢、标注者间一致性（Inter-rater agreement）难以保障
- 适用：高风险场景（医疗、金融、法律）；模型上线前的最终验证

**Deterministic Metrics（确定性指标）：**
基于规则或精确计算的指标，如精确率、召回率、F1、BLEU、ROUGE。
- ✅ 优点：零成本、完全一致、可跨版本对比
- ❌ 缺点：只能评估精确匹配类任务；无法衡量语义质量
- 适用：有明确正确答案的任务（实体抽取、分类、计算）

**生产评估策略组合：**

**Thinking Company 推荐的三级评估策略：**
1. **第一级 — 确定性指标（持续跑）：** 用于功能测试和回归检测。每次部署前自动运行，确保基础功能正常
2. **第二级 — LLM-as-Judge（每轮部署跑）：** 在发布流程中自动触发，评估输出质量和安全性
3. **第三级 — 人工评估（抽样跑）：** 从生产流量中随机抽样 1-5% 的请求，请人类评估员评分

**面试加分：** 「Galileo 提出了一个很实用的概念——**评估金字塔（Evaluation Pyramid）**——底部是大量低成本确定性测试（几百个），中间是中等数量的 LLM-Judge 测试（几十个），顶部是少量人工评估（几个）。这个金字塔的比例随生产成熟度变化：种子阶段人工评估占比高，成熟阶段确定性测试占比高。另一个越来越受认可的实践是 **Consensus Judge（共识判决）**——使用 2-3 个不同的 Judge 模型分别评分，只有当它们一致通过时才标记为通过。分歧结果自动升级到人工评估。这种方式在成本和可靠性之间取得了很好的平衡。」
