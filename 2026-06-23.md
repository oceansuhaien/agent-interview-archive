### Q1. 什么是 GraphRAG？它如何解决传统向量检索 RAG 在处理「关系型问题」时的局限性？请描述它的核心架构和工作流程。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Microsoft、CoPrep、字节跳动、Google、2026年高频新考题
**答：**

GraphRAG（图检索增强生成）是 2026 年最热门的 RAG 演进方向之一。CoPrep 的面试指南和 Microsoft 的 GraphRAG 研究都强调了它的重要性。

**核心痛点：** 传统向量 RAG 擅长找「相似段落」，但无法回答需要「理解实体间关系」的问题。比如「我们银行哪些英国员工在 Project Titan 公开前就接触过它？」——向量 RAG 只能找到包含「英国员工」和「Project Titan」的文档，但无法建立员工→邮件→项目之间的关联路径。

**GraphRAG 的架构：**
1. **知识图谱构建阶段：** 从非结构化文档中提取实体（人员、项目、日期）和关系（「发邮件给」「参与」「提及」），构建为 (节点-边-节点) 的知识图谱。微软的方法使用 LLM 自动进行实体抽取和关系链接。
2. **混合检索阶段：** 用户查询同时触法向量检索（找相关文本段落）和图遍历（找关系路径）。两者结果融合后交给 LLM 生成答案。
3. **图遍历推理：** Agent 将用户问题转化为图查询语言（如 Cypher 或 GraphQL），在图结构中沿边搜索，找到连接多个实体的完整路径。

**面试加分：** 「GraphRAG 的价值不在于节点本身，而在于 Agent 在边之间导航的能力。当标准 RAG 给出『这些文档都提到了 A 和 B，但不知道它们之间的关系』时，GraphRAG 可以说『A 在时间 T 通过事件 E 联系了 B』。CoPrep 指出面试中的常见错误是把 GraphRAG 描述成『另一种索引方法』而非『推理框架』。」

### Q2. 在设计一个 Autonomous Agent 时，Planner（规划器）和 Executor（执行器）的职责如何拆分？Monitor/Critic 模块如何防止 Agent 跑偏或陷入死循环？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、CoPrep、OpenAI、微软、所有大厂 Agent 团队、核心高频题
**答：**

CoPrep 的指南和 Anthropic 的《Building Effective Agents》都强调了规划-执行-监控分离架构是生产级 Agent 的基础。

**Planner 的职责：** 将高层目标分解为可执行的子任务列表。例如「组织团队团建」→ `[1. 调查大家时间, 2. 研究场地, 3. 获取预算审批, 4. 预订场地, 5. 安排交通...]`。Planner 通常使用单独 LLM 调用 + 专门元提示来生成这些任务。生成的任务列表是 Agent 的核心状态。

**Executor 的职责：** 从任务队列取出第一个任务，使用 ReAct 循环执行它。调用工具（如 SlackMessenger 发送问卷）、观察结果、完成任务后报告。每个任务完成后，将结果写回共享状态，Planner 决定下一步。

**Monitor/Critic——防止跑偏的秘密武器：** 这是最关键的组件。在每个任务后（或每 N 步操作后），独立的 Monitor 模块评估 Agent 是否仍在正轨上。它会问：「给定『组织团建』这个总目标，最近几步操作是否合理？状态是否有意义的变化？」当 Agent 反复尝试预订已被标记为不可用的场地时，Monitor 检测到循环，中断 Executor，强制 Planner 重新计划。

**面试加分：** 「CoPrep 强调 'metacognition'（元认知）是自主 Agent 最难的部分——不是执行循环本身，而是自我校正的架构。Monitor 应该有一个 'loop detection' 功能——如果同一步骤被执行 3 次以上且结果相同，自动触发 plan 更新。CallSphere 也提到 Agent framework 应该严格校验 action 列表，防止 LLM 调用不存在的工具。」

### Q3. Agent 评估框架中，轨迹指标（Trajectory Metrics）和结果指标（Outcome Metrics）分别衡量什么？为什么生产环境需要两者结合？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Galileo、Thinking Company、微软、高级评估工程师
**答：**

Galileo 的评估框架和 Thinking Company 的生产指南都强调了这个关键区分——这是 2026 年 Agent 评估面试的最核心概念。

**结果指标（Outcome Metrics）：** 衡量 Agent「是否完成了任务」。二元指标——成功/失败。例如「客户退款处理」Agent 的结果指标：退款是否成功执行？退款金额是否正确？处理时长是否在 SLA 内？结果是用户真正关心的——「我拿到退款了吗」。但是结果指标有一个致命问题：它告诉你失败了，但不告诉你为什么失败。

**轨迹指标（Trajectory Metrics）：** 衡量 Agent「如何完成任务」。包括：工具调用顺序是否合理？有多少次不必要的重试？Agent 是否绕了弯路？决策链是否合乎逻辑？例如：同一个退款任务，Agent A 用了 3 步（查订单→确认退款→执行），Agent B 用了 12 步（查订单→查订单→查用户→查订单→...→迷路→最终退款）。结果指标上两者都是「成功」，但轨迹指标揭示 Agent B 的生产能力有严重问题。

**为什么两者需要结合：** 结果指标监控「是否可用」，轨迹指标诊断「为什么出问题」。没有结果指标，你不知道 Agent 是不是在正常工作；没有轨迹指标，你无法定位故障原因。Galileo 的框架建议生产过程监控：结果指标作为 SLI 告警，轨迹指标用于持续改进。

**面试加分：** 「Thinking Company 引入了 'Human Intervention Rate'（人工干预率）作为桥梁指标——既反映结果（需要人工介入 = 系统失败），又隐含轨迹问题（Agent 做出糟糕决策导致需要人类纠正）。当人工干预率超过 1% 时，立即分析轨迹指标找出根因。」

### Q4. 在 ReAct 循环中，Agent 如何陷入「幻觉循环」（Hallucination Loop）？有什么技术手段可以从架构层面防止这种问题？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、CoPrep、OpenAI、字节跳动、2026年面试必问高频题
**答：**

CoPrep 的面试指南将这个问题列为「区分候选人的关键问题」——面试官会问：「如何防止 Agent 在 ReAct 循环中陷入幻觉死循环？」

**幻觉循环的形成：** Agent 调用工具获取 Observation，但 LLM 在下一个 Thought 中「编造」了 Observation 不包含的信息，基于幻觉继续行动，产生更错误的结果。典型的循环模式：Agent 问「餐厅是否有空位」→ 工具返回「周二有空」→ Agent 幻觉说「我查了周三的」→ 基于错误时间继续推理→ 越来越偏离轨道。恶性循环是因为 Agent 没有「事实校验」机制来纠正自己的错误解读。

**架构层面的防止手段：**

1. **严格 Schema 约束：** 工具返回必须使用结构化格式（JSON Schema），LLM 无法自由「解读」Observation。Observation 字段中增加 `explicit_facts` 列表，强制 LLM 只能引用该列表中的事实。
2. **观察-断言（Observation-Assertion）模式：** 在每个 Thought 输出前，Agent 必须显式断言「根据 Observation 中字段 X 的值 Y，我认为...」。如果论断与 Observation 不一致，系统注入纠正提示。
3. **独立验证器（Independent Verifier）：** 在每 2-3 步后，用一个独立的轻量 LLM（或规则引擎）检查 Thought 是否与 Observation 一致。发现幻觉时重置到上一个一致的状态。
4. **终止条件（Halting Condition）：** 设置严格的安全网——最大循环次数（如 10 步），超时则降级为「需要人工协助」。

**面试加分：** 「CoPrep 强调工具本身的设计质量是关键——'Your agent is only as good as the feedback from its tools'。一个好的错误消息不是 'API_ERROR'，而是 `'ERROR: Invalid parameter 'location'. Must be latitude/longitude.'`。结构化的观察结果让 LLM 更难产生幻觉。」

### Q5. 什么是 Context Caching（上下文缓存）？在 Agent 系统中如何利用它来降低成本和延迟？有哪些设计权衡？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Anthropic、OpenAI、Google、微软、2026年工程优化新题
**答：**

Context Caching 是 2026 年 LLM 提供商（Anthropic、OpenAI）推出的关键基础设施能力。Anthropic 和 Google 都支持对重复出现的上下文前缀进行缓存。

**核心原理：** Agent 的每次 LLM 调用通常包含一个巨大的 System Prompt（工具 Schema、系统指令、示例），而**这些内容在连续调用之间几乎不变**。Context Caching 允许将这部分「静态前缀」的 KV Cache 复用于后续请求——只计算新消息（用户输入、工具返回）的部分。例如：System Prompt 占 2000 Token，每次用户输入 200 Token。没有缓存时每次消耗 2200 Token；有缓存时只需计算 200 Token + 缓存读取成本，延迟降低 40-60%。

**Agent 系统中的使用场景：**
1. **ReAct 循环中：** 每次 Thought-Action-Observation 迭代中，System Prompt 和对话历史前半段可以缓存。
2. **Multi-Agent 共享上下文：** 多个 Specialist Agent 共享相同的系统规则和工具 Schema，各自维护独立的缓存。
3. **重复任务的短缓存：** 客服 Agent 在对话中反复查询类似数据，缓存「查询工具」相关的 Prompt 片段。

**设计权衡：**
- 缓存有效期：OpenAI 的 Prompt Caching 有 5-10 分钟 TTL；Anthropic 的缓存有更长的保留时间。长时间运行的 Agent 需要刷新缓存。
- 缓存粒度：精确命中需要前缀完全一致——即使多一个空格也会导致 cache miss。
- 成本模型：缓存命中按更低费率计费（约 50-90% 折扣），但缓存写入有额外成本。

**面试加分：** 「Agent 中一个巧妙的优化是 '**缓存感知的 Prompt 构造**'——将最静态的内容放在 Prompt 最前面（系统指令、基础规则），其次是半静态内容（工具 Schema，每几轮才变一次），最后是动态内容（用户输入、最近的 Observation）。这样最大化缓存命中率。CallSphere 提到的 'cost budget' 约束也可以用缓存在满足预算的同时保留更多上下文。」

### Q6. Agent 系统中如何实现「冷热工具分离」设计？如何根据工具的调用模式进行动态缓存和预加载优化？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、LangChain、高吞吐 Agent 团队
**答：**

Agent 的工具集规模在 2026 年迅速膨胀——一些生产 Agent 注册了 50+ 个工具。将所有工具的 Schema 塞入上下文既不现实也不经济。冷热工具分离是解决这个问题的工程模式。

**工具分类：**
- **热工具（Hot Tools）：** 高频调用（>10次/小时）、低延迟要求（<500ms）。例如：搜索、计算器、天气查询。这些工具的 Schema 常驻在 Prompt 中。
- **温工具（Warm Tools）：** 中等频率（1-10次/小时）、中等延迟。例如：数据库查询、文档检索。Schema 在会话开始时加载，但可被紧急工具覆盖淘汰。
- **冷工具（Cold Tools）：** 低频调用（<1次/小时）、或仅特定场景需要。例如：PDF 导出、权限审批、报表生成。Schema 按需动态加载——当 Agent 的推理路径暗示需要某工具时实时注入。

**动态加载策略：**
1. **查询意图预测：** Agent 收到用户消息后，先用一个轻量分类器预测可能需要的工具领域（如「财务类」「通讯类」），只加载该领域的工具 Schema 到上下文。
2. **LRU 缓存：** 维护工具的 LRU（最近最少使用）缓存。热门工具常驻，极少使用的工具自动被淘汰出 Prompt。
3. **延迟绑定（Lazy Binding）：** Agent 的推理步骤完成后才加载决定调用哪个工具——第一次 Tool Call 失败时再加载备选工具。

**面试加分：** 「LangChain 和 MCP 协议都支持 '**Runtime Tool Discovery**'——Agent 在运行时通过 `tools/list` 发现可用工具集。冷工具不需要在 System Prompt 中声明，Agent 在需要时查询。这大幅减少了每轮调用的输入 Token。Anthropic 也强调 'keep the format close to what the model has seen'——工具 Schema 的格式一致性比工具数量更重要。」

### Q7. 什么是模型路由（Model Routing）？如何为 Agent 的不同子任务选择合适大小的模型来平衡成本、质量和延迟？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、OpenAI、Anthropic、高成本敏感 Agent 团队
**答：**

CallSphere 和 Galileo 的评估指南都提到了模型路由作为 Multi-Agent 系统的成本控制关键。模型路由是 2026 年 Agent 系统架构师的核心能力。

**模型路由的核心思想：** 不是所有子任务都需要最强大的模型。Agent 的一个复杂请求可能分解为：查询天气（简单）→ 分析趋势（中等）→ 写市场报告（复杂）。对简单任务用 Haiku/GPT-4o-mini（成本低、速度快），复杂推理用 Opus/Claude 4 Sonnet。

**路由策略：**
1. **静态规则路由：** 根据任务类型硬编码模型映射。例如「工具调用解析 → 小模型」「多跳推理 → 大模型」「总结 → 中等模型」。
2. **动态路由（Model Router Agent）：** 用一个轻量 Router Agent 分析当前子任务的复杂度，动态选择模型。Router 本身跑在最小的模型上，只输出 `{"model": "claude-haiku", "reason": "简单查询无需复杂推理"}`。
3. **级联回退（Cascading Fallback）：** 先尝试小模型 + 严格的格式校验，如果小模型的输出质量不达标（置信度低或格式错误），自动升级到大模型重跑。

**设计要点：**
- 选择器延迟：路由本身的延迟不能超过 50ms，否则得不偿失
- 模型切换的一致性：不同模型可能对同一工具 Schema 的理解有差异，需要统一工具定义格式
- 成本监控：每个任务累计模型调用成本必须跟踪，超过预算阈值自动降级

**面试加分：** 「CallSphere 提到一个关键的数字——Multi-Agent 系统的成本可以比单 Agent 高 5-10 倍，因为每个 Specialist 独立调用 LLM。模型路由是控制这种爆炸的关键。'**Task-specific model selection**'——Agent 中的 'research' 子任务用 Claude Opus（擅长长文推理），'data extraction' 子任务用 GPT-4o-mini（结构化输出能力强），可以节省 60-70% 的成本而质量几乎没有下降。」

### Q8. Agent 如何处理多模态工具的集成？当 Agent 需要同时处理文本、图像、音频、代码执行等不同类型的工具时，架构上需要注意哪些问题？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、Google、Microsoft、Anthropic、多模态 Agent 团队
**答：**

2026 年的 Agent 越来越多地集成多模态工具——视觉分析（OCR/图像理解）、语音转录、代码执行、文件操作等。DataCamp RAG 指南和 CoPrep 也暗示了多模态信息检索的复杂性。

**多模态集成的架构挑战：**

1. **工具调用的输入/输出格式不统一：** 文本工具返回字符串，图像工具返回 base64 编码，音频工具返回二进制流。Agent 需要统一的中间表示（Intermediate Representation）——将所有非文本输出结构化描述化。例如图像分析工具的输出从 base64 直接返回改为 `{"description": "...", "objects": [...], "text_extracted": "..."}` 的结构化 JSON。
2. **上下文窗口的饥饿问题（Context Starvation）：** 图像和长音频的 Token 消耗极大（一张图 ≈ 2000 Token）。Agent 需要在这些多模态内容进入上下文前进行压缩——用 VLM 提取文本摘要，只将摘要传给推理引擎，原始内容存放在外部存储中按需引用。
3. **调度优先级：** 不同类型的工具有不同的 SLA。音频转录可能需要实时返回，图像分析可以稍后处理。Agent 的调度器需要为多模态工具设置不同的超时和重试策略。
4. **错误传播的复杂度增加：** 流水线中一个工具的输出是另一个工具的输入——「音频转录"→"文本分析"→"图像检索"」。前一个工具的微小误差可能在后一个工具中被放大。建议每步做质量检查。

**面试加分：** 「一个有效的模式是 **Modality-Aware Context Budget**——Agent 为文本、图像、代码分别分配上下文预算。例如 32K 上下文中：文本 50%、图像 30%（最多 3 张缩略图摘要）、代码 20%。基于任务类型动态调整比例——代码调试任务给代码 40%，视觉分析任务给图像 50%。Anthropic 提到 'model-native tool use' 在多模态场景中特别重要——模型本身理解图像、代码、文本，才能做出正确的多模态路由决策。」

### Q9. Agent 的动作预算（Action Budget）和时间预算（Time Budget）如何设计？如何在运行时检测和终止「失控 Agent」？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | CallSphere、Anthropic、OpenAI、微软、生产安全高频题
**答：**

CallSphere 的 Agent 安全指南将预算机制作为「Agent 安全栈」的第三层——仅次于 action classification 和 sandboxing。这是防止 Agent 造成真实世界损害的最后一道防线。

**三重预算设计：**

1. **动作预算（Action Budget）：** Agent 单次任务中允许的最大工具调用次数。简单任务（查询天气）设上限 3-5 次；复杂任务（调研竞品）设 15-20 次。超限时强制降级——Agent 必须用已有信息回答，不能再发起新调用。
2. **Token 预算（Token Budget）：** 最大 Token 消耗量。防止 Agent 在循环中浪费大量 Token（每调用一次 LLM 就消耗几千 Token）。建议按模型分层设置——小模型 10K Token 预算，大模型 50K Token 预算。
3. **时间预算（Time Budget）：** 硬超时（Hard Timeout）。复杂任务设 5 分钟，超出后 Agent 必须返回当前结果或声明失败。时间预算优先于动作预算——即使动作次数未超限但时间到了也终止。

**失控检测机制：**

- **循环检测（Loop Detection）：** 分析最近的工具调用模式——如果 Agent 连续 3 次调用同一工具且参数完全相同，触发循环警报。
- **非预期行为检测：** Agent 尝试调用未注册的工具、访问未授权的数据域、或试图绕过安全约束，立即终止。
- **成本异常告警：** 单次任务的 Token 消耗超过历史均值的 3 倍标准差时告警。

**面试加分：** 「Anthropic 强调一个关键原则——'**All budget limits should be hard ceilings, not soft suggestions**'。当 Agent 接近预算上限（80%）时，系统可以注入一个提醒；但到达 100% 时必须强制执行。同时，Action Classification——对工具调用按风险分级：只读操作（10 次限额）、低风险写入（5 次限额）、高风险操作（1 次且需要人工确认）。Thinking Company 也建议 'Human Intervention Rate' 作为预算设计的参考指标——如果高频触发预算上限，说明预算过于严格，需要调整。」

### Q10. Agent 在生产中如何实现持续评估与回归测试？描述一个类似于 CI/CD 的 Agent 评估 Pipeline。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Galileo、Thinking Company、LangChain、微软、高级评估工程师
**答：**

Galileo 和 Thinking Company 都强调：Agent 的评估必须是「自动化和持续」的，而不是「手动和周期性」的。每一次 Prompt 修改、工具变更、模型切换都必须触发评估。

**Agent 评估 Pipeline（类似 CI/CD）：**

```
代码/配置变更 → 触发 Eval Suite → （并行执行 3 层测试）→ 生成报告 → 自动门控（通过/拒绝）
```

**三层测试架构：**

**第一层 — 确定性测试（Deterministic Tests）：** 针对有明确预期输出的任务。例如「预约 3 月 30 日 14:00 的会议」→ 测试 Agent 是否正确调用了 booking 工具，参数是否正确。这部分完全自动化，执行速度快。建议覆盖 50+ 个基准场景。

**第二层 — LLM-as-Judge 测试：** 针对需要质量评估的开放任务。例如「调研并总结最近 AI 安全论文」→ 用另一个 LLM（如 Claude Opus）打分：相关性、完整性、准确性。建议使用 Galileo 的评分 Rubric——结构化评分标准，而不是模糊的「好不好」。每项指标 1-5 分，设定通过线（如总分 > 4.0）。

**第三层 — 人类评估（Human Evaluation）：** 对真实用户交互的随机样本进行人工评分。周期性执行（如每周一次），评估维度包括：有用性、安全性、效率。这是黄金标准，但成本高、频率低。

**回归测试门控（Gating）：** 每次变更后，Agent 必须在评估套件上的得分不低于基线（或指定的容忍度，如 2% 以内）。如果下降则拒绝部署。同时对抗性评估（Adversarial Evaluation）也很重要——系统性地修改测试任务（改变名字、数字、增加干扰项），检查 Agent 是否依然稳定。

**面试加分：** 「Galileo 推荐 **'Shadow Testing'（影子测试）**——新 Agent 配置在影子模式下运行，与生产 Agent 的输出对比但不影响用户。如果影子 Agent 在所有指标上都持平或更好才正式上线。这防止了 '越改越差' 的悲剧。Thinking Company 还建议将评估结果汇总到一个仪表板，追踪长期趋势——如果 Agent 的某项指标连续 3 天下滑，自动触发告警并回滚到上一个稳定版本。」
