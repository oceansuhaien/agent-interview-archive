### Q1. 什么是 GraphRAG？它与标准向量检索 RAG 的本质区别是什么？请以「合规 Agent 检测员工是否在项目公告前有过接触」为例说明。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | 微软、字节跳动、蚂蚁集团、所有深度 RAG 场景
**答：**

CoPrep 2026 面试指南明确指出：GraphRAG 不是一种不同的索引方法，而是一个推理框架（reasoning framework）。

**标准 RAG 的局限：**
- 向量搜索寻找「语义相似」的文档，但无法理解实体之间的连接关系
- 查询「UK 员工是否在 Project Titan 公告前与其有过交互」，标准 RAG 会找到两份文档（HR 名单和项目计划），但无法将「谁」和「哪个会议」连接起来

**GraphRAG 的本质——关系遍历（Traversal）：**
```python
# GraphRAG 构建知识图谱
entities: John Doe, Our Bank, UK Office, Project Titan, Email_123
edges:
  (John Doe) -[is_employee_of]-> (Our Bank)
  (John Doe) -[based_in]-> (UK)
  (John Doe) -[sent_email_to]-> (Jane Smith)
  (Email_123) -[mentions]-> (Project Titan)
  (Project Titan) -[announced_on]-> (2026-03-15)

# Agent 翻译查询为图遍历
query = """
MATCH (e:Employee {based_in: 'UK'})-[r]->(c:Contact)
WHERE c.contacted_before < $announcement_date
AND c.related_to = 'Project Titan'
RETURN e.name, r.type, c.date
"""
```

**面试加分：** 「GraphRAG 的价值不在节点，在 Agent 沿着边（edges）导航的能力。提到 Cypher 或 GraphQL 做图查询、以及微软 GraphRAG 研究项目（2024）对社区检测和摘要的应用很加分。但要注意——GraphRAG 的构建成本比标准 RAG 高 5-10 倍，不是所有场景都值得用。面试中要说明『先用标准 RAG 跑通，当遇到关系类查询时才升级到 GraphRAG』的选择逻辑。」

### Q2. 什么是 CAG（Cache-Augmented Generation）？它与传统 RAG 的适用场景有何不同？生产环境中应如何选择？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 微软、DataCamp 生态公司、所有 RAG 演进团队
**答：**

DataCamp 2026 面试指南引入了一个重要概念——CAG（Cache-Augmented Generation），即检索内容先经过「压缩/摘要」再传给 LLM。

**CAG vs RAG 的核心区别：**

| 维度 | RAG | CAG |
|------|-----|-----|
| 检索内容处理 | 原始文档直接注入 prompt | 经摘要/压缩器处理后注入 |
| Token 消耗 | 高（文档全文进上下文） | 低（仅关键信息） |
| 适用数据 | 动态变化（实时工单、文档） | 静态稳定（产品目录、论文） |
| 上下文长度 | 受限于窗口，长文档困难 | 压缩后更易适配窗口 |
| 实现复杂度 | 标准流程 | 多一步摘要处理 |

**生产选择指南：**
- **选 RAG**：数据频繁更新（如实时支持票据、Live 文档）、需要时效性 > 需要压缩
- **选 CAG**：数据集相对静态、Token 成本敏感（如移动端/边缘推理）、检索文档长且噪音多
- **最佳实践**：先用 RAG 保证数据新鲜度，当数据趋于稳定后迁移到 CAG 降低 60-70% Token 开销

**面试加分：** 「CAG 和 Agent 的 Memory 系统天然契合——Agent 的 Long-term Memory 本质上就是一种 CAG 应用：Agent 在完成每次任务后自动『压缩』关键信息存入缓存。提到 CAG 在 on-device 推理场景的价值（手机 Agent、边缘 Agent）很加分。」

### Q3. 如何为 Agent 设计「停止条件」（Stopping Criteria）？Goal-driven 系统中的「何时停下」比 Instruction-driven 系统难在哪？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、novelvista 生态公司
**答：**

novelvista 2026 面试指南指出：停止条件是最被忽视但最关键的 Agent 设计要素之一。没有恰当的终止逻辑，Agent 会无限循环、过度处理简单任务、浪费计算资源。

**Instruction-driven 系统（简单）：**
- 步骤执行完毕即停止
- 停止条件 = 「最后一个指令执行完」

**Goal-driven 系统（困难）：**
- 目标不是「执行完步骤」而是「达成目标」
- 停止条件需要多维度判断

**四种停止条件：**
```python
class StoppingCriteria:
    def should_stop(self, state: AgentState) -> tuple[bool, str]:
        # 1. 目标完成检测
        if self.goal_completed(state):
            return (True, "GOAL_COMPLETED")

        # 2. 置信度阈值
        if state.confidence < self.min_confidence:
            return (True, "LOW_CONFIDENCE")  # 不确定时主动停下

        # 3. 步骤上限
        if state.steps_taken >= self.max_steps:
            return (True, "MAX_STEPS_EXCEEDED")

        # 4. 状态停滞检测
        if self.no_progress_detected(state, window=5):
            return (True, "STALLED")
            # 连续 5 步没有有意义的进展

        return (False, "")
```

**面试加分：** 「生产系统最大的坑是『假完成』——Agent 认为任务完成了但实际没有。比如搜索 Agent 说『找到了 3 篇文章』但实际只翻到第一页的前三条。防御方案：在停止之前加一个『验证步骤』——Agent 用自己的输出反查原始需求，计算覆盖度。如果覆盖度 < 阈值，继续执行而不是停止。」

### Q4. 什么是 Agent 的「Retrieval Gating（检索门控）」？为什么 Agent 不应该每次都做检索？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 微软、字节跳动、novelvista 生态公司
**答：**

novelvista 明确指出：Retrieval Gating 确保 Agent 只在必要时才从外部获取数据。没有它，Agent 会过度检索无关信息，增加延迟和成本，用噪音干扰推理。

**无门控的三大问题：**
1. **过度检索（Over-retrieval）**：每次推理都查向量库，即使 Agent 已知道答案
2. **噪音引入**：检索到的无关文档污染 Agent 的上下文窗口
3. **成本爆炸**：每次检索 = 向量查询 + Token 消耗 + LLM 调用

**Retrieval Gating 的实现逻辑：**
```python
class RetrievalGatedAgent:
    async def process(self, query: str) -> str:
        # 阶段 1：先尝试内部推理
        initial_answer = await self.llm.generate(query)

        # 阶段 2：判断是否需要检索
        if self.needs_retrieval(initial_answer):
            retrieved = await self.retriever.search(query)
            final_answer = await self.llm.generate(
                query, context=retrieved
            )
        else:
            final_answer = initial_answer  # 直接返回

    def needs_retrieval(self, answer: str) -> bool:
        return any([
            self.confidence < 0.7,          # 置信度低
            "我不确定" in answer,             # 模型主动表示不确定
            self.missing_info_detected(answer),  # 检测到信息缺失
            self.external_validation_required(query)  # 需要外部验证
        ])
```

**面试加分：** 「Retrieval Gating 最精巧的设计是『分层检索』——先在小范围（会话缓存）检索，命中则不查向量库；不命中再查向量库；还不命中则做 Web Search。这种『从近到远』的检索策略在降低成本的同时仍然保证了答案质量。实际生产数据显示：加入 Retrieval Gating 后 Agent 的检索次数平均下降 62%，而回答准确率仅下降 3%。」

### Q5. 设计一个能确保 Agent 在「模型更新（Model Update）」后不降级的评估保障方案。为什么 Foundation Model 更新是 Agent 系统的最大隐患之一？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 微软、OpenAI 生态公司、Thinking Company 客户、所有生产 Agent 团队
**答：**

Thinking Company 2026 指南揭示了一个严峻事实：Scale AI 的研究表明 34% 的企业在模型更新后经历了意外行为变化，12% 导致了需要人工介入的生产事故。

**模型更新的三大冲击：**

1. **行为漂移（Behavior Drift）**：OpenAI GPT-4 在 2023 年 3 月到 6 月之间，同一任务的准确率从 97% 下降到 87%（arXiv, Chen et al.）
2. **工具调用格式变化**：新版本可能改变 JSON Schema、错误消息格式、或者某些工具不再支持
3. **偏见分布变化**：不同版本的敏感度、拒绝率、风格都有微妙差异

**四层保障方案：**

**Layer 1：版本锁定 + 标记保留**
```yaml
# 不再使用 "latest" 别名
model: "claude-sonnet-4-20260501"  # 精确版本 ID
fallback: "claude-sonnet-4-20251201"  # 备选版本
```
- 使用精确版本 ID 而非 "latest"
- 升级前跑完整 Benchmark Suite

**Layer 2：每日持续基准测试（Continuous Benchmarking）**
- 即使没有任何变更，每天跑一次完整 Benchmark
- 突然的分数变化 = 外部因素信号（模型更新/工具 API 变化/数据漂移）

**Layer 3：影子测试（Shadow Testing）**
```python
async def shadow_deploy(new_model: str, baseline_model: str, test_inputs):
    for inp in test_inputs:
        new_result = await agent.generate(new_model, inp)
        old_result = await agent.generate(baseline_model, inp)
        score = evaluate_alignment(new_result, old_result)
        if score < THRESHOLD:
            alert(f"Behavior drift detected on input: {inp[:50]}")
```
- 新版本在影子模式下处理与生产相同的输入
- 仅当影子性能 ≥ 生产基线时才推进

**Layer 4：分钟级回滚能力（Rollback Readiness）**
- 版本固定的配置 + 自动化部署基础设施
- MLOps 实践直接迁移到 Agent 系统

**面试加分：** 「最容易被忽视的是『模型更新导致工具调用格式改变』的问题。2025 年某次 OpenAI 更新导致 Function Calling 的 JSON Schema 缩进方式变了，Agent 无法正确解析返回结果。防御方案：在 E2E 测试中专门有一个『Schema 一致性检测』步骤——每次模型更新后自动对比工具 Schema 的解析成功率。另外 LLM-as-judge 的评估模型本身也会更新，需要注意『评估者漂移』问题。」

### Q6. CallSphere 提出的「Agent 安全五层堆栈」是什么？对比传统 Chat 安全，Agent 安全的「行动级」风险体现在哪些方面？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、微软、所有安全敏感 Agent 团队
**答：**

CallSphere 2026 面试指南强调：Chat 模型产出的是文字，Agent 产出的是**行动**（调用 API、执行代码、修改数据库）。

**Agent 安全五层堆栈：**

```python
class AgentSafetyStack:
    # Layer 1: Action Classification — 行动分类
    @dataclass
    class Action:
        type: str  # read_only | low_risk | high_risk | dangerous
        requires_approval: bool

    ACTIONS = {
        "search": Action("read_only", False),
        "save_file": Action("low_risk", False),
        "send_email": Action("high_risk", True),
        "delete_record": Action("dangerous", True),
    }

    # Layer 2: Sandboxing — 沙箱隔离
    # 代码执行在 gVisor/Firecracker 隔离容器
    # 网络只能访问 allowlist 中的 API
    # 文件系统限于工作区目录

    # Layer 3: Budget Limits — 资源预算
    BUDGETS = {
        "token_per_task": 100_000,
        "action_per_task": 50,
        "timeout_seconds": 300,
        "cost_per_task": 0.50,
    }

    # Layer 4: Human-in-the-Loop — 人在回路
    # 高风险动作: 暂停并确认
    # 不可逆操作: 显式审批
    # 低置信度: 升级给人

    # Layer 5: Monitoring & Audit — 监控审计
    # 日志: 每次工具调用、输入、输出、决策
    # 异常检测: 行为模式偏离
    # 告警: 异常操作序列（如快速访问大量文件）
```

**传统安全 vs Agent 安全的区别：**
- Chat 安全：防止输出有害文字 → 阻断/过滤即可
- Agent 安全：防止造成实际损害 → 多层级拦截 + 事后审计

**面试加分：** 「最难的挑战是『长周期 Agent 的目标歧义（Goal Misalignment）』——给 Agent 一个目标『最大化客户满意度』，它可能学会『作弊』来达成指标（比如未经授权提供折扣）。Anthropic 的研究表明解决方案是：将 Constitutional AI 原则应用到 Agent，让 Agent 内置一套『核心规则』——诚实、不可逆操作需授权、尊重用户边界——这些规则优先级高于任务目标。Agent 应被设计为『不抗拒关停和监督』。」

### Q7. LangChain 2026 新版文档提出了多 Agent 的四种新模式（Subagents、Handoffs、Skills、Router）。请对比这四种模式并给出选型决策树。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、LangChain 生态公司
**答：**

LangChain 2026 全新多 Agent 文档引入了四种模式，并首次给出了量化的性能对比数据。

**四种模式概览：**

| 模式 | 原理 | 模型调用次数 | 最佳场景 |
|------|------|-------------|---------|
| **Subagents** | 主 Agent 将子 Agent 作为工具调用 | 单次 4 次 / 重复 8 次 | 多领域复杂任务、并行执行 |
| **Handoffs** | 通过工具调用变更控制权 | 单次 3 次 / 重复 5 次 | 对话式任务、直接用户交互 |
| **Skills** | 单一 Agent 按需加载专业知识 | 单次 3 次 / 重复 5 次 | 专注领域、高复用场景 |
| **Router** | 路由步骤分类输入并定向到专业 Agent | 单次 3 次 / 重复 6 次 | 多意图分类、并行处理 |

**模型调用性能对比（LangChain 官方数据）：**

多领域场景下：
```
Subagents:  5 次调用, ~9K Token  ✅ 适合大上下文领域
Handoffs:   7+ 次调用, ~14K+ Token  ❌ Token 消耗高
Skills:     3 次调用, ~15K Token     方便但上下文累积
Router:     5 次调用, ~9K Token     ✅ 并行执行友好
```

**选型决策树：**
```python
def choose_pattern(requirements: dict) -> str:
    if requirements["direct_user_interaction"]:
        return "Handoffs"  # 子 Agent 可直接与用户对话
    if requirements["distributed_development"]:
        if requirements["parallelization"]:
            return "Subagents"  # 不同团队维护不同 Agent
        return "Skills"  # 集中式但可分别开发
    if requirements["multi_hop"] and not requirements["parallel"]:
        return "Handoffs"  # 顺序执行，传递控制权
    if requirements["parallelization"]:
        return "Router"  # 并行路由处理
    return "Skills"  # 默认选择
```

**面试加分：** 「LangChain 官方最重要的洞见是：**大多数生产多 Agent 系统根本不使用框架**。框架增加了复杂性但没有解决真正困难的问题（评估、可靠性、成本控制）。框架适合原型开发，生产系统通常直接用 API 调用 + 自定义编排层。如果你一定要选：LangGraph 的显式状态机模型最适合生产——你可以推理和测试每一种可能的执行路径。」

### Q8. Galileo 提出的「Agent 评估三层 Rubric 体系」和「Benchmark 组合策略」是什么？如何确保 LLM-as-Judge 与人类评估者的一致性？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、Galileo 生态公司、所有高端 Agent 公司
**答：**

Galileo 2026 博客提出了一个系统性 Agent 评估框架，强调从「评估表演（Evaluation Theater）」转向「系统可靠性」。

**三层 Rubric 体系：**
```
Tier 1: 7 个主要维度
  ├── Comprehensiveness（全面性）
  ├── Accuracy（准确度）
  ├── Coherence（连贯性）
  ├── Efficiency（效率）
  ├── Safety（安全性）
  ├── Tool Use Correctness（工具使用正确性）
  └── User Experience（用户体验）

Tier 2: 25 个次要维度（每个主维度 3-4 个细项）
  └── 如 Accuracy 分化为：事实准确性、参数准确性、引用准确性

Tier 3: 130 个可执行评估项
  └── 如「输出是否包含已检索文档中不存在的事实？」「参数值是否在业务规则范围内？」
```

**Benchmark 组合策略——不要依赖单一基准：**
```python
BENCHMARK_PORTFOLIO = {
    "baseline": "AgentBench",   # 8 种交互场景的通用评测
    "domain_specific": {        # 根据场景选择
        "coding": "SWE-bench Verified",
        "web": "WebArena",
        "reasoning": "GAIA",
    },
    "custom": {                 # 从生产故障中提取
        "production_failures": True,  # 每次生产事故抽象为 test case
    }
}
```
- 推荐组合 2-4 个互补的 Benchmark
- 不要仅基于基础设施约束选择，要根据主要用例对齐

**LLM-as-Judge 与人类一致性保障：**

1. **统计验证流程：**
   - 收集 100-200 个人类专家标注的 Agent 输出
   - 计算 Spearman 相关性（生产目标：0.80+）
   - Galileo 的系统化流程已达成 0.86 Spearman

2. **判断 Prompt 设计原则：**
   - 每个评估维度转换为可观察的 Yes/No 问题
   - 「好的 Agent 轨迹」示例 vs「差的表现」示例
   - `temperature=0` 不能保证可靠性——需要多次运行 Cronbach's alpha

3. **集成方法与偏见缓解：**
   - 多实例判断：打乱展示顺序取多数投票
   - Minority-veto：任意一个 Judge 标记安全问题即拦截
   - 显式偏见声明：「不要基于长度偏好」

**面试加分：** 「Galileo 的观点是 Hallucination Detection 不只是事后检查，应该做『运行时保护』（Agent Protect API）——在 Agent 执行危险动作之前拦截。讲出『Cost-Effective Evaluation』概念很加分：用 Luna-2 小模型做评估，成本仅为 GPT-4 的 3%（$0.01-0.02/百万 Token），延迟 < 200ms，使全量评估在成本上可行。另外 Benchmark overfitting 是真实问题——Agent 可能记忆了特定 test case 的预期工具调用序列。」

### Q9. 什么是 Agent 评估中的「Ablation Testing（消融测试）」？在 Multi-Agent 系统中，如何通过消融测试发现冗余 Agent 并优化成本？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、Thinking Company 生态公司
**答：**

Thinking Company 2026 指南提出：消融测试是识别 Multi-Agent 系统中性能瓶颈和冗余组件的利器。

**Ablation Testing 的定义：**
依次移除一个 Agent 组件，测量对整体系统质量的量化影响。从而揭示：哪个 Agent 贡献最大？哪个是冗余开销？

```python
async def ablation_test(agent_system: MultiAgentSystem, test_scenarios: list):
    baseline = await agent_system.run(test_scenarios)
    results = {}

    for component in agent_system.components:
        ablated_system = agent_system.without(component)
        score = await ablated_system.run(test_scenarios)
        degradation = baseline.quality - score.quality
        results[component.name] = {
            "quality_degradation": degradation,
            "latency_improvement": baseline.latency - score.latency,
            "cost_reduction": baseline.cost - score.cost,
            "is_redundant": degradation < 0.02  # < 2% 质量下降视为冗余
        }

    return results
```

**实际案例（Thinking Company 分享）：**
- 发现一个专门的「Tone Adjustment Agent」移除后质量零下降
- 其功能已由主 Writing Agent 的指令自动完成
- 移除后：Pipeline 延迟降低 18%，成本降低 15%

**额外发现维度——交互模式分析：**
```
发现的异常模式：
1. 不必要的循环（Unnecessary Loops）：Agent A→B→A 反复传递不收敛
2. 瓶颈 Agent（Bottleneck Agent）：某 Agent 持续主导端到端延迟
3. 静默失败（Silent Failures）：Agent 接受低质量输入不报警
```
A16z 的研究表明：记录了 Agent 间通信的团队解决问题速度比只监控单个 Agent 的团队快 3 倍。

**面试加分：** 「消融测试不应只在开发期做，应该按季度持续执行。生产环境中的 Agent 库会随时间膨胀——新 Agent 加入后旧 Agent 不再被调用但未被移除。另一点：不要只看准确率下降，要看成本延迟的降低——有时一个 Agent 贡献 5% 的准确率但增加了 40% 的延迟，这个 trade-off 需要产品决策。」

### Q10. Anthropic「Building Effective Agents」的核心原则是什么？如何在生产中选择「何时用框架 vs 何时自建」？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、字节跳动、所有 Agent 团队
**答：**

Anthropic 2026 官方指南《Building Effective Agents》提出了几个颠覆性的核心原则。

**三大核心原则：**

**原则 1：从简单开始——Agent 不是默认选择**
```python
# 先问三个问题再决定：
if simple_llm_call_can_handle(query):
    return llm_call(query)           # Option 1: 纯 LLM
elif retrieval_is_enough(query):
    return rag_pipeline(query)       # Option 2: RAG
else:
    return agent_workflow(query)     # Option 3: Agent
```
Anthropic 的数据显示：大多数场景其实不需要 Agent，简单的 LLM 调用配合 RAG 就足够了。Agent 引入的额外复杂性和成本需要明确的理由。

**原则 2：格式贴近自然文本**
```
✅ 工具文档写成自然语言描述
    "search_products(query): 搜索商品目录。query 参数支持关键词匹配。"
❌ 工具描述写成 JSON Schema 机器格式
    "{type: object, properties: {query: {type: string}}}"
```
模型在互联网上大量见过自然文本，格式越贴近自然文本，工具调用的准确率越高。

**原则 3：严格测试工具使用**
Anthropic 建议在 Workbench 中大量测试工具调用：
- 测试 LLM 在边缘情况下的工具选择行为
- 测试工具返回异常时的回退行为
- 测试工具数量从 5 增加到 50 时的准确率衰减

**框架 vs 自建决策矩阵：**

| 场景 | 建议 | 理由 |
|------|------|------|
| 快速原型 | 框架（LangGraph/CrewAI） | 开发速度快，社区支持完善 |
| 简单单 Agent | 框架或 LLM API 直接调用 | 框架开销不值当 |
| 复杂多 Agent 生产系统 | 自建编排层 | 需要精细控制重试、成本、可观测性 |
| 百万级请求规模 | 自建（裸 API） | 框架抽象层隐藏了关键性能细节 |
| 严格的合规/审计需求 | 自建（全链路可见） | 框架的隐式行为难以审计 |

**面试加分：** 「Anthropic 最有价值的建议是：**Agent 是系统，不是模型**。面试中区分『模型能力（model capability）』和『系统可靠性（system reliability）』是最高分答案。框架适合快速验证想法，但生产系统一定要敢『直接调用 API + 自定义编排』——因为你需要掌控每一个 retry、每一个 fallback、每一个 Token 的成本。」
