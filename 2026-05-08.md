### Q1. LLM 的 Function Calling 在生产环境中有哪些常见的可靠性问题？你会如何防护？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | OpenAI 生态公司、字节跳动、Anthropic 生态公司
**答：**

Function Calling 看起来简单，但在生产环境中坑很多。Anthropic 的官方指南明确指出："Test how the model uses your tools: Run many example inputs in our workbench to see what mistakes the model makes."

**常见可靠性问题及防护：**

| 问题类型 | 具体表现 | 防护方案 |
|---------|---------|---------|
| 参数幻觉 | LLM 编造不存在的 ID、名称 | Schema 强校验 + 正则匹配 |
| 格式错误 | 数字传成字符串、日期格式不对 | Pydantic Validator 前置校验 |
| 工具选择错误 | 应该调 A 工具却调了 B | 工具描述优化 + Few-shot 示例 |
| 参数语义错误 | 语法正确但业务含义错 | 业务上下文二次校验 |
| 超时/死循环 | 反复调用同一个工具重试 | 最大重试次数 + 退避策略 |

**防护代码示例：**

```python
class ToolCallGuard:
    def __init__(self, max_retries=3, timeout=30):
        self.max_retries = max_retries
        self.timeout = timeout
        self.call_log = []

    def safe_call(self, tool_name: str, params: dict, context: dict):
        # 1. Schema 校验
        validate_schema(tool_name, params)

        # 2. 业务语义校验
        business_validate(tool_name, params, context)

        # 3. 重复调用检测（防止死循环）
        recent = [c for c in self.call_log[-5:]
                  if c["tool"] == tool_name and c["params"] == params]
        if len(recent) >= 2:
            raise LoopDetectedError(f"疑似死循环：{tool_name} 被连续调用相同参数")

        # 4. 带超时执行
        result = asyncio.wait_for(
            execute_tool(tool_name, params),
            timeout=self.timeout
        )

        # 5. 结果合理性校验
        if not result.success:
            return self._handle_failure(tool_name, params, result)

        self.call_log.append({"tool": tool_name, "params": params})
        return result
```

**Anthropic 的建议：** 保持工具描述格式贴近模型在训练数据中自然见到的文本格式，而不是用复杂的自定义 Schema。工具越多，测试越要充分。

### Q2. 什么是 Agentic Workflow？请对比 Prompt Chaining、Routing、Parallelization、Orchestrator-Workers 四种设计模式。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic 生态公司、字节跳动、微软
**答：**

Anthropic 在 "Building Effective Agents" 指南中明确区分了 Agentic Workflow 和完全自主的 Agent。Agentic Workflow 是预定义的工作流，LLM 参与其中但不是完全自主决策。

**四种设计模式对比：**

| 模式 | 核心思想 | 适用场景 | Token 效率 |
|------|---------|---------|-----------|
| Prompt Chaining | 任务分解为固定步骤串行执行 | 流程确定的任务（如写作→审稿→润色） | 中 |
| Routing | 用 LLM 做意图分类，再路由到专门子流程 | 多类型输入（如客服分部门） | 高 |
| Parallelization | 多个子任务并行执行后汇总 | 可解耦的独立子任务 | 低（并发） |
| Orchestrator-Workers | LLM 动态拆解任务并分配给 Worker | 复杂、不可预知的任务结构 | 低 |

**代码示例：**

```python
# Prompt Chaining：串行流水线
def prompt_chain(user_input):
    draft = llm.generate(f"写初稿：{user_input}")
    reviewed = llm.generate(f"审阅并指出问题：{draft}")
    final = llm.generate(f"根据审阅意见修改：{reviewed}")
    return final

# Routing：意图路由
def routing(user_input):
    intent = llm.classify(user_input, categories=["billing", "tech_support", "sales"])
    handlers = {"billing": billing_agent, "tech_support": tech_agent, "sales": sales_agent}
    return handlers[intent].handle(user_input)

# Parallelization：并行执行
async def parallel(user_input):
    tasks = [
        fact_check(user_input),
        style_check(user_input),
        safety_check(user_input)
    ]
    results = await asyncio.gather(*tasks)
    return synthesize(results)
```

**关键洞察：** Anthropic 建议从最简单的模式开始，只有当简单模式无法满足需求时才引入更复杂的 Agent 模式。过度工程化是新手最常犯的错误。

### Q3. 生产环境中如何优化 Agent 系统的 LLM 调用成本？请给出至少 5 种策略。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 快手、美团、阿里云
**答：**

Agent 系统的成本比传统 Web 服务高 10-100 倍，因为一个用户请求可能触发多次 LLM 调用。成本优化是面试中区分"做过生产"和"只做过 demo"的分水岭。

**7 种成本优化策略：**

**1. 模型分级路由（最关键的策略）**
```python
def select_model_by_complexity(query: str) -> str:
    complexity = classify_complexity(query)
    if complexity == "simple":
        return "gpt-4o-mini"    # 成本约 $0.15/1M tokens
    elif complexity == "medium":
        return "gpt-4o"         # 成本约 $2.5/1M tokens
    else:
        return "o3"             # 成本约 $10/1M tokens，仅用于复杂推理
```

**2. 语义缓存（Semantic Cache）**
```python
class SemanticCache:
    def get_or_compute(self, query: str, threshold=0.95):
        cached = self.vector_db.search(embed(query), top_k=1)
        if cached and cached[0].score > threshold:
            return cached[0].response  # 命中缓存，零成本
        response = llm.generate(query)
        self.vector_db.insert(embed(query), response)
        return response
```

**3. Context 压缩**：用小模型对长对话做摘要，减少输入 Token。

**4. 结构化输出（Structured Output）**：强制 JSON 输出，避免 LLM 啰嗦。

**5. 工具 Schema 精简**：只传必填参数的 Schema，可选参数用 hint 说明。

**6. 批处理优化**：多个独立查询合并为一个 Prompt 批处理。

**7. 置信度提前退出**：简单问题用 few-shot 直接回答，不走完整 Agent 循环。

**Benchmark：** 综合使用上述策略可降低 60-80% 的 LLM 调用成本，同时保持 95% 以上的任务完成率。

### Q4. 什么是 Guardrails？如何为 Agent 系统构建多层输入输出安全防护？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、所有金融/医疗 AI 公司
**答：**

Guardrails 是 Agent 系统的安全围栏，确保 LLM 的输入和输出都在可控范围内。2026 年 NIST AI RMF 和欧盟 AI Act 都要求高风险 AI 系统必须有明确的 Guardrails。

**四层 Guardrails 架构：**

**第 1 层：输入防护（Input Guardrails）**
- Prompt Injection 检测：用分类模型或规则引擎检测恶意注入
- PII 脱敏：在送入 LLM 前自动脱敏身份证号、手机号等敏感信息
- 意图分类：检测越权操作（如普通用户试图执行管理员命令）

**第 2 层：推理防护（Inference Guardrails）**
```python
class InferenceGuard:
    def check(self, prompt: str, response: str, context: dict) -> GuardResult:
        checks = [
            self.check_hallucination(response, context),  # 幻觉检测
            self.check_toxicity(response),                # 有害内容检测
            self.check_relevance(response, prompt),       # 相关性检测
            self.check_format(response, expected_format), # 格式合规检测
        ]
        return GuardResult(
            passed=all(c.passed for c in checks),
            violations=[c for c in checks if not c.passed]
        )
```

**第 3 层：输出防护（Output Guardrails）**
- URL 白名单：只允许输出预期域名的链接
- 敏感信息泄露检测：检查输出是否包含 API Key、密码等
- 合规审核：医疗/金融场景下的合规声明自动附加

**第 4 层：行为防护（Behavioral Guardrails）**
- 最大迭代次数限制：防止 Agent 陷入死循环
- 工具调用频率限制：防止滥用外部 API
- 成本预算限制：单次会话 Token 上限

**关键提醒：** Guardrails 不是万能的。最有价值的防护是"深度防御"——即使某一层被绕过，其他层仍然能拦住。

### Q5. ReAct（Reasoning + Acting）模式的核心原理是什么？它和纯推理、纯行动模式有什么区别？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 百度、小红书、Anthropic 生态公司
**答：**

ReAct 是 2022 年 Yao et al. 提出的范式，也是 2026 年绝大多数 Agent 框架的底层推理模式。

**核心原理：** 每一步都同时进行"推理"（Thought）和"行动"（Action），推理指导行动，行动的结果又影响下一步推理。

**三种模式对比：**

| 维度 | 纯推理（CoT） | 纯行动（Act-only） | ReAct |
|------|--------------|-------------------|-------|
| 流程 | Think → Answer | Act → Observe → Act | Think → Act → Observe → Think → ... |
| 信息来源 | 仅靠 LLM 内部知识 | 依赖外部工具返回 | 推理 + 外部信息结合 |
| 错误恢复 | 无法修正（推理完即输出） | 可能盲目重试 | 推理分析失败原因后调整策略 |
| 可解释性 | 高（有推理链） | 低（只有操作日志） | 最高（推理+操作都有） |
| Token 效率 | 低（一次性推理） | 中 | 较高（多轮交互） |

**ReAct 的典型执行流程：**

```
用户：帮我查一下特斯拉最近一个季度的营收和同比增长

Thought 1: 我需要查找特斯拉最新季度的营收数据。先搜索一下。
Action 1: search("Tesla Q1 2026 revenue")
Observation 1: Tesla Q1 2026 revenue was $25.17B

Thought 2: 我找到了 Q1 2026 的营收。现在需要找去年同期数据来计算同比增长。
Action 2: search("Tesla Q1 2025 revenue")
Observation 2: Tesla Q1 2025 revenue was $21.30B

Thought 3: 两个数据都拿到了。同比增长 = (25.17 - 21.30) / 21.30 ≈ 18.2%
Final Answer: 特斯拉 2026 Q1 营收 251.7 亿美元，同比增长约 18.2%
```

**面试加分：** 提到"ReAct 的最大风险是推理幻觉引导错误行动，解决方案是在关键决策点加入事实锚定（Grounding）——强制 Agent 先查数据再推理"。

### Q6. 如何为 Agent 设计一个带 checkpoint 的状态机，支持长时间运行和断点恢复？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、LangChain 生态公司
**答：**

生产环境中的 Agent 任务可能运行数分钟甚至数小时（如数据分析、文档审核），必须支持暂停、恢复和故障恢复。

**状态机设计：**

```python
from enum import Enum
from dataclasses import dataclass
import json

class AgentStatus(Enum):
    PLANNING = "planning"
    EXECUTING = "executing"
    WAITING_APPROVAL = "waiting_approval"
    WAITING_INPUT = "waiting_input"
    COMPLETED = "completed"
    FAILED = "failed"

@dataclass
class AgentCheckpoint:
    task_id: str
    status: AgentStatus
    step_index: int
    completed_steps: list
    current_state: dict
    created_at: str

class CheckpointManager:
    def __init__(self, store):
        self.store = store  # Redis/DB

    def save(self, checkpoint: AgentCheckpoint):
        self.store.set(
            f"checkpoint:{checkpoint.task_id}",
            json.dumps(checkpoint.__dict__, default=str),
            ex=86400 * 7  # 7 天过期
        )

    def restore(self, task_id: str) -> AgentCheckpoint:
        data = self.store.get(f"checkpoint:{task_id}")
        if data:
            return AgentCheckpoint(**json.loads(data))
        return None

# 在 Agent 执行循环中自动 checkpoint
async def run_agent_with_checkpoint(task_id: str, goal: str):
    mgr = CheckpointManager(redis_store)
    cp = mgr.restore(task_id)

    if cp and cp.status == AgentStatus.WAITING_APPROVAL:
        # 恢复：等待用户审批
        approval = await wait_for_approval(cp.current_state["pending_action"])
        cp.status = AgentStatus.EXECUTING

    while cp.status not in (AgentStatus.COMPLETED, AgentStatus.FAILED):
        result = await execute_step(cp)
        cp.step_index += 1
        cp.completed_steps.append(result)
        mgr.save(cp)  # 每步都存 checkpoint
```

**关键设计要点：**
- 每个关键步骤后自动存 checkpoint，而非只在开始/结束时
- Checkpoint 包含足够的上下文信息，恢复时不需要重放全部历史
- 外部状态（工具调用结果）也要持久化，不能只存 LLM 的输出
- 设置 checkpoint 过期时间，避免存储无限增长

### Q7. Agent 评估中"轨迹指标"和"结果指标"有什么区别？为什么生产环境需要同时看两者？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Galileo AI、Anthropic 生态公司、字节跳动
**答：**

Galileo 的 Agent 评估框架明确指出："Outcome metrics tell you if your agent works; trajectory metrics tell you why. Production needs both perspectives."

**核心区别：**

| 维度 | 结果指标（Outcome Metrics） | 轨迹指标（Trajectory Metrics） |
|------|--------------------------|------------------------------|
| 衡量内容 | 最终答案是否正确 | 推理过程是否合理 |
| 类比 | 考试得了多少分 | 解题步骤是否规范 |
| 典型指标 | 任务完成率、准确率、F1 | 工具选择准确率、推理步数、重试次数 |
| 难点 | 正确答案难以自动判定 | 轨迹标注成本高 |
| 诊断价值 | 低（只知道错了） | 高（知道错在哪一步） |

**生产中的联合使用：**

```python
class AgentEvaluator:
    def evaluate(self, trace: AgentTrace) -> dict:
        # 结果指标
        outcome = {
            "task_completed": trace.final_answer is not None,
            "answer_correct": self.judge_correct(trace.query, trace.final_answer),
            "latency_ms": trace.total_time_ms,
        }

        # 轨迹指标
        trajectory = {
            "tool_selection_accuracy": self.check_tool_choices(trace),
            "reasoning_steps": len(trace.thoughts),
            "unnecessary_retries": self.count_retries(trace),
            "hallucination_in_reasoning": self.detect_hallucination(trace.thoughts),
            "context_utilization": self.check_context_usage(trace),
        }

        return {"outcome": outcome, "trajectory": trajectory}

    def diagnose(self, eval_result: dict):
        if eval_result["outcome"]["task_completed"]:
            return "healthy"
        # 结果失败时，用轨迹指标定位根因
        if eval_result["trajectory"]["tool_selection_accuracy"] < 0.5:
            return "tool_selection_issue"  # 工具选择有问题
        if eval_result["trajectory"]["unnecessary_retries"] > 3:
            return "retry_loop"  # 陷入重试
        if eval_result["trajectory"]["hallucination_in_reasoning"]:
            return "reasoning_hallucination"  # 推理幻觉
        return "unknown"
```

**面试加分：** 提到"结果指标用于告警（知道系统坏了），轨迹指标用于排障（知道哪里坏了）。两者结合才能实现快速诊断"。

### Q8. Agent 的工具调用失败时，应该如何优雅地处理？请设计一个带重试、降级和用户通知的错误处理策略。

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、美团、Salesforce
**答：**

Anthropic 在构建 Agent 指南中特别强调："What if the API is down or returns a 500 error? The agent needs a fallback." 工具调用失败是生产环境最常见的问题。

**三层错误处理策略：**

```python
class ToolCallErrorHandler:
    def __init__(self):
        self.retry_config = {"max_retries": 3, "backoff_base": 1.0}
        self.fallback_registry = {}

    async def execute_with_recovery(self, tool_name: str, params: dict, context: dict):
        # 第 1 层：带退避的重试
        for attempt in range(self.retry_config["max_retries"]):
            try:
                result = await self._call_tool(tool_name, params)
                if result.success:
                    return result
                # 非网络错误不重试（如参数错误）
                if not result.is_retryable:
                    break
            except TimeoutError:
                pass
            await asyncio.sleep(self.retry_config["backoff_base"] * (2 ** attempt))

        # 第 2 层：降级方案
        fallback = self.fallback_registry.get(tool_name)
        if fallback:
            return await fallback.execute(params, context)

        # 第 3 层：返回结构化错误给 LLM，让它自愈
        return ToolResult(
            success=False,
            error_message=f"""
            工具 {tool_name} 调用失败（已重试 {self.retry_config['max_retries']} 次）。
            可能原因：服务暂时不可用或限流。
            建议：请告知用户该功能暂时不可用，或使用替代方案。
            """,
            metadata={"retries_exhausted": True, "tool": tool_name}
        )

# 注册降级方案
handler.fallback_registry["search_web"] = CachedSearchFallback()
handler.fallback_registry["get_weather"] = SimpleWeatherFallback()
```

**关键设计原则：**
- 重试区分可重试错误（超时、限流）和不可重试错误（参数错误、权限不足）
- 降级方案应保持功能的核心价值（如搜索降级为缓存搜索，而非完全不返回）
- 错误消息要返回给 LLM 有意义的信息，而非简单的 error code
- 记录所有失败用于后续优化（哪些工具最不稳定、什么时段容易超时）

### Q9. LangChain 文档中的 Multi-Agent 编排模式有哪些？Subagents 和 Distributed 模式有什么区别？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、LangChain 生态公司、微软
**答：**

LangChain 官方文档定义了两种主要的 Multi-Agent 编排模式，并提出了 5 个评估维度。

**两种核心模式：**

| 维度 | Subagents 模式 | Distributed 模式 |
|------|---------------|-----------------|
| 路由方式 | 主 Agent 统一协调，所有路由经过主 Agent | Agent 之间直接通信 |
| 拓扑 | 星型拓扑（中心化） | 网状拓扑（去中心化） |
| 适合场景 | 任务明确、子任务可预知 | 动态协作、需要 Agent 间协商 |
| 调试难度 | 低（所有路由可追踪） | 高（需要跨 Agent 追踪） |
| 团队协作 | 主 Agent 团队负责编排 | 各团队独立维护各自 Agent |

**LangChain 的 5 个评估维度：**

```python
EVAL_DIMENSIONS = {

    "parallelization": "多个 Agent 能否并发执行？",

    "direct_user_interaction": "子 Agent 能否直接与用户对话？",
    "fault_tolerance": "单个 Agent 故障是否影响整体？"
}
```

**Subagents 实现示例：**

```python
from langgraph.prebuilt import create_react_agent

# 主 Agent 将子 Agent 作为工具调用
research_agent = create_react_agent(model, tools=[search, extract])
code_agent = create_react_agent(model, tools=[python_repl, file_ops])

# 主 Agent 编排
main_agent = create_react_agent(
    model,
    tools=[
        research_agent.as_tool("research", "搜索和调研"),
        code_agent.as_tool("code", "编写和执行代码"),
    ]
)
```

**面试加分：** 提到"选择哪种模式取决于两个因素：(1) 任务是否可预知——可预知用 Subagents，不可预知用 Distributed；(2) 团队组织结构——如果各 Agent 由不同团队维护，Distributed 更合适"。

### Q10. 如何为 Agent 系统设计 Streaming 架构，让用户实时看到 Agent 的推理过程？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、OpenAI 生态公司、阿里云
**答：**

Streaming 是提升 Agent 用户体验的关键。用户不想等待 30 秒看到一个完整答案，而是想看到 Agent 正在思考、正在搜索、正在执行。

**三层 Streaming 架构：**

```python
import asyncio
from typing import AsyncGenerator

class AgentStreamEvent:
    """统一的事件类型"""
    def __init__(self, event_type: str, data: dict):
        self.type = event_type  # thinking | tool_call | tool_result | token | done
        self.data = data

class StreamingAgent:
    async def run(self, query: str) -> AsyncGenerator[AgentStreamEvent, None]:
        # 1. 推理过程 streaming
        yield AgentStreamEvent("thinking", {"content": ""})
        async for chunk in llm.stream(thinking_prompt):
            yield AgentStreamEvent("thinking", {"content": chunk})

        # 2. 工具调用通知
        tool = self.select_tool(query)
        yield AgentStreamEvent("tool_call", {
            "tool": tool.name,
            "params": tool.params,
            "message": f"正在调用 {tool.name}..."
        })

        # 3. 工具执行结果
        result = await tool.execute()
        yield AgentStreamEvent("tool_result", {
            "tool": tool.name,
            "success": result.success,
            "summary": result.summary  # 不是完整结果，是摘要
        })

        # 4. 最终回答 streaming
        yield AgentStreamEvent("answer_start", {})
        async for token in llm.stream(answer_prompt):
            yield AgentStreamEvent("token", {"content": token})

        yield AgentStreamEvent("done", {"total_time_ms": elapsed})

# 前端 SSE 接收
async def stream_to_client(agent_stream):
    async for event in agent_stream:
        yield f"event: {event.type}\ndata: {json.dumps(event.data)}\n\n"
```

**关键设计要点：**
- 事件粒度要合适：太粗（只 stream 最终答案）用户没感知；太细（每个 token 都发）带宽浪费
- 工具调用开始/结束要有明确的事件标记，前端可以展示进度条
- 错误也要 stream 出来，不要静默失败
- 使用 SSE（Server-Sent Events）而非 WebSocket，因为 Agent 流是单向的

**用户看到的效果：**
```
🤔 思考中...「我需要先查询特斯拉的财报数据」
🔍 正在搜索 "Tesla Q1 2026 earnings"...
✅ 搜索完成，找到相关结果
🤔 继续分析...「数据拿到了，正在计算同比增长」
💬 特斯拉 2026 Q1 营收 251.7 亿美元，同比增长约 18.2%...
```

**面试加分：** 提到"Streaming 不只是前端体验优化，还是后端的健康信号——如果某个事件阶段卡住了，说明对应的服务有问题，可以作为监控指标"。
