### Q1. 设计一个完整的 Agent 评估指标体系：Trajectory Metrics（轨迹指标）和 Outcome Metrics（结果指标）分别衡量什么？为什么生产环境需要两者结合？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、Galileo 生态公司、所有 Agent 平台团队
**答：**

Galileo 2026 的 Agent 评估框架明确指出：Outcome metrics tell you if your agent works; trajectory metrics tell you why. Production needs both perspectives.

**Outcome Metrics（结果指标）—— 回答「Agent 工作了吗？」**
- 任务完成率（Task Success Rate）：Agent 产出正确/可用结果的比例
- 用户满意度（User Satisfaction）：基于显式反馈（点赞/踩）或隐式信号（会话时长、复访率）
- 端到端延迟（End-to-End Latency）：从用户发起到收到最终响应的时间
- **局限：** 只能告诉你「失败了」，无法告诉你「为什么失败」

**Trajectory Metrics（轨迹指标）—— 回答「Agent 是怎么工作的？」**
- 推理步数分布（Step Count Distribution）：正常任务 vs 异常任务的步数差异
- 工具调用成功率（Tool Call Success Rate）：每个工具的调用成功/失败比例
- 分支决策质量（Branch Quality）：Agent 在关键分支点的选择是否正确
- 回退率（Backtrack Rate）：Agent 需要撤销或重做某一步的频率
- 循环检测率（Loop Detection Rate）：Agent 陷入循环后被系统强制停止的频次
- **价值：** 在故障发生前预警——轨迹异常往往早于结果异常

**面试加分：** 「现实中常见误区是只监控 Outcome Metrics（任务完成率 95% 看起来很好），但结合 Trajectory Metrics 可能发现：完成任务的 Agent 平均走了 12 步而不是预期的 5 步——说明有严重的效率问题。两个指标结合才能全面评估 Agent 健康度。推荐公式：Agent Health Score = 0.6 × Outcome Score + 0.4 × Trajectory Score。」

### Q2. MCP（Model Context Protocol）协议如何支持 Agent 的动态工具发现（Dynamic Tool Discovery）？请描述 MCP Server 的 Capability Advertisement 机制。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、Anthropic/MCP 生态公司
**答：**

MCP 协议的核心设计之一是「动态工具发现」——Agent 不需要在启动时硬编码所有工具，而是在运行时通过「能力声明」动态获取可用工具。

**Capability Advertisement 机制流程：**

MCP Client（Agent）在初始化时发送 `initialize` 请求，Server 返回能力声明：
```json
{
  "capabilities": {
    "tools": { "listChanged": true },
    "resources": { "subscribe": true },
    "prompts": { "listChanged": true }
  },
  "protocolVersion": "2025-03-26",
  "serverInfo": { "name": "mcp-database-server", "version": "2.1.0" }
}
```

**后续交互包括：**
- Agent 通过 `tools/list` 获取 Server 的所有工具定义（名称、描述、JSON Schema）
- 当 Server 的工具发生变更时，通过 `notifications/tools/list_changed` 推送通知
- Agent 收到通知后重新调用 `tools/list` 获取最新列表
- `tools/call` 支持元数据和流式传输（`text/event-stream` 格式）

**面试加分：** 「MCP 的动态发现机制解决了 Agent 工具管理的两个痛点：一是 Agent 不再需要预定义的 tool registry，运行时自动发现；二是 Server 可以主动通知工具变更，解决了经典问题——第三方 API 升级了但 Agent 还在用旧 Schema。面试中提到 MCP 的 'transport-agnostic' 设计（支持 stdio、SSE、WebSocket）也很加分。」

### Q3. 什么是 Agent 的 Self-Reflection（自我反思）机制？它与 ReAct 中的 Thought 步骤有什么本质区别？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | 微软、Google DeepMind、字节跳动、Anthropic 生态公司
**答：**

Self-Reflection 是 2026 年 Agent 能力提升的重要方向，与 ReAct 的基础 Thought 步骤有本质区别。

**ReAct 的 Thought——短期推理：**
- Scope：只考虑当前步骤（「这一步我应该做什么？」）
- Depth：浅层，聚焦下一步行动选择
- Memory：依赖 LLM 的上下文窗口，不持久化
- 本质：Thought 是「操作指令的附注」——解释「我为什么选择这个工具」

**Self-Reflection——元认知：**
- Scope：回顾已完成的多步执行过程（「我过去几步做对了吗？」）
- Depth：深层，评估推理链的质量和一致性
- Memory：独立存储，跨会话可复用
- 本质：Reflection 是「对 Thought 的思考」——验证之前的推理是否正确

**实现模式：**
```python
class SelfReflectiveAgent:
    async def run(self, task: str) -> str:
        # 阶段 1：正常执行（ReAct 循环）
        result = await self.react_execute(task)

        # 阶段 2：反思（Reflection）
        reflection = await self.llm.reflect(
            task=task,
            actions=self.trace.actions,
            result=result,
            prompt="""Review the execution steps:
1. Did the agent follow the task correctly?
2. Were the tool calls appropriate?
3. Any missing steps or incorrect assumptions?"""
        )

        # 阶段 3：修正（Revision）
        if reflection.issues_detected:
            result = await self.revise(task, reflection)
        return result
```

**面试加分：** 「推荐两种模式：微观反思（每 3-5 步触发一次，检查方向是否正确）+ 宏观反思（任务完成时触发，检查整体质量）。Google DeepMind 的研究表明，加入宏观反思后 Agent 的任务完成率平均提升 15-22%。」

### Q4. 在 Agent 系统中，如何检测和防御 LLM 的「事实幻觉」（Factual Hallucination）？请设计一个适用于 Agent 场景的幻觉检测方案。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、微软、所有生产 Agent 公司
**答：**

2026 年的大厂面试中，幻觉检测从「主观题」变成了「系统设计题」。

**Agent 场景中幻觉的三大特点：**
1. **上下文幻觉：** Agent 从检索到的文档中「编造」不存在的信息
2. **参数幻觉：** 工具调用参数中的「值」是编造的（如虚构的日期、ID）
3. **推理幻觉：** Agent 的逻辑链正确但起始前提错误

**四层检测方案：**

**第一层：检索级检测（Retrieval-level）**
使用 NLI（自然语言推理）模型检查 Agent 输出是否被检索到的文档支持。每次工具调用后评估「答案 - 文档」的一致性分数。

**第二层：工具级验证（Tool-level Verification）**
Agent 调用工具后，用工具返回的实际结果验证 Agent 的「预期」——比如 Agent 说「用户张三有 5 个订单」，实际数据库返回 3 个，说明 Agent 产生了幻觉。

**第三层：交叉验证（Cross-Validation）**
用同一个问题问两次（不同模型或不同 temperature），比较两个回答的一致性。如果语义相似度低于阈值（如 85%），标记为可能幻觉。

**第四层：Judge Model**
用独立的评估模型（通常比 Agent 模型小但专用）做最终裁决。按照预定义的评分标准逐项评估输出质量。

**面试加分：** 「最容易被忽视的是『工具参数幻觉』——Agent 编造了不存在的参数值，比如传一个不存在的用户 ID。防御方案是『调用前参数校验』：对每个工具参数做业务规则验证，而不是完全信任 LLM 的输出。」

### Q5. 如何设计一个支持多会话、跨重启的 Agent 状态持久化方案？对比 Database-backed 和 Checkpoint-based 两种模式。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、LangChain/LangGraph 生态公司
**答：**

**模式一：Database-backed Persistence（基于数据库的持久化）**
- **原理：** 每次 Agent 状态变化时，将 `state_id, session_id, step_number, agent_state (JSONB), created_at` 写入关系型数据库
- **优点：** 支持随机访问、多 Agent 共享状态、历史审计
- **缺点：** 写入延迟、JSONB 序列化开销、并发写冲突
- **适用：** 需要跨 Agent 共享状态、需要审计回溯的场景

```python
class DBStateManager:
    async def save(self, session_id, step, state):
        await self.db.execute(
            "INSERT INTO states (session_id, step, state) VALUES ($1, $2, $3)",
            session_id, step, json.dumps(state)
        )

    async def load(self, session_id, step_num=None):
        if step_num:
            row = await self.db.fetchrow(
                "SELECT state FROM states WHERE session_id=$1 AND step=$2",
                session_id, step_num
            )
        else:
            row = await self.db.fetchrow(
                "SELECT state FROM states WHERE session_id=$1 ORDER BY step DESC",
                session_id
            )
        return json.loads(row["state"])
```

**模式二：Checkpoint-based Persistence（基于检查点的持久化）**
- **原理：** 在关键里程碑（任务拆解完成、检索完成、答案生成完成）保存整个 Agent 状态快照
- **优点：** 写入次数少（~10 次 vs ~100 次）、可原子性恢复、开销低
- **缺点：** 无法细粒度回溯、两次 checkpoint 之间的状态丢失可能
- **适用：** 长周期任务、资源敏感的 Agent 运行时

**面试加分：** 「生产系统通常结合两者：DB 存储 Agent 的『元状态』（当前进度、会话信息），Checkpoint 存储 Agent 的『完整状态』（全部上下文）。恢复流程：先查 DB 找到 session 的最新 checkpoint_id，再加载对应的 checkpoint 完整恢复。」

### Q6. 什么是 Agent 的 Streaming Action（流式行动）？在设计流式工具调用时，如何处理「部分完成」的状态？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团、实时交互场景
**答：**

Streaming Action 是让 Agent 在工具执行过程中向用户反馈阶段性进展的能力。

**Streaming Action 的工作原理：**
```python
class StreamingActionAgent:
    async def process(self, request):
        yield {"type": "status", "message": "正在分析你的问题..."}
        retrieval_task = asyncio.create_task(self.search(request))

        async for chunk in self.fetch_checkpoints():
            yield {"type": "progress", "current": chunk.step,
                   "total": chunk.total, "partial_result": chunk.partial()}

        yield {"type": "final", "content": final_answer}
```

**部分完成（Partial Completion）的三种处理策略：**

**策略 1：乐观展示（Optimistic Display）**
- Agent 在工具返回前基于之前已观察到的信息「预填」部分结果
- **风险：** 预填可能与实际结果不一致
- **适合：** 低风险信息类查询

**策略 2：渐进式填充（Progressive Fill）**
- 先将用户引导到最终结果结构上，各部分逐步填充
- 如显示 `数据获取中... → ✅ 获取完成 / 验证中... → 45%`

**策略 3：保留位策略（Placeholder Strategy）**
- 工具返回前输出「待定」占位符，返回后通过 WebSocket 更新
- **适合：** 需要精确展示的场景

**面试加分：** 「Streaming Action 最大的工程挑战是『状态一致性』——Agent 在流式输出的同时，用户可能发来一条新消息打断。生产系统必须处理好新旧请求的衔接：要么丢弃旧的流式输出，要么将阶段性结果合并到新的上下文中。」

### Q7. 在 Agent 通信中，Agent-to-Agent 的消息协议除了数据序列化外，还需要哪些关键设计要素？请从身份、信任、协商三个角度分析。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 微软、Google、字节跳动、多 Agent 系统团队
**答：**

多 Agent 系统面试中，候选人常聚焦「数据怎么传」，但面试官更关注「Agent 怎么建立信任和协作」。

**要素 1：身份与认证（Identity & Authentication）**
Agent 在通信前需确认对方身份：`Agent A：「我是 Customer-Service-Agent, ID: CSA-001, 属于域: tech-support」`
- ID 应具备层次性（域 + 类型 + 实例号）
- 双方交换「能力证书（Capability Certificate）」声明能做/不能做的事
- 证书应有时效性和签名校验，防止伪造

**要素 2：信任与安全（Trust & Security）**
信任是分级的，不是二元的：
```python
class AgentTrustManager:
    def can_trust(self, peer: AgentInfo, action: str) -> bool:
        if peer.domain == self.domain:
            return True  # 同一部署域，完全信任
        if peer.certificate and self.verify_certificate(peer.certificate):
            return True  # 已验证的第三方 Agent
        if action in self.whitelist["public_actions"]:
            return True  # 未知 Agent，最小权限
        return False
```

**要素 3：协商与契约（Negotiation & Contract）**
Agent 在协作前需协商契约：
- 发起方提出需求，响应方给出能力约束
- 契约以 Contract Object 形式持久化，包含 SLA、数据范围、终止条件
- 双方定期 ping 确认对方存活

**面试加分：** 「A2A 协议的创新点是 Agent Card 中的 Rate Limits 和 SLA 字段——Agent 在协商前就公开自己的服务能力上限（每秒 100 请求、P99 < 500ms）。面试中提到 'Contract-first Communication' 概念很加分。」

### Q8. 如何为 Agent 系统设计一个全面的「测试自动化」方案？需要覆盖哪些测试层级和测试类型？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团、快手
**答：**

Thinking Company 强调：traditional testing fails for AI agents because the output space is non-deterministic。

**Agent 测试的五个层级（自底向上）：**

**L1：单元测试（Unit Testing）—— 确定性测试**
测试独立组件：Tool 函数正确性、Prompt Template 渲染、路由逻辑、参数校验。
- **工具：** pytest、unittest

**L2：组件测试（Component Testing）—— 半确定性**
测试单个功能单元，Mock LLM 返回值：
```python
@pytest.mark.parametrize("llm_response,expected_tool", [
    ("调用工具 search_products", "search_products"),
    ("调用工具 get_price", "get_price"),
])
def test_routing(llm_response, expected_tool):
    mock_llm = MockerLLM(response=llm_response)
    agent = Agent(llm=mock_llm)
    assert agent.route("商品查询") == expected_tool
```

**L3：集成测试（Integration Testing）—— Agent 全链路**
带真实 LLM 但 Mock 外部工具。预设 50-100 个黄金用例，自动比较实际轨迹与期望轨迹的相似度。

**L4：端到端测试（E2E Testing）—— 真实环境**
全部真实组件，关注延迟、成功率、幻觉率。

**L5：回归测试（Regression Testing）—— 持续保障**
每次 Prompt/代码/工具变更时自动运行，检测 5% 以上的质量下降。

**面试加分：** 「Agent 测试最大难点是 Oracle Problem——开放性问题没有标准答案。最佳实践是用 Judge Model 评估输出质量（Rubric-based Evaluation），替代人工标注。」

### Q9. 设计一个 Agent 系统的「监控与告警引擎」——当 Agent 表现异常时，如何自动检测、分类和告警？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团、快手
**答：**

**异常检测器（Anomaly Detector）的三个维度：**
```python
class AnomalyDetector:
    def detect(self, trace: AgentTrace) -> list[Anomaly]:
        anomalies = []
        # 1. 统计异常（3σ 检测）
        for metric in ["steps", "latency_ms", "tokens"]:
            value, mean, std = trace[metric], self.baseline.mean, self.baseline.std
            if abs(value - mean) > 3 * std:
                anomalies.append(Anomaly(type="statistical", severity="critical"))
        # 2. 模式异常（死循环检测）
        if self.detect_dead_loop(trace.actions):
            anomalies.append(Anomaly(type="dead_loop", severity="critical"))
        # 3. 语义异常（语无伦次检测）
        if self.detect_confabulation(trace.outputs):
            anomalies.append(Anomaly(type="confabulation", severity="critical"))
        return anomalies
```

**告警路由（Alert Router）：**
- critical → pagerduty + sms（5 分钟内响应）
- warning → slack + email（30 分钟内响应）
- info → dashboard（日报）

**根因分析（Root Cause Analysis）：**
延迟飙升时自动排查：API Provider 降级？→ 向量库性能？→ 某工具超时？→ 步数增加？

**监控面板四象限：**
```
健康度（任务完成率/人肉介入率） | 效率（推理步数/Token 消耗）
安全（注入拦截/越权检测）       | 成本（模型费用/缓存命中率）
```

**面试加分：** 「最好的监控是自我修复的监控——检测到 Agent 步数增多时自动精简 System Prompt，观察 N 分钟是否改善，改善则保留否则回滚。就像 DevOps 的 Self-healing 理念应用到 Agent 运维中。」

### Q10. Agent 路由策略（Routing Strategy）从「规则路由」到「语义路由」再到「学习型路由」的演进路径是什么？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、微软、所有有 Agent 路由需求的公司
**答：**

**第一代：规则路由（Rule-based Routing）—— 确定性**
```python
class RuleRouter:
    def route(self, query: str) -> str:
        if "退款" in query or "退货" in query: return "refund_agent"
        elif "查询" in query: return "order_query_agent"
        else: return "general_agent"
```
- **优点：** 简单、零延迟、可解释
- **缺点：** 规则需人工维护，模糊查询匹配不到
- **适用：** 关键词清晰、流量稳定的场景

**第二代：语义路由（Semantic Routing）—— 基于 Embedding**
将每个路由目标编码为 embedding 向量，用户查询也用 embedding 匹配最近的路由。
- **优点：** 能处理模糊查询，扩展只需加 embedding
- **缺点：** 路由边界模糊，embedding 漂移
- **适用：** 意图多样化的场景

**第三代：学习型路由（Learning-based Routing）—— 基于反馈闭环**
使用轻量级分类器，结合用户画像、时段、历史行为等特征做预测。引入 Epsilon-Greedy 策略做探索 vs 利用。
- **优点：** 持续优化路由准确率，自动适应查询分布变化
- **缺点：** 需要反馈数据，冷启动阶段准确率低
- **适用：** 大规模生产系统（日请求 > 10 万）

**面试加分：** 「Anthropic 的《Building Effective Agents》原则：从简单的开始。建议路径是：先用规则跑通 → 语义路由提升覆盖 → 数据积累后做学习型精细优化。不要一步跳到第三代——没有历史数据的模型比规则还差。」
