### Q1. Agent 系统的安全防护分为哪五个层级？针对 Prompt Injection 攻击，有哪些在生产环境中验证有效的防御策略？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | OpenAI、Google、Microsoft、蚂蚁集团、所有 Agent 平台安全团队
**答：**

2026 年，随着 Agent 自主性越来越强，安全已经成了面试必考题。综合 CallSphere 的「安全五层堆栈」和 AEM Institute 的指导，Agent 安全纵深防御分为五个层级：

**层级 1 — 输入验证层（Input Sanitization）：** 所有用户输入在进入 Agent 前经过清洗，移除特殊指令标记、嵌入的提示词。基础但必须。

**层级 2 — 上下文隔离层（Context Isolation）：** 系统提示词（System Prompt）与用户输入分离，采用结构化模板而非拼接式注入。关键原则：用户输入永远不能覆盖系统指令。

**层级 3 — 工具调用过滤层（Tool Call Guard）：** 工具调用参数经过校验——即使 LLM 生成了恶意参数，工具层也能拒绝执行。例如文件删除工具拒绝 path 参数包含 `../` 的调用。

**层级 4 — 输出净化层（Output Sanitization）：** Agent 输出经过安全检查，防止间接注入（Indirect Injection）——Agent 从外部源（网页、数据库）读取的内容可能包含隐藏的指令。

**层级 5 — 审计与日志层（Audit Trail）：** 所有 Agent 推理轨迹、工具调用、输入输出被完整记录，支持事后取证。

**Prompt Injection 的有效实战策略：**
- **结构化输入：** 用 JSON/XML 模板而非纯文本拼接分隔系统和用户输入
- **权限最小化：** Agent 的每个工具都有独立权限域，即使被注入也无法访问越权工具
- **输入输出 Schema 校验：** 定义严格的参数 Schema 拒绝异常调用。参考 Anthropic 的「用自然语言写 Tool 描述」原则——描述中明确写入「何时不应调用此工具」

### Q2. Agent 中实现持久化记忆（Persistent Memory）有哪三种主流模式？各自适合什么场景？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、CallSphere 生态公司、所有用户类 Agent 团队
**答：**

CallSphere 2026 面试指南将 Agent 记忆分为三种模式，这是面试中经常被问到的高频考点：

**模式 1 — 临时记忆（Ephemeral Memory）：** 只在单次会话中有效，用对话历史做上下文。实现简单（维护 messages 数组），但会话结束后遗忘。适合：一次性问答、无状态的客服对话。

**模式 2 — 摘要记忆（Summarized Memory）：** 每次会话结束后，Agent 自动将关键信息压缩成摘要存入数据库。下次会话时加载摘要。典型实现：用 LLM 对整段对话做 summarize，存入向量数据库。适合：需要跨会话了解用户偏好的场景，如推荐系统、个性化助手。

**模式 3 — 结构化记忆（Structured Memory）：** 将用户信息提取为结构化数据（键值对、关系图谱）。例如 Agent 从对话中提取 `user.preferred_language = "中文"`、`user.timezone = "Asia/Shanghai"`。下次会话直接加载，无需重新推理。适合：需要精确状态管理的企业级应用，如 CRM 集成 Agent、医疗健康 Agent。

**面试加分：** 「实际生产系统通常采用**混合记忆模式**——短期用摘要记忆降低 Token 开销，长期用结构化记忆确保关键信息不丢失。另外，记忆的**重要性过滤**很重要：不是所有信息都值得存，需要设计一个重要性评分模型来决定哪些信息进入长期记忆。比如用户说『帮我订去北京的机票』是临时的，但『我常去北京出差』值得存。」

### Q3. 当 Agent 调用的外部 API 返回错误或超时时，有哪些成熟的容错设计模式？请结合 Anthropic 的指导说明。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、所有生产 Agent 团队、面试高频
**答：**

Anthropic《Building Effective Agents》指南和 CoPrep 都强调：工具失败的处理方式是区分优秀与平庸 Agent 工程师的关键。

**五大容错设计模式：**

**模式 1 — 重试（Retry with Backoff）：** 对临时性错误（网络超时、429 Rate Limit）使用指数退避重试。Implementation：Agent 检测到错误后自动发起重试，最多 3 次，每次等待时间加倍。

**模式 2 — 降级回退（Graceful Degradation）：** 主工具失败时调用备选工具。例如天气 API 挂了，切到备用的天气数据源。关键是 Agent 需要知道「有哪些替代方案」。

**模式 3 — 告知用户（Inform User）：** 不可恢复错误时 Agent 主动向用户说明：「订单查询服务暂时不可用，我已经记录下了你的请求，稍后会自动重试。」——这比沉默失败要好得多。

**模式 4 — 断点续传（Checkpoint & Resume）：** 复杂的多步骤 Agent 任务在每步完成后保存 checkpoint。失败后可从最近的 checkpoint 恢复，而非从头开始。在 LangChain/LangGraph 中通过 StateGraph 的 checkpoint 机制实现。

**模式 5 — 人工兜底（Human-in-the-Loop）：** 当 Agent 无法决定备选方案时，将决策权交还给人类。关键在于设计清晰的「升级条件」。

**面试加分：** 「Anthropic 特别强调：Agent 的工具描述中应该包含**错误处理指引**——在 Tool Schema 的 description 里写『如果返回 500，3 秒后重试』能让 Agent 更智能地处理异常。另外，工具返回的错误信息本身不要直接暴露给用户（安全考量），应该由 Agent 翻译成友好的用户消息。」

### Q4. Chain-of-Thought（CoT）与 Tree-of-Thoughts（ToT）在 Agent 推理中分别扮演什么角色？生产场景下如何选择？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Google DeepMind、OpenAI、微软、字节跳动
**答：**

CoT 和 ToT 是 2026 年 Agent 推理的两大核心范式，面试官喜欢问它们的适用场景取舍。

**CoT（思维链）—— 线性推理：**
- Agent 将问题分解为一系列步骤，每一步输出 Thought → Action → Observation
- 本质是 ReAct 模式的推理延伸
- **优点：** 实现简单（仅需 prompt 引导）、Token 消耗低、适合有明确路径的任务
- **缺点：** 无法回溯——一旦选择错误路径就无法回头
- **适用：** 有明确解决路径的任务（计算、按流程办事、文档处理）

**ToT（思维树）—— 分支探索：**
- Agent 在每一步生成多个候选推理路径，评估每条路径的可能性，选择最优路线
- 本质是多路径搜索 + 评估剪枝
- **优点：** 可以回溯和探索多个假设，适合复杂创新任务
- **缺点：** Token 开销大（O(b^d) 级增长）、延迟高、实现复杂
- **适用：** 开放性任务（代码 debug、数学难题、规划生成）

**生产选择指南：**
- 简单确定性任务 → CoT 即可（90% 的生产场景）
- 复杂开放任务 → ToT，但要设置**步数上限**和**搜索宽度限制**
- **最佳实践：** 先用 CoT 试跑，如果在预定步数内没有解则自动升级到 ToT

### Q5. Multi-Agent 系统有哪几种常见的通信拓扑模式？Supervisor（监督者）模式和 Network（网络）模式各自的优劣是什么？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | LangChain、微软、字节跳动、蚂蚁集团、高频考题
**答：**

LangChain 2026 多 Agent 文档和 CallSphere 面试指南都指出：通信拓扑模式决定了多 Agent 系统的可扩展性和可靠性。

**五种主流通信拓扑模式：**

**模式 1 — Supervisor（监督者/集中式）：**
- 一个主 Agent 协调所有子 Agent，所有通信通过主 Agent 路由
- ✅ 优点：控制集中、调试容易、权限管理清晰
- ❌ 缺点：单点瓶颈、主 Agent 可能成为 Token 黑洞、扩展困难
- 适用：团队较小、通信量可控的场景

**模式 2 — Network（网络/点对点）：**
- Agent 之间直接通信，无中央协调者
- ✅ 优点：高并发、无单点故障、天然支持分布式
- ❌ 缺点：调试困难、通信协议复杂、可能出现通信风暴
- 适用：Agent 数量多且需频繁交互的场景

**模式 3 — Handoff（转交式）：**
- Agent 在完成自身部分后将控制权转交给下一个 Agent，类似接力赛
- 适用：流水线式任务

**模式 4 — Router（路由器式）：**
- 一个 Router Agent 根据任务类型将请求分发到 Specialist Agent
- 适用：需要路由不同类型的任务

**模式 5 — Skill-based（技能式）：**
- 子 Agent 作为主 Agent 的「技能」注册，主 Agent 像调用工具一样调用子 Agent
- 适用：LangChain 推荐的默认模式

**面试加分：** 「生产系统中常采用**混合拓扑**——高层面用 Supervisor 做任务路由，低层面子团队内部用 Network 模式。LangChain 2026 推出的 Subagents 模式本质是 Skill-based + Supervisor 的混合体，是生产推荐方案。面试中提到『每种模式的通信开销（latency per hop）和故障传播半径』非常加分。」

### Q6. 传统的 Agent 评估指标（准确率、任务完成率）有哪些局限性？Galileo 提出的「替代指标」体系包括哪些新维度？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Galileo 生态、蚂蚁集团、字节跳动 AI 平台团队
**答：**

Galileo 2026 的 Agent Evaluation 指南和 Thinking Company 的生产评估文章指出：传统指标在 Agent 场景下有严重局限。

**传统指标的三大问题：**

1. **准确率是「事后指标」——** 等你发现准确率下降了，系统已经出了十几个问题了。Agent 需要的是「预警指标」。
2. **任务完成率无法揭示「why」——** 90% 的完成率看起来很好，但失败的那 10% 是同一个根因吗？
3. **缺乏过程量化——** 一个任务完成了但多走了 20 步，你应该给它 A+ 还是 D−？

**Galileo 替代指标体系（6个新维度）：**

| 维度 | 定义 | 价值 |
|------|------|------|
| **Loop Detection Rate** | Agent 循环调用的检出率 | 防止无限循环 |
| **Tool Call Efficiency** | 完成单位任务的工具调用次数 | 衡量 Agent 效率 |
| **Backtrack Rate** | Agent 撤销/重做某步的频率 | 反映规划质量 |
| **Branch Quality** | 关键分支点的选择正确率 | 衡量决策能力 |
| **Recovery Success Rate** | 从错误中恢复的成功率 | 衡量鲁棒性 |
| **Human Intervention Rate** | 人类需要干预的频率 | 衡量自主性 |

**面试加分：** 「Thinking Company 特别强调：**Human Intervention Rate** 是 2026 年最被低估的指标。一个 Agent 如果只有 1% 的干预率但在那 1% 上耗费了大量人力成本，生产上可能还不如干预率 5% 但每次干预只需点击一下的系统好。指标设计永远要结合**人力成本**和**系统收益**来权衡。」

### Q7. RAG 系统中 Chunking（分块）策略对检索质量影响巨大。请对比 Fixed-size Chunking、Semantic Chunking 和 Agentic Chunking 三种策略的优劣和适用场景。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、微软、所有 RAG 生产团队、高频考点
**答：**

DataCamp 2026 RAG 面试指南指出：Chunking 策略是 RAG 系统的「第一公里」，直接影响检索质量。

**策略 1 — Fixed-size Chunking（固定大小分块）：**
- 按固定 Token 数切分（如 512 tokens/块），通常带 overlap（如 10% 重叠）
- ✅ 优点：实现最简单、处理速度最快
- ❌ 缺点：语义不完整——一句话可能被切成两半
- 适用：文档结构简单、快速原型

**策略 2 — Semantic Chunking（语义分块）：**
- 利用 LLM 或 NLP 模型识别自然段落/话题边界，按语义单元切分
- ✅ 优点：保持语义完整、检索命中更准确
- ❌ 缺点：速度慢（需额外模型推理）、chunk 大小不均匀
- 适用：长文档、需要高检索质量的正式系统

**策略 3 — Agentic Chunking（智能分块）：**
- Agent 动态决定如何切分——对代码块按函数切，对论文按章节切，对对话按话题边界切
- ✅ 优点：最灵活、检索准确率最高
- ❌ 缺点：实现复杂、需要额外的 Agent 推理、成本高
- 适用：多格式混合文档、企业级知识库

**面试加分：** 「实践中最常用的不是单一策略，而是**混合策略**——先用语义分块保准召回的语义完整性，再用固定大小截断异常长的语义块以适配 Embedding 模型的最大长度。另外，Chunk 的 embedding 策略也很关键：给 chunk 加上段落标题和上下文的 embedding 做 'contextual embedding' 能显著提升检索准确率。」

### Q8. Agent 系统的可观测性（Observability）应关注哪三大支柱？如何用 Trajectory 日志定位 Agent「陷入循环」的问题？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Galileo、Thinking Company、字节跳动、蚂蚁集团
**答：**

Thinking Company 2026 生产 Agent 评估指南强调：Agent 的可观测性与传统微服务有本质区别——你不仅要看到「在调用什么」，还要看到「在想什么」。

**Agent 可观测性三大支柱：**

**支柱 1 — 推理轨迹（Trajectory）：** Agent 每步的 Thought → Tool Call → Observation 完整链路。这是 Agent 特有的可观测性数据，传统 Metrics/Logs/Traces 无法覆盖。

**支柱 2 — 工具调用指标（Tool Metrics）：** 每个工具的调用次数、成功率、延迟、错误分布。可三角定位具体哪个工具有问题。

**支柱 3 — 状态变更日志（State Change Log）：** Agent 的内部状态（记忆更新、决策分支、条件跳转）都被记录下来。

**定位「循环」问题的方法：**
- **用 Loop Detection 特征：** 监控连续 N 步调用同一个工具且输入高度相似
- **用相似度检测：** 计算最近 K 步的 embedding 相似度，如果相似度超过阈值且持续增长说明在循环
- **Trajectory 回放：** 通过轨迹日志逐帧回放 Agent 的决策过程，观察它在哪一步陷入重复

```python
# 循环检测的简化实现
def detect_loop(trajectory, window=5, threshold=0.95):
    recent_calls = trajectory[-window:]
    if len(recent_calls) < window:
        return False
    tool_names = [step['tool'] for step in recent_calls]
    if len(set(tool_names)) == 1:  # 连续调用同一个工具
        embeddings = [step['observation_embedding'] for step in recent_calls]
        if cosine_similarity(embeddings[0], embeddings[-1]) > threshold:
            return True  # 检测到循环
    return False
```

### Q9. 在知识图谱增强的 Agent 系统中，当多个知识来源对同一事实给出冲突信息时，Agent 应如何解决冲突？请设计一个合规场景下的冲突解决流程。

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐⭐ | 蚂蚁集团、微软、字节跳动金融 Agent 团队、高频场景题
**答：**

知识冲突是 Agent 在生产环境中最棘手的问题之一。结合 CoPrep 提到的 GraphRAG 推理框架和合规场景，可以设计以下冲突解决流程。

**场景：** 合规 Agent 查询员工张三的入职日期。HR 系统说 2025-01-15，项目管理系统说 2025-02-01，邮件显示 2024-12-20。

**四步冲突解决流程：**

**Step 1 — 置信度评分（Confidence Scoring）：**
Agent 对每个数据源按可信度排序：结构化系统（HR）> 半结构化（项目系统）> 非结构化（邮件）。HR 系统权重 0.9，项目系统 0.7，邮件 0.5。

**Step 2 — 时间权重（Temporal Weighting）：**
记录时间越近的数据置信度越高。HR 的入职日期是最近更新的，权重加分。

**Step 3 — 溯源推理（Source Tracing）：**
Agent 追问数据来源。发现 HR 系统的数据来自合同扫描件（权威），项目系统的数据是手动录入（可能有误），邮件是入职前的沟通（非正式）。

**Step 4 — 人工确认兜底（Escalation）：**
当置信度加权后仍然有无法消除的冲突（如两个权威来源冲突），Agent 生成冲突报告提交给合规审核员做最终裁决，并将结果存入知识图谱作为后续推理的基准。

**面试加分：** 「一个好的 Agent 在解决冲突后应该自动更新知识图谱的**冲突标记**——标注两数据源之间的不一致，并在后续查询中优先使用已仲裁的结果。提到 GraphRAG 中 edge weight 的动态调整（每次仲裁后权重自动变化）非常加分。最重要的是：Agent 不能『悄悄选一个』——它在任何冲突场景下都应将冲突信息传达给用户并说明自己选择了哪个来源及原因。」

### Q10. 在 Multi-Agent 系统中，Agent 之间如何建立信任和认证机制？请结合 MCP 协议与 OWASP LLM Top 10 的安全建议说明。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | Anthropic、微软、Google、蚂蚁集团、MCP 生态
**答：**

当多个 Agent 相互协作时，信任与认证是安全基座。AEM Institute 2026 指南和 CallSphere 的 Multi-Agent 安全内容都指出：Agent 之间的互信不能依赖「谁叫我我就听谁的」。

**Agent 间信任机制的四个层次：**

**层次 1 — 身份认证（Identity）：**
每个 Agent 持有唯一的 API Key / JWT Token，调用方必须出示有效凭证。MCP 协议支持在 initialize 阶段交换身份信息。

**层次 2 — 权限授权（Authorization）：**
每个 Agent 有明确的操作权限域。调用文件系统的 Agent 不能调用数据库。参考 OWASP LLM Top 10 中的 LLM05（过度代理权限）——Agent 权限应遵循最小权限原则。

**层次 3 — 调用审计（Audit）：**
每次 Agent 间调用都被记录：谁调用了谁、调用了什么工具、传入了什么参数、返回了什么结果。这是事后追溯和攻击取证的基础。

**层次 4 — 输出验证（Output Verification）：**
Agent 不应盲目信任其他 Agent 返回的结果。关键决策需要交叉验证。例如财务 Agent 在收到另一个 Agent 返回的「批准付款」指令后，应自行验证付款信息与原始请求是否一致。

**MCP 协议的支持：**
MCP 的能力声明（Capability Advertisement）天然支持信任协商——Agent A 声明自己有能力做什么（如 `tools/call` 可用的工具列表），Agent B 据此决定是否调用 Agent A。MCP 的 Resource 订阅机制也可以用来监控信任状态变更。

**面试加分：** 「面试中提到 2026 年的新趋势——**Agent Identity Provider（Agent IDP）** 概念很加分。就像人用 OAuth 登录，Agent 也需要一个中央身份提供商来管理 Agent 身份的发放、吊销和轮换。另外，MCP 协议中 'transport-agnostic' 的设计虽然灵活，但也意味着 TLS/mTLS 需要在传输层自行保障——这是生产部署中容易忽略的安全细节。」
