### Q1. 在构建 AI Agent 系统时，什么时候应该用「工作流（Workflow）」模式，什么时候应该用「Agent」模式？Anthropic 提出的设计选型框架是什么？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、微软、字节跳动、所有大厂 Agent 团队、2026年必问
**答：**

Anthropic《Building Effective Agents》明确指出：「Workflows are systems where LLMs and tools are orchestrated via predefined code paths. Agents, on the other hand, are systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks.」两者的核心区别在于「控制权的归属」。

**设计选型框架——什么时候选 Workflow：**
1. **任务确定性高**：如果任务的步骤是固定的、可枚举的（如每周自动生成销售报表：查询数据→格式化→邮件发送），用 Workflow 更可靠。Workflow 的每一步可以专门优化（如仅查询步骤用 LLM 提取 SQL 参数，其余步骤用确定性代码）。
2. **可解释性要求严格**：在金融、医疗等强监管行业，每一步的具体逻辑需要被完全理解。Workflow 的决策路径是预定义的，比 Agent 的黑箱推理更容易通过合规审计。
3. **延迟要求严格**：Workflow 的每一步耗时可预测（规则分支一般在 5-50ms），适合对响应时间有严格 SLA 的场景。

**什么时候选 Agent：**
1. **任务不确定性强**：需要 Agent 自己决定「怎么做」——研究竞品时应该先看财报还是先看产品页面？Agent 自主决定。
2. **需要大量工具选择**：当工具数量 > 20 时，预定义所有可能的调用路径不现实，Agent 的运行时选择更灵活。
3. **任务环境持续变化**：Agent 可以在运行时根据观察到的结果调整策略，Workflow 的模式则需要人工重新编排。

**面试加分：** 「Anthropic 的核心建议是『The simplest solution that works is best』——不要为了用 Agent 而用 Agent。一个常见的反模式是：一个只需要三步查询的任务，工程师却用了一个带 Planner-Executor-Critic 的 Agent 架构。这增加了延迟、成本和故障点。正确的做法：先用 Workflow 实现 MVP，在监控中发现『Agent 需要动态选择工具』时才升级为 Agent 模式。LangChain 的多 Agent 设计也遵循同样的原则——子代理模式（Subagent Pattern）本质上是一个 Workflow 化的 Agent 架构。」

### Q2. Goal-driven（目标驱动）系统和 Instruction-driven（指令驱动）系统有什么区别？在设计 AI Agent 时，如何根据场景选择合适的驱动模式？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | novelvista、CoPrep、字节跳动、OpenAI、2026年Agent设计新考题
**答：**

novelvista 的 2026 面试指南将此列为区分 Agent 设计深度的关键问题：「Simply saying 'it's autonomous' is not enough—you must explain how autonomy is achieved.」

**Goal-driven 系统：** Agent 只接收一个高阶目标（如「提高 Q2 用户留存率 5%」），然后自主规划执行路径。Agent 自主分解目标为子任务、选择工具、评估进展、调整策略。优势：灵活性极高，适合复杂多变的任务环境。劣势：不可预测性高，Agent 可能选择与预期完全不同的路径，需要更强的监控和安全机制。

**Instruction-driven 系统：** Agent 接收详细的步骤指令（如「先查询 Q1 留存率，再对比 Q2 数据，然后生成报告发送给经理」）。Agent 严格按指令执行，遇到意外情况时向人类请求澄清。优势：可预测、可控、适合流程化的业务操作。劣势：缺乏灵活性，无法应对指令未覆盖的边界情况。

**选型指南：**
- 目标驱动适合：创新研究类任务（探索未知知识领域）、战略规划类（需要多路径尝试）、环境高度动态的场景
- 指令驱动适合：标准操作流程（客服 SOP、合规检查）、高风险操作（支付处理、数据删除）、新员工操作的辅助 Agent

**面试加分：** 「一个实用的混合模式是 **Instruction-driven with Goal-oriented Exception Handling**——Agent 按指令执行，但当遇到指令未覆盖的异常情况时，Agent 进入『目标驱动模式』，基于『保护用户利益』『最小化风险』等高阶目标自主决策。novelvista 强调 Agent 真正的智能体现在『当指令和现实冲突时』的处理方式——不是盲目跟随指令而是理解目标内涵。」

### Q3. 在 Multi-Agent 系统中，「超级代理（Supervisor Agent）」模式和「网格（Mesh）」模式各自适用于什么场景？LangChain 文档中描述的两种模式有何本质区别？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | LangChain、CallSphere、微软、字节跳动、2026年Multi-Agent高频题
**答：**

LangChain 的多 Agent 文档描述了两种核心编排模式，CallSphere 的 2026 面试题也重点讨论了这两种模式的权衡。

**超级代理模式（Supervisor / Subagent Pattern）：** 一个主 Agent 作为中央调度器，将子 Agent 视为「工具」统一管理。所有请求先经过主 Agent，主 Agent 决定委派给哪个子 Agent，子 Agent 的结果返回给主 Agent 合成。优势：集中管理，适合「一个用户请求需要多个 Specialist 协作完成」的场景（如法律咨询 Agent → 税务专员 + 合同法专员 + 知识产权专员）。劣势：主 Agent 成为瓶颈（单点失效、上下文窗口限制），子 Agent 之间不能直接通信。

**网格模式（Mesh / Network Pattern）：** Agent 之间可以任意直接通信，没有中央调度器。每个 Agent 有自己的职责域，Agent 之间通过消息队列或共享黑板（Blackboard）进行协作。优势：分布式、无单点瓶颈、Agent 可水平扩展、子 Agent 之间可以接力协作。劣势：调试困难（消息路径不可预测）、容易出现循环消息、安全性更难保证。

**选型建议：**
- 超级代理模式：团队规模小（2-5 个 Specialist）、需要集中审计日志、子 Agent 独立性较高的场景
- 网格模式：Agent 数量多（10+）、Agent 之间需要大量横向协作、高吞吐需求的场景

**面试加分：** 「LangChain 文档中提到的 '**Distributed development**' 是关键区分维度——在超级代理模式下，子 Agent 的开发者可以独立维护，主 Agent 只需知道子 Agent 的 API（工具接口）；而在网格模式下，每个 Agent 需要知道其他 Agent 的协议。实际部署中常见 **Hybrid 模式**——核心业务用超级代理模式（安全可控），外围扩展用网格模式（灵活可扩展）。CallSphere 也强调——面试官想知道候选人对『Agent 间通信是否需要中介』这个核心问题的理解深度。」

### Q4. AI Agent 在生产环境中如何设计可观测性（Observability）埋点？需要采集哪些关键信号才能快速定位 Agent 行为异常？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | Galileo、Thinking Company、微软、字节跳动、所有生产 Agent 团队必问
**答：**

Thinking Company 的生产指南和 Galileo 的评估框架都强调：Agent 的可观测性不仅是日志采集，而是「不用发布新代码就能理解 Agent 为什么做了某个决策」的能力。

**四大关键信号维度：**

**1. 调用日志（Call Logs）**：每次 LLM 调用的完整记录——时间戳、模型名、输入 Token 数、输出 Token 数、延迟、是否命中缓存。这是成本分析的原始数据。Galileo 推荐在每个调用日志中附加 `trace_id` 串联同一次任务的多次 LLM 调用。

**2. 工具调用轨迹（Tool Trajectory）**：每个工具调用的请求/响应序列——工具名、参数（脱敏后）、耗时、状态码、是否重试。Thinking Company 强调轨迹数据必须包含「决策上下文」——Agent 为什么在这个时间点选择调用这个工具。

**3. 决策路径（Decision Path）**：Agent 的 Thought 序列——每个推理步骤的摘要、置信度、考虑的备选方案。这是最难采集但对调试最有价值的数据。方案：Agent 在每个 Thought 后自动输出结构化决策日志 `{"step": N, "thought_summary": "...", "selected_action": "...", "alternatives_considered": [...], "confidence": 0.85}`。

**4. 信号指标（Signal Metrics）**：聚合后的关键 KPI——任务成功率（基于最终结果）、步骤效率（平均工具调用数/任务）、人工干预率（需要人类纠正的比例）、每个 Agent 实例的 Token 消耗趋势。Galileo 建议将所有指标加上「趋势线」——单点波动是正常的，但连续 3 个数据点向坏方向移动就应该告警。

**面试加分：** 「**Trace Context Propagation** 是 Agent 可观测性的基础设施——每个传入的请求从头到尾携带一个 `trace_id`，所有子 Agent 调用、工具调用、LLM 调用都继承这个 ID。这样从用户反馈的『我的请求返回了错误答案』可以一路追溯到具体的 Agent 步骤和工具返回数据。Thinking Company 还推荐一个实用的 '**Drill-Down Dashboard**'——从『任务成功率下降 5%』→『发现主要是财务类任务失败』→『检查财务 Agent 的工具返回』→『发现内部财务 API Schema 变更未通知』。层级下钻让 5 分钟的定位工作压缩到 30 秒。」

### Q5. 用 LLM-as-Evaluator 评估 Agent 输出时，有哪些已知的可靠性陷阱？如何设计和校准评估器的判断标准？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Galileo、Thinking Company、OpenAI、微软、评估团队高频题
**答：**

Galileo 和 Thinking Company 的评估指南都警告了「用 LLM 评估 LLM」的内在不一致性问题。LLM-as-Evaluator 是一个强大的工具，但存在多个已知偏差。

**五大可靠性陷阱：**

**1. 位置偏差（Position Bias）**：评估器倾向于选择列表中的第一个或最后一个答案。在比较 A/B 版本的 Agent 输出时，如果总是 A 在前 B 在后，评估器可能系统性偏袒 A。缓解：随机打乱候选顺序，多次评估取平均。

**2. 自利偏差（Self-enhancement Bias）**：评估器更偏好与自己相似的答案风格。用 Gemini 评估 Gemini Agent 的答案时，会系统性给更高分（同模型偏好）。缓解：用不同模型（如 Claude 评估 GPT 的输出）交叉验证，或使用专用的细粒度评估器（Fine-tuned Evaluator）。

**3. 长度偏差（Length Bias）**：评估器倾向于给更长的答案更高的分，即使长答案包含无关信息。Galileo 的框架建议在评估标准中显式加入「简洁性」维度来对抗此偏差。

**4. 评分漂移（Score Drift）**：同一个评估器在不同时间点对同一输入的评分不一致。原因：模型更新、System Prompt 微小变化、甚至一天中不同时间的模型行为波动。缓解：在每次评估批次中加入标准参考样例（Golden Set），用相对分而不是绝对分。

**5. 评估维度模糊（Rubric Ambiguity）**：当评估标准不明确时（如「请判断答案的质量」），不同评估器调用会从不同角度理解「质量」。缓解：使用结构化的 Rubric——每个维度独立打分并给出具体理由。Galileo 推荐使用 `explicit_facts` 列表让评估器逐条校验事实。

**面试加分：** 「**Calibration through Human Alignment** 是最可靠的校准方式——定期（每周或每数据批次）让人类标注 100-200 条样例，将 LLM 评估器的评分与人类标注对齐。计算 KS 统计量，如果评估器评分分布与人类标注的差异超过阈值（如 10%），重新调整评估器 Prompt 或更换评估模型。Thinking Company 强调——没有经过 Human Alignment 的 LLM-as-Evaluator 在生产中可能产生『看起来很美但实际错误的』评估结果。」

### Q6. 为什么 Anthropic 强调工具 Schema 的格式要对齐模型训练数据的自然分布？好的工具 Schema 设计原则有哪些？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐ | Anthropic、OpenAI、字节跳动、2026年 Agent 工具设计高频题
**答：**

Anthropic《Building Effective Agents》中有一句关键的话：「Keep the format close to what the model has seen naturally occurring in text on the internet.」这意味着工具 Schema 的描述方式应该尽量贴近模型在训练语料中见过的自然语言模式。

**为什么格式对齐如此重要：**
LLM 本质上是一个「模式匹配引擎」。如果工具 Schema 的格式与模型在训练中见过的代码文档、API 规范、函数签名等模式相似，模型就能更准确地理解该工具的用途和参数含义。反之，如果使用高度抽象或非常规的 Schema 格式（如将所有工具的描述放在同一个 JSON 对象中），模型的工具选择准确率会明显下降。

**好的工具 Schema 设计原则：**

**原则 1 — 自然语言优先：** 工具名称和参数描述使用完整的自然语言句子，而不是过于简短的标签。坏例子：`tool: "srch"`, `desc: "search db"`。好例子：`tool: "search_company_database"`, `desc: "Search the internal company database for employee records, project information, and organizational data"`。

**原则 2 — 参数约束明确：** 每个参数清晰地标注类型、格式约束、默认值、是否必填。对枚举类型列出所有合法值。对日期、JSON、ID 等格式给出示例。Anthropic 的实践：在参数描述中加入「使用示例」帮助模型理解正确格式。

**原则 3 — 错误返回结构一致：** 所有工具的错误返回格式保持一致——`{"error": true, "error_code": "...", "error_message": "...", "suggestion": "..."}`。结构化的错误信息让 Agent 的 LLM 能自动理解失败原因并纠正。

**原则 4 — 避免冗余和冲突：** 工具之间的功能不要重叠——如果两个工具可以做同一件事，Agent 不知道该选哪个。工具名称和描述应该清晰区分职责边界。

**面试加分：** 「Anthropic 还强调了 **Testing Tool Usage Extensively**——在开发环境用 'Workbench' 反复测试同样的工具 50-100 次，观察模型对工具的描述理解和调用一致性。一个常见的发现是：工具名过于抽象（如 `data_accessor`）会让模型不知道何时调用，改名为 `fetch_sales_data_by_date_range` 后准确率提升 10-15%。工具 Schema 也需要版本管理——修改工具描述后必须重新测试 Agent 的调用行为。」

### Q7. AI Agent 的渐进式部署（Progressive Deployment）与灰度策略如何设计？如何在不影响现有用户的前提下验证新版本 Agent 的安全性？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Thinking Company、Galileo、微软、字节跳动、生产运维方向
**答：**

Thinking Company 的生产指南强调：Agent 的更新比传统软件更新风险更高——Agent 行为是由 LLM 驱动的，单元测试无法覆盖所有可能的用户输入。渐进式部署是必需的。

**四阶段灰度策略：**

**Phase 1 — Shadow Mode（影子模式）：** 新版本 Agent 同步处理生产流量但不对外展现结果。输出与生产 Agent 对比，计算「行为一致性指标」——工具调用路径的相似度、最终答案的语义相近度。当不一致率 > 5% 时，自动化分析差异并暂停灰度。影子模式不改变用户体验，风险最低。

**Phase 2 — Canary with Human-in-the-Loop（金丝雀+人工审核）：** 1-5% 的用户流量路由到新 Agent，但所有输出经过人工审核后才展示给用户。审核员标注「接受/修正/拒绝」——Galileo 的评估框架将此定为「最有效的安全验证方式」。持续 24-48 小时，收集不少于 200 条人工审核记录。

**Phase 3 — Traffic Ramp-up with Guardrails（流量爬坡+护栏）：** 以 10% → 25% → 50% 逐步增加新 Agent 的流量。同时设置护栏——自动成功率监控（新 Agent 成功率必须 ≥ 老 Agent 的 95%）、自动回滚条件（人工干预率超过 3% 或错误率上升 5%）、异常检测（工具调用模式偏离训练时的分布）。

**Phase 4 — Full Rollout（全量部署）：** 达到 100% 流量后继续监控 48 小时，确认所有指标稳定后宣布部署完成。

**关键安全验证：**
- **回归测试套件**：包含过去 30 天用户真实输入中的边界案例
- **对抗性输入测试**：Prompt 注入、乱码输入、超长输入
- **工具调用压力测试**：Agent 在工具超时、返回异常数据时的行为表现

**面试加分：** 「**A/B Testing with Behavioral Alignment** 是 Thinking Company 推荐的核心方法——在 Shadow Mode 中不仅要比较『任务是否成功』，还要比较 Agent 的行为『路径』是否一致。一个新 Agent 虽然结果成功率相同，但走了完全不同的工具调用路径（如多调用了 2 个不必要的 API），这暗示其推理逻辑有偏差。Galileo 的轨迹指标在这里非常有用——对比新旧 Agent 的轨迹分布 KL 散度，超过阈值则暂停部署。」

### Q8. 当 AI Agent 同时使用外部知识库（RAG）和内部长期记忆时，如何保持两者之间的一致性？如果两者发生冲突，Agent 应该优先相信谁？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | DataCamp、CallSphere、字节跳动、2026年 Agent 记忆系统高频题
**答：**

当 Agent 既依赖外部检索（RAG）又依赖内部记忆时，冲突是不可避免的——外部知识库是最新版本，但内部记忆可能由 Agent 自己在前几轮对话中积累的「知识」构成。

**一致性维护策略：**

**策略 1 — 时间戳优先级（Timestamp Priority）：** 每个信息片段（无论是来自外部检索还是内部记忆）都有最后更新时间戳。Agent 在回答时优先采用最新信息。如果外部知识库在 T1 更新了，而内部记忆在 T0 存储了旧版本，Agent 会自动使用外部知识库。这是一种「时间即真相」的策略。

**策略 2 — 来源标注与可信度分级（Source Annotation & Trust Scoring）：** 每条信息标注来源类型——「外部知识库（置信度 0.95）」vs「Agent 内部记忆（置信度 0.7）」vs「用户陈述（置信度 0.5）」。Agent 在冲突时优先采纳高置信度的信源。CallSphere 的设计中，内部记忆的置信度会随时间衰减——30 天内未验证的记忆置信度自动降至 0.3。

**策略 3 — 交叉验证模式（Cross-Validation）：** 当内部记忆和外部检索结果冲突时，Agent 不直接选择任一，而是启动交叉验证——针对冲突点向外部知识库发起更精准的检索，用检索结果「验证」内部记忆的正确性。如果外部知识库支持内部记忆，则更新内部记忆的置信度；如果外部知识库否定内部记忆，则标记内部记忆为「已过期」并触发记忆更新。

**策略 4 — 冲突告警与人工裁决（Conflict Escalation）：** 对高风险场景（如金融数据、医疗信息），冲突不自动解决。Agent 生成一份「冲突报告」展示两者的差异，提交给人工审计处理。

**面试加分：** 「DataCamp 的 RAG 指南强调了一个关键原则——**外部知识库应作为 'Ground Truth'，内部记忆应作为 'Working Hypothesis'**。Agent 可以基于内部记忆快速回答，但必须定期回验外部知识库。一个实用的设计是：Agent 在每次使用内部记忆回答时，自动在后台启动一次异步的外部验证——如果发现差异，Agent 在后端更新记忆并记录一次『记忆修正事件』。这样 Agent 的响应速度不受影响，同时保持了记忆的准确性。」

### Q9. 在某些延迟敏感的场景中（如客服实时对话、交易执行），AI Agent 如何确保在严格的时间约束下完成关键推理？有哪些任务调度策略可以平衡速度和质量？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、Google、所有实时 Agent 团队
**答：**

实时 Agent 场景的挑战在于：用户期望的响应时间（<2s）与 LLM 推理时间（可能 5-15s）之间存在根本冲突。这需要用专门的调度策略来弥合。

**四种延迟优化策略：**

**策略 1 — 预测性预推理（Predictive Pre-reasoning）：** Agent 在用户还在输入时就基于已收到的部分输入开始推理。例如客服 Agent 在用户打字时，自动从已输入的关键词推测意图并预加载相关上下文和工具。当用户发送完整消息时，Agent 的「预热」已完成，生成时间缩短 40-60%。风险：如果预测错误，预推理结果是浪费的。

**策略 2 — 层次化超时（Hierarchical Timeouts）：** 为 Agent 的推理过程设置粒度超时——子任务级超时 < 整体任务级超时。例如：整体响应时间 SLA 为 3 秒，则每个工具调用超时为 1 秒，每次 LLM 调用超时为 1.5 秒。任何一层超时触发时，Agent 降级到「快速回复模式」——用规则引擎或预置模板生成即时回应，后续再补充详细信息。

**策略 3 — 渐进式推理（Progressive Reasoning）：** Agent 先给出一个「快速初步回应」（如「好的，我正在查询您的订单信息，请稍候～」），然后在后台进行完整推理。完成后通过 WebSocket 或 Server-Sent Events 增量更新结果。这种「先回复后填充」的模式在客服场景中非常有效——用户等待焦躁感大幅降低。

**策略 4 — 模型级联调度（Model Cascade Scheduling）：** 对延迟敏感的任务，Agent 的调度器先尝试小模型（如 Claude Haiku、GPT-4o-mini），设置严格的时间预算（如 500ms）。如果小模型在预算内完成了可靠推理，直接返回结果；如果超时或模型置信度低，升级到大模型（Claude Sonnet/Opus）运行，但此时用户已经收到了初步回应。

**面试加分：** 「**Response Time Budget Allocation** 是精细化管理的关键——Agent 将总时间预算分配到推理流程的每一步。例如 3 秒预算：输入预处理 200ms → 意图分类 300ms → 检索 500ms → LLM 推理 1500ms → 输出格式化 500ms。如果检索超时（>500ms），从 LLM 推理时间中扣除超出的部分，确保总预算不被突破。实践中最常见的超时瓶颈是 LLM 推理阶段——为此可以预先准备 2-3 个不同长度和复杂度的 Prompt 模板，Agent 根据剩余时间自动选择适合的模板。」

### Q10. 如何为 AI Agent 系统设计 A/B 测试框架？Agent 的 A/B 测试与传统软件测试相比，有哪些额外的挑战和需要特殊处理的地方？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | Galileo、Thinking Company、微软、字节跳动、数据驱动团队
**答：**

Agent 的 A/B 测试比传统软件 A/B 测试复杂得多，因为 Agent 的行为不是确定性的——同一次 Prompt、同一组输入，Agent 可能产生不同的输出。Galileo 和 Thinking Company 的指南都强调了 Agent 评估的特殊统计挑战。

**Agent A/B 测试的设计方案：**

**维度 1 — 随机化的单元选择：** 传统 A/B 测试中每个用户随机分配到 A/B 组即可。Agent 的 A/B 测试建议按「任务/会话」而非「用户」随机分组——因为同一用户的不同任务可能落在不同 Agent 版本上。关键：分配必须在任务入口处确定，不在执行过程中切换版本。

**维度 2 — 多维度评估指标：** Agent A/B 测试不能只看「任务成功率」。Galileo 推荐同时跟踪 4 类指标——结果指标（任务完成率、用户满意度）、效率指标（工具调用步数、Token 消耗）、安全指标（人工干预率、注入攻击成功率）、体验指标（响应延迟、用户重试次数）。

**维度 3 — 统计显著性检验的挑战：** Agent 输出的高方差性意味着需要更大的样本量才能达到统计显著。Galileo 建议：在 Agent A/B 测试中使用 Bootstrap 方法（带放回重采样 1000-5000 次）计算置信区间，而不是假设数据服从正态分布的 t 检验。当 A/B 组的置信区间无重叠时判断为显著差异。

**维度 4 — 行为漂移的检测：** Agent 可能在新版本上线后「逐渐」改变行为模式（非突发性变化）。Thinking Company 推荐在 A/B 测试期间每天计算轨迹分布的 KL 散度——如果 KL 散度在测试周期内持续增长，说明 Agent 的行为正在漂移，需要终止测试并检查原因。

**额外挑战：**
- **Carry-over Effect**：用户对 Agent 的信任可能受之前交互的影响，导致测试结果偏差
- **版本间污染**：如果同一 Agent 实例在测试期间切换了版本，前后行为不一致
- **长期效果评估**：Agent 行为变化可能在短期内不明显，但长期积累有显著影响

**面试加分：** 「**Sequential Testing with Early Stopping** 在 Agent A/B 测试中特别有价值——传统固定样本量测试在 Agent 场景中耗时过长（因为方差大）。顺序测试允许每处理 N 个任务后（如 100 个）进行一次显著性检验，一旦达到显著水平就可以提前停止实验。Galileo 的框架支持这个功能，使用 'mSPRT'（混合序贯概率比检验）来控制假阳性率。关键：必须使用 Sequential Testing 而不是传统的 Frequentist 统计，否则连续的『窥视效应』会大大增加假阳性概率。」
