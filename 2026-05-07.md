### Q1. 你线上的 Agent 系统突然出现任务完成率从 92% 暴跌到 60%，作为 on-call 工程师，你的排查思路是什么？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团
**答：**

这是 Agent 工程面试中最经典的场景题之一。面试官考察的是你有无完整的**生产排障 SOP**，而非猜原因。

**排查框架："

1. **确认范围 & 对比**：先确认是全部 Agent 任务都下降，还是特定类型。对比正常时段和异常时段的日志，看变化发生在哪个时间点。模型升级、工具 API 变更、数据源更新是最常见的三大触发因素。

2. **分层排查（从外到内）：**

   | 排查层 | 检查点 | 工具/方法 |
   |--------|--------|----------|
   | 模型层 | 是否有模型版本变更？温度参数是否被覆盖？ | 对比 LLM 调用的 model_version 字段 |
   | 工具层 | 是否有外部 API 超时/限流？工具返回结构是否有变化？ | 检查工具调用错误率和返回格式 |
   | Prompt 层 | System prompt 是否有灰度变更？上下文压缩策略是否改了？ | 对比前后 prompt 模板 diff |
   | 数据层 | 检索数据的分布是否漂移？新文档是否引入了低质量内容？ | 检查最近数据摄入的批次质量 |

3. **核心指标排查——先看"人肉干预率"**：Anthropic 的运营数据表明，人肉干预率从 5% 爬升到 12% 通常预示着一周内会有系统级质量事故。如果人肉干预率在上升，说明 Agent 不是突然坏掉的，而是"慢慢变差"的，底层原因可能是数据漂移。

4. **根因定位后的修复流程：**
   - 紧急：回滚最近变更（模型/Prompt/数据）
   - 短期：对被影响的任务类型做针对性修复
   - 长期：建立回归测试 + 持续监控体系，确保下次提前发现

**面试加分：** 提到"先回滚止血，再根因分析"而非第一时间排查代码——生产环境中稳定 > 完美。

### Q2. Chain-of-Thought (CoT) 与 Tree-of-Thoughts (ToT) 在 Agentic Planning 中的应用有什么区别？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 百度、微软、小红书
**答：**

CoT 和 ToT 是 Agent 推理的两个重要范式，2026 年面试中越来越常被放在一起对比。

**核心区别：**

| 维度 | Chain-of-Thought (CoT) | Tree-of-Thoughts (ToT) |
|------|----------------------|----------------------|
| 推理路径 | 单一线性路径 | 多条分支并行探索 |
| 决策机制 | 逐步推理，每一步只有一个选择 | 每个决策点同时评估多个候选，可回溯 |
| 失败恢复 | 走到死路只能重头开始 | 可以回溯到分叉点尝试其他分支 |
| Token 成本 | 低（O(n)） | 高（O(b^d)，b=分支数，d=深度） |
| 适用场景 | 流程明确的推理任务 | 需要试错的开放性问题 |
| 实现复杂度 | 简单，Few-shot 即可 | 需要额外的评估+回溯机制 |

**在 Agent 中的具体应用：**

```python
# CoT 风格：Agent 线性推理
def cot_agent_plan(query):
    thought = "第一步：我需要先查询用户身份"
    action = "tool: get_user_profile"
    obs = execute(action)
    # 继续下一步推理...
    thought2 = f"拿到用户信息了，第二步：基于 {obs} 查询订单"
    ...

# ToT 风格：Agent 并行探索
def tot_agent_plan(query):
    # 同时考虑多条路径
    branches = [
        "路径A：先查库存 → 再查价格 → 再下单",
        "路径B：先查用户信用 → 再查库存 → 再下单",
        "路径C：直接查订单历史 → 看是否有过购买记录"
    ]
    # 评估每条路径的可行性
    scores = [evaluate_branch(b) for b in branches]
    best_path = branches[argmax(scores)]
    # 执行最优路径，卡住时回溯
    result = execute_with_backtrack(best_path)
```

**面试加分：** 提到"ToT 不是 CoT 的替代品，而是互补品。在实际系统中，先 CoT 快速尝试，失败后再用 ToT 做多路径探索，比纯 ToT 更高效"。

### Q3. 你如何为一个 Agent 系统设计完整的生产环境评估体系（Evaluation Pipeline）？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 微软、字节跳动、Anthropic 生态公司
**答：**

Agent 评估比传统软件测试难得多，因为 LLM 的输出是概率性的、非确定性的。2026 年 LangChain 的调查显示，评估和质量保障是 Agent 上线的最大障碍。

**四层评估体系：**

**Layer 1：能力基准测试（Pre-Deployment）**
每个 Agent 上线前必须通过能力基准验证：
- 维护 50-100 个代表性场景，按难度分层（简单 30% / 中等 50% / 困难 20%）
- 每个场景包含输入、期望输出特征（非精确文本）、评分标准
- 每次修改 Prompt/Pipeline 后全量跑一次

**Layer 2：集成测试（Agent 边界测试）**
微软研究显示 61% 的多 Agent 系统故障发生在 Agent 之间的交接点：
- Handoff Fidelity：Agent A 的输出格式是否匹配 Agent B 的输入 Schema
- 工具可靠性：工具调用失败重试、限流处理是否正常
- 上下文传播：关键上下文是否在所有 Agent 间正确传递
- 边界条件：超长文档、空输入、对抗性 Prompt

**Layer 3：生产监控（Post-Deployment）**

| 指标类别 | 核心指标 | 告警阈值 |
|---------|---------|---------|
| 任务完成 | 端到端完成率 | < 85% P1 |
| 人肉干预 | 人肉干预率 | 一周内从 5%→12% 预警 |
| 质量 | LLM-as-Judge 评分 | 连续下降 5% |
| 安全 | 超出权限的动作 | > 0 立即告警 |

**Layer 4：人工反馈闭环**
- 抽样审查：每日随机抽 5-10% 的 Agent 输出做人工评分
- 升级分析：人肉干预的 case 按失败类型聚类，发现系统性问题
- 盲测 A/B：新旧版本盲测对比，避免评估者偏见

**成本估算：** 评估基础设施通常占 Agent 运营成本的 10-25%，但投入后会降低 3-5 倍的故障应急成本。

### Q4. 在多 Agent 系统中，如何解决不同 Agent 对同一业务数据理解不一致的问题？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 美团、阿里、字节跳动
**答：**

这是 2026 年多 Agent 系统面试中最常遇到的**场景挖坑题**。面试官想知道你有没有经历过"各个 Agent 各自为政"的真实痛苦。

**问题的本质：**
不同 Agent 直接编写 SQL 或调用底层 API 查询数据，但缺乏统一的语义层。比如客服 Agent 理解的"活跃用户"是"7 天内登录的用户"，而分析 Agent 理解的"活跃用户"是"30 天内产生过交易的用户"。同一个词在不同 Agent 嘴里意思完全不同。

**三步解决方案：**

**第一步：建设统一语义服务层（Semantic Layer）**
```python
# ❌ 错误：每个 Agent 各自写 SQL
class CustomerServiceAgent:
    def get_active_users(self):
        return db.query("SELECT * FROM users WHERE last_login > NOW() - 7")

class AnalyticsAgent:
    def get_active_users(self):
        return db.query("SELECT * FROM users WHERE last_trade > NOW() - 30")

# ✅ 正确：统一语义服务层
class SemanticLayer:
    def get_metric(self, metric_name: str, definition: str, params: dict):
        """所有 Agent 通过语义层获取数据"""
        # definition 必须是预定义且经过审批的
        sql = self.get_approved_query(metric_name, definition)
        return db.query(sql, params)

# Agent 只能调用封装的 API
class Agent:
    def query_data(self, metric: str):
        return semantic_layer.get_metric(
            metric_name="active_users",
            definition="standard",  # 只能用公司统一的标准定义
            params={"team_id": self.team_id}
        )
```

**第二步：工具调用前置校验**
在 Agent 调用任何工具前，中间件对参数做强制校验：
- 检查参数名称是否在允许列表中
- 检查值是否符合预期格式（如枚举值必须在白名单内）
- 检查权限范围（不能越权查询其他团队的数据）

**第三步：元数据治理**
- 所有业务指标有一个统一的注册中心，包含名称、定义、口径、负责人
- Agent 只能通过这个注册中心发现和调用业务指标
- 指标口径变更时，自动通知所有受影响的 Agent

**面试加分：** 提到"这本质上是一个数据治理问题，而不是 AI 问题。技术上可以通过 Semantic Layer 解决，但组织上需要定义清楚谁负责维护指标口径"。

### Q5. Agent 的 Tool Calling 中，LLM 生成的参数语法正确但语义错误（如把用户 ID 传成了订单 ID），如何防护？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、Salesforce
**答：**

这是生产环境中非常隐蔽的一类错误——代码不会报错，但业务语义完全不对。比参数缺失更难发现。

**多层防护方案：**

**第 1 层：Schema 前置校验（必做）**
```python
from pydantic import BaseModel, Field, validator

class RefundParams(BaseModel):
    order_id: str = Field(description="订单ID，格式：ORD-XXXXXXXX")
    user_id: str = Field(description="用户ID")
    amount: float = Field(description="退款金额，单位：元")

    @validator("order_id")
    def validate_order_id(cls, v):
        if not v.startswith("ORD-"):
            raise ValueError(f"订单ID格式错误：{v}，应以 ORD- 开头")
        return v

    @validator("amount")
    def validate_amount(cls, v, values):
        if v <= 0:
            raise ValueError("退款金额必须大于0")
        if v > 50000:
            raise ValueError(f"退款金额 {v} 超过单笔上限 50000 元，需人工审批")
        return v
```

**第 2 层：业务上下文校验**
```python
def business_validation(tool_name: str, params: dict, context: dict) -> bool:
    """基于当前会话上下文的业务语义校验"""
    if tool_name == "refund":
        # 校验：订单的归属用户是否就是当前会话用户
        order = order_service.get_order(params["order_id"])
        if order.user_id != context["current_user_id"]:
            return False  # ❌ 用户尝试退别人的订单
        # 校验：退款金额是否不超过订单实付金额
        if params["amount"] > order.paid_amount:
            return False  # ❌ 退款金额超过实付金额
    return True
```

**第 3 层：错误消息注入（让 LLM 自愈）**
当校验失败时，不要简单返回错误码，而是返回结构化的错误信息，帮助 LLM 修正参数：
```python
# ❌ 不好：返回简单错误
return {"error": "VALIDATION_FAILED"}

# ✅ 好：返回带解释的错误消息
return {
    "error": "参数校验失败",
    "details": "字段 'user_id' 的值 'ORD-20260507' 格式有误。user_id 应为 'U' 开头加数字组合，例如 'U-12345'。你误传了订单ID，请使用正确的用户ID后再重试。",
    "field": "user_id",
    "expected_format": "U-{数字}",
    "example": "U-123456"
}
```

**第 4 层：语义层预验证**
对于高风险操作（退款、删除、转账），在执行前用另一个 Agent 或同一个 Agent 的第二次推理做交叉验证：
```python
# 二级验证：用 LLM 自己判断参数是否合理
verification_prompt = f"""
以下是即将执行的操作，请判断参数是否合理：
工具：{tool_name}
参数：{params}
会话上下文：{context}

请回答以下问题：
1. user_id 是否与当前会话用户一致？
2. 操作金额是否在合理范围内？
3. 是否存在任何可疑的语义错误？
"""
```

### Q6. 什么时候应该用单 Agent（Monolithic）架构，什么时候应该用多 Agent 架构？请用具体场景说明。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、微软、腾讯
**答：**

2026 年面试官最反感的就是"只会多 Agent"的候选人。真正有经验的工程师知道单 Agent 在很多场景下是更好的选择。

**决策矩阵：**

| 决策维度 | 单 Agent | 多 Agent |
|---------|---------|---------|
| 任务复杂度 | 简单到中等复杂度 | 复杂、需不同专业能力 |
| 上下文关联度 | 高度关联，需统一上下文 | 任务可解耦为独立子任务 |
| 工具数量 | ≤ 10 个工具 | ≥ 20 个工具 |
| 延迟要求 | 高（端到端 < 3s） | 可接受高延迟 |
| 并行需求 | 串行为主 | 可并行执行子任务 |
| 调试难度 | 低 | 高（需要跨 Agent 追踪） |

**具体场景示例：**

**场景 A：用单 Agent（推荐）**
```
需求："帮我查一下今天的天气，然后推荐适合的穿搭"
- 流程简单：天气 API → 穿搭推荐
- 上下文紧密：需要将天气结果直接传给推荐逻辑
- 工具少：2 个工具（天气 API + 知识库）
→ 单 Agent 就够，多 Agent 反而增加不必要的通信开销和延迟
```

**场景 B：用多 Agent（推荐）**
```
需求："分析 Q1 的销售数据，生成一份包含图表和文字分析的周报"
- 子任务可解耦：数据分析、图表生成、文档撰写
- 需要不同专业能力：SQL Agent、Chart Agent、Writer Agent
- 可并行：数据分析和图表可以同时进行
→ Supervisor Agent 拆解任务 → 三个 Worker Agent 并行执行 → 汇总
```

**场景 C：混合架构（最优方案）**
最佳实践不是"非此即彼"，而是 Supervisor + Worker 模式：
- 一个 Supervisor Agent 负责意图识别和任务拆解
- 对简单子任务，Supervisor 自己处理
- 对复杂子任务，Supervisor 委托给专门的 Worker Agent
- 这样可以兼顾单 Agent 的高效和多 Agent 的专业性

**面试加分：** "多 Agent 系统失败率高达 41%-86%（UC Berkeley 研究），主要失败原因不是 Agent 能力不足，而是协调问题。所以引入多 Agent 前，先问自己：单 Agent 真的搞不定吗？"

### Q7. Agent 的记忆系统如何避免"记忆过时"问题？比如用户半年前偏好 A，现在已经转向 B。

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 阿里、美团、Anthropic 生态公司
**答：**

"用户六个月前喜欢 X，但现在已经不再用了，Agent 还一直引用旧偏好"——这是生产环境中最常见的记忆问题之一。

**记忆过时的四种解决方案：**

**方案 1：时间衰减排序（Time-Decay Ranking）**
```python
def retrieve_memories(user_id: str, query: str, top_k: int = 5):
    results = vector_db.search(query, filter={"user_id": user_id})
    # 对每个记忆按时间加权
    now = time.time()
    for mem in results:
        days_old = (now - mem.timestamp) / 86400
        time_score = math.exp(-days_old / 30)  # 半衰期 30 天
        mem.relevance = mem.semantic_score * 0.6 + time_score * 0.4
    return sorted(results, key=lambda m: m.relevance, reverse=True)[:top_k]
```

**方案 2：矛盾信号覆写**
当用户表现出与新偏好矛盾的信号时，Agent 应优先考虑新信号：
```python
# 用户之前说：喜欢早上航班 → 存入记忆
# 用户这次说：帮我订下午的航班
# → Agent 应该优先这次的新指令，而不是坚持旧记忆

def get_user_preference(preference_key: str, current_input: str, memory: list):
    # 检查当前对话中是否有明确的矛盾信息
    if current_input contains explicit_preference(preference_key):
        return extract_from_current(current_input)  # 用最新的
    # 否则才从记忆中检索
    return search_memory(preference_key, memory)
```

**方案 3：主动探测**
当 Agent 不确定用户偏好是否改变时，主动询问：
```
Agent: "我记得你之前喜欢早上航班，这次还是要早上的吗？还是改主意了？"
```

**方案 4：记忆归档与摘要**
- 每 100 次交互后，将原始交互日志摘要为用户画像（结构化数据）
- 原始日志归档（不用于实时检索），用户画像作为长期记忆
- 用户画像本身加上"置信度"和"最后更新时间"字段

**核心原则：** 短期行为覆盖长期记忆。用户当前对话中明确表达的偏好 > 最近 30 天的行为模式 > 归档的历史画像。

### Q8. 什么是 Shadow Testing（影子测试）？如何用它来安全地升级 Agent 的模型版本？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、快手、微软
**答：**

Shadow Testing 是生产环境中安全升级 Agent 的最佳实践。Scale AI 调查显示，2025 年 34% 的企业因为模型升级经历了意外的 Agent 行为变化，12% 导致了生产事故。

**什么是 Shadow Testing？**
在生产环境中，新版本 Agent 与当前线上版本并行运行，**接收相同输入但不影响真实用户**。新版本的输出被记录但不展示给用户。通过对比新旧版本的输出来评估新版本的质量。

**实现方案：**

```python
class ShadowTestOrchestrator:
    def __init__(self):
        self.production_agent = Agent("production")  # 线上版本
        self.shadow_agent = Agent("candidate")       # 候选版本
        self.comparison_store = []                   # 对比结果存储
    
    def handle_request(self, user_request: dict):
        # 1. 主路径：线上版本处理并返回给用户
        prod_result = self.production_agent.run(user_request)
        
        # 2. 影子路径：候选版本异步处理，不影响用户
        asyncio.create_task(self._run_shadow(user_request, prod_result))
        
        # 3. 返回线上版本的结果给用户（用户无感知）
        return prod_result
    
    async def _run_shadow(self, user_request: dict, prod_result: dict):
        try:
            shadow_result = await self.shadow_agent.run_async(user_request)
            # 对比新旧版本
            comparison = self.compare(
                query=user_request,
                production=prod_result,
                shadow=shadow_result
            )
            self.comparison_store.append(comparison)
            
            # 如果新版本表现差，自动告警
            if comparison.score < THRESHOLD:
                alert_team(f"新版本在 query '{user_request}' 上表现差于线上版本")
        except Exception as e:
            log_error(f"Shadow test failed: {e}")
    
    def compare(self, query, production, shadow) -> dict:
        """用 LLM-Judge 对比新旧版本输出质量"""
        judge_prompt = f"""
        对比以下两个 Agent 对同一用户问题的回答：
        
        用户问题：{query}
        
        版本 A（线上）：{production}
        版本 B（候选）：{shadow}
        
        请从准确度、安全性、完整性三个维度评分（1-5），并给出哪个版本更好。
        """
        return llm_judge(judge_prompt)
```

**模型升级的安全流程：**

```
新模型版本
    ↓
1. Benchmark 回归测试（离线） → 如果通过继续，不通过退回
    ↓
2. Shadow Testing（生产环境影子模式）→ 运行 24-48 小时，收集至少 1000 次对比
    ↓
3. 如果 Shadow 结果 ≥ 线上版本 → Canary 5% → 10% → 50% → 100%
    ↓
4. 如果 Shadow 结果 < 线上版本 → 退回，分析差异原因
```

**关键陷阱：**
- Pin 模型版本！用 `gpt-4o-2026-05-01` 而非 `gpt-4o-latest`
- 持续做 Benchmark 回归，即使没做任何变更——因为模型提供商可能在无通知的情况下更新
- 做好回滚准备：确保能在几分钟内切回旧版本

### Q9. Agent 系统中，如何平衡"安全限制"和"用户体验"？审批流程太严让用户失去耐心怎么办？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 拼多多、美团、字节跳动
**答：**

这是一个很真实的两难场景。安全团队想要"所有操作都审批"，产品团队想要"一键搞定"，Agent 工程师在中间被夹击。

**核心思路：分级风险管控（Risk-based Grading）**

**第一步：将操作按风险分级**

| 风险级别 | 操作类型 | 审批策略 | 示例 |
|---------|---------|---------|------|
| L0 - 安全 | 只读查询、信息检索 | 自动执行，无需审批 | 查天气、查订单状态 |
| L1 - 低风险 | 状态更新、基本信息修改 | 弹窗一次确认即可 | 改收货地址、改昵称 |
| L2 - 中风险 | 涉及资金或隐私的操作 | 二次确认 + 可撤销 | 退款、改密码 |
| L3 - 高风险 | 不可逆操作 | 强制人工审核 | 销户、删除数据、转账 |

**第二步：智能审批自适应**
```python
class AdaptiveApproval:
    def __init__(self, user_id: str):
        self.user_id = user_id
        self.trust_score = self.load_trust_score(user_id)
    
    def should_require_approval(self, action: Action) -> bool:
        # 1. 按操作类型的基础风险等级
        base_risk = self.get_base_risk(action)
        
        # 2. 根据用户信任分动态调整
        # 老用户、操作历史良好的用户可减少审批
        trust_factor = self.trust_score / 100  # 0~1
        
        if base_risk == "L0":
            return False  # 永远不需要审批
        elif base_risk == "L1":
            return trust_factor < 0.3  # 高信任用户免审批
        elif base_risk == "L2":
            return True  # 始终需要审批
        elif base_risk == "L3":
            return "human_review"  # 始终需要人工审核
        
    def update_trust_score(self, action_result: dict):
        # 用户正确执行操作 → 加分
        # 用户操作异常/可疑 → 减分
        if action_result.success and not action_result.abnormal:
            self.trust_score = min(100, self.trust_score + 1)
        else:
            self.trust_score = max(0, self.trust_score - 10)
```

**第三步：批量授权 + 单次会话授权**
```python
# 用户可授权 Agent 在本次会话中执行特定级别的操作
user: "帮我把地址从北京改到上海，然后订这个商品，不用每次问我了"
agent: "我明白了，本次会话中涉及地址修改和下单操作，我将自动处理。
       涉及退款或删除等高风险操作时，我仍然会请你确认。可以吗？"
user: "可以"
# → 会话级别的 Scope-Based Authorization
```

**第四步：审批体验优化**
- 展示审批原因 + 完整上下文，而不是冷冰冰的"请确认确认"：❌ "请确认此操作" → ✅ "我将把收货地址从'北京市海淀区'改为'上海市浦东新区'，本次申请是因为你要求'换到新家地址'"
- 支持批量审批（"全部同意"）
- 设置审批超时机制（5 分钟无响应 → 自动取消）

**核心原则：** 安全不是"全都 blocking"，而是"正确的事情快速通过，错误的事情坚决拦住"。

### Q10. 请设计一个 Agent 系统的 Tool Retriever（工具检索）机制，当 Agent 有 100+ 个工具时，如何只让 LLM 看到最相关的几个？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、阿里、微软
**答：**

当工具有 100 个以上时，把所有工具 Schema 塞进 Prompt 的弊端很明显：浪费 Token（每个工具 Schema 约 200-500 tokens）、模型选择准确率下降（选择太多反而选错）、延迟增加。

**解决方案：分层 Tool Retrieval 架构**

**第一层：意图路由（粗粒度）**
```python
# 将 100+ 工具按领域分组
TOOL_GROUPS = {
    "customer_service": {
        "description": "客服相关，处理订单、退款、咨询",
        "embedding": embed_model.encode("客服 订单 退款 咨询"),
        "tools": ["query_order", "refund", "return_goods", ...]
    },
    "data_analysis": {
        "description": "数据分析，SQL查询、报表生成",
        "embedding": embed_model.encode("数据分析 SQL 报表 统计"),
        "tools": ["run_sql", "generate_chart", "export_csv", ...]
    },
    "system_admin": {
        "description": "系统管理，用户管理、权限配置",
        "embedding": embed_model.encode("系统管理 用户 权限 配置"),
        "tools": ["list_users", "update_permission", ...]
    },
    # ... 通常 5-10 个分组
}
```

**第二层：语义检索（细粒度）**
```python
class ToolRetriever:
    def __init__(self):
        # 所有工具的向量索引
        self.tool_index = VectorStore()
        for tool_name, tool_schema in TOOL_REGISTRY.items():
            self.tool_index.add(
                id=tool_name,
                embedding=embed_model.encode(tool_schema["description"]),
                metadata={"group": tool_schema["group"]}
            )
    
    def retrieve(self, query: str, top_k: int = 5) -> list:
        # Step 1: 粗粒度——预测需要的工具组
        query_emb = embed_model.encode(query)
        group_scores = {}
        for group_name, group_info in TOOL_GROUPS.items():
            similarity = cosine_similarity(query_emb, group_info["embedding"])
            group_scores[group_name] = similarity
        
        top_groups = sorted(group_scores, key=group_scores.get, reverse=True)[:2]
        
        # Step 2: 细粒度——从选中的工具组中语义检索最相关的工具
        retrieved_tools = []
        for group in top_groups:
            tools = self.tool_index.search(
                query_emb,
                filter={"group": group},
                top_k=top_k
            )
            retrieved_tools.extend(tools)
        
        # Step 3: 合并排序
        return sorted(retrieved_tools, key=lambda t: t.score, reverse=True)[:top_k]
```

**第三层：运行时优化**
```python
class ToolSchemaCompressor:
    """工具 Schema 压缩，减少 Token 消耗"""
    def compress(self, tools: list, context: dict) -> list:
        compressed = []
        for tool in tools:
            schema = tool["schema"]
            # 如果有 default 值的参数，在 Prompt 中省略（LLM 知道用默认值）
            params = [p for p in schema["parameters"] if p.get("required", False)]
            # 只保留必填参数的详细描述
            compressed.append({
                "name": schema["name"],
                "description": schema["description"],
                "required_params": params,
                # 可选参数折叠成一句话
                "optional_params_hint": f"还有 {len(schema['parameters']) - len(params)} 个可选参数"
            })
        return compressed
```

**第四层：反馈闭环**
```python
# 记录工具选择命中率，自动调整检索策略
class ToolRetrievalLearner:
    def record_feedback(self, query: str, retrieved_tools: list, actual_tool_used: str):
        """记录用户实际使用的工具与检索结果"""
        if actual_tool_used in retrieved_tools:
            # 命中：加强该工具的权重
            self.boost(actual_tool_used, query)
        else:
            # 未命中：分析为什么没命中，调整 embedding 或分组策略
            self.log_miss(query, actual_tool_used, retrieved_tools)
```

**Benchmark 数据：**
- 全量注入（100 tools）：Prompt 约 30K tokens，准确率 65%，延迟 2.5s
- Tool Retrieval（top-5）：Prompt 约 1.5K tokens，准确率 88%，延迟 0.3s
- ⟹ Token 节省 95%，准确率提升 23%，延迟降低 88%
