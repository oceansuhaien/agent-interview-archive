### Q1. 在 Agent 系统中，幻觉（Hallucination）比纯 LLM 场景更严重。请至少说出 3 种 Agent 特有的幻觉类型及其缓解策略。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、Anthropic 生态公司、所有金融/医疗 Agent 公司
**答：**

传统 LLM 幻觉研究的是"模型编造事实"，而 Agent 系统中的幻觉有三类独特的放大器：

**1. 推理幻觉（Reasoning Hallucination）**
Agent 在 ReAct 循环中"想象"出了错误的中间推理步骤，然后基于错误推理调用工具。例如 Agent 推理"用户可能是 VIP 用户"然后直接调用了 VIP 专享 API，但实际用户不是 VIP。
**缓解策略：** 关键决策点强制"事实锚定"——Agent 必须先查证再推理，不能凭猜测做决策。

**2. 工具选择幻觉（Tool Selection Hallucination）**
工具数量超过一定阈值后（测试表明 GPT-4 在 20+ 工具时准确率降到 70% 以下），Agent 会选错工具或自己编造不存在的工具名。
**缓解策略：** 工具分层分组（按领域聚类不同工具组），用 Routing Agent 先选组再选具体工具；同时做"工具存在性校验"——调用前检查工具是否在已注册列表中。

**3. 参数填充幻觉（Parameter Hallucination）**
这是 Function Calling 中最隐蔽的幻觉——LLM 自己"脑补"了不存在的参数值，比如给一个不存在的用户 ID 或商品 SKU。
**缓解策略：** 所有参数做实体存在性校验；用 Pydantic Validator 前置验证；返回结构化错误信息让 Agent 自愈。

**分层防御框架（5-Pillar）：** Grounding（锚定）、Validation（校验）、Verification（验证）、Fallback（降级）、Monitoring（监控）——组合使用可实现 2800% 的幻觉降低率（2026 年生产数据）。

### Q2. 你是如何为 RAG 系统选择 Chunking 策略和 Embedding 模型的？请基于 2026 年的最佳实践说明。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 微软、腾讯、字节跳动、所有深度做 RAG 的公司
**答：**

RAG 在 2026 年的一个核心教训是"分块不是按 token 数切的，而是按文档结构切的"（Lilian Weng、LangChain 等多方共识）。

**分块策略选择：**

| 策略 | 推荐场景 | 实现方式 |
|------|---------|---------|
| 语义分块 | 长文档、书籍 | 用 LLM 识别语义段落边界，不跨主题 |
| 结构化分块 | Markdown/HTML/PDF | 按标题层级（H1→H2→H3）切割 |
| 递归分块 | 通用场景 | 先按段落→太长再按句子→再按字符（LangChain 默认方案） |
| Agentic Chunking | 高精度需求场景 | 用小模型判断文档中哪些部分应该合为一块 |

**关键原则**：不是所有块一样大，而是"每个块都是一个完整的信息单元"。块与块之间加 10-15% 的 overlap。

**Embedding 模型选型（2026 年标准）：**

```python
EMBEDDING_CHOICE = {
    "高精度（内部知识库）": {
        "model": "text-embedding-3-large",
        "dimensions": 3072,
        "advantages": "最适合语义密集场景，OpenAI 生态兼容性好"
    },
    "性价比（通用场景）": {
        "model": "voyage-2",
        "dimensions": 1024,
        "advantages": "MTEB 前三，价格仅为 OpenAI 的 1/3"
    },
    "多语言（国际化场景）": {
        "model": "intfloat/multilingual-e5-large",
        "dimensions": 1024,
        "advantages": "开源，中英文混合场景最优"
    }
}
```

**面试加分：** 提到"Embedding 模型选型不要只看 MTEB 分数，要测试你的领域数据。我经历过一个 case：通用排名第一的模型在医药领域反而比领域专用 embedding 差了 15%。建议用小模型做初筛，再用大模型做重排序（Reranker）"。

### Q3. 在多 Agent 系统中，Agent 之间如何通信？请对比三种主流通信协议/模式。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、LangChain 生态公司、微软
**答：**

多 Agent 通信是 2026 年最活跃的研究领域之一。coprep 和 CallSphere 的数据都指向"通信设计不当"是 41%-86% 多 Agent 系统失败的首要原因。

**三种主流通信模式：**

**模式 1：基于消息的发布/订阅（Message Bus）**
```python
# 所有 Agent 通过消息总线通信
class MessageBus:
    def __init__(self):
        self.topics = {}  # topic → [subscribers]
        self.history = [] # 消息历史

    def publish(self, topic: str, message: dict, agent_id: str):
        self.history.append({"topic": topic, "msg": message, "from": agent_id})
        for subscriber in self.topics.get(topic, []):
            subscriber.notify(message)

# Agent 订阅感兴趣的主题
analytics_agent = Agent("analytics")
analytics_agent.subscribe("data_updated")
```
**适用场景：** Agent 数量多、松耦合、需要事件驱动。Argo Events、NATS、Kafka 常见。

**模式 2：直接通信（Peer-to-Peer）**
Agent 直接调用其他 Agent 暴露的 API。LangChain Distributed 模式就是这种——Agent 之间可以直接对话。
**适用场景：** Agent 数量少（< 10）、拓扑结构固定、需要实时协商。

**模式 3：中介通信（Orchestrator 模式）**
所有通信经过一个中心 Orchestrator。LangChain Subagents 模式即此——主 Agent 统一路由全部请求。
**适用场景：** 需要强控制、审计追踪严格、安全敏感。

**选型决策：** 通信模式的选择本质上是在"耦合度"和"调试能力"之间做 trade-off。Message Bus 灵活性最高但调试最困难——"一个消息谁发的、谁收到了、谁忽略了"都需要额外追踪设施。

### Q4. 设计一个用于生产环境的 Agent 监控仪表板。需要追踪哪些关键指标？异常阈值如何设定？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、快手、所有生产 Agent 公司
**答：**

Thinking Company 的 2026 指南明确指出：Agent 监控必须同时看"结果好不好"和"过程有没有异常"。

**三组关键指标：**

**1. 成功/结果指标（告诉你好不好）**
| 指标 | 正常范围 | P0 告警阈值 |
|------|---------|------------|
| 任务完成率 | > 92% | < 85% |
| 用户满意度反馈 | > 4.0/5.0 | < 3.0/5.0 |
| 人肉干预率 | < 5% | 一周内从 5% → 12%（预警信号） |
| 零回复率（Agent 无法回答）| < 3% | > 10% |

**2. 过程/轨迹指标（告诉你为什么不好）**
| 指标 | 正常范围 | 关注点 |
|------|---------|--------|
| 平均推理步数 | 3-6 步 | > 10 步 → 可能死循环或规划低效 |
| 工具调用成功率 | > 90% | < 70% → 外部 API 故障或工具选择出错 |
| 平均端到端延迟 | < 8s | > 15s → 可能需要优化或缓存 |
| Token 消耗/请求 | < 10K | > 30K → 上下文可能膨胀或效率差 |

**3. 异常/安全指标（告诉你要不要紧急处理）**
| 指标 | 要求 |
|------|------|
| prompt injection 检测命中数 | > 0 立即告警 |
| 超出工具权限的动作 | > 0 立即告警 |
| 单用户 Token 消耗暴增 | 对比历史均值 > 3 倍标准差 |

**监控架构设计：**
```python
class AgentMonitor:
    def __init__(self):
        self.metrics = PrometheusMetrics()
        self.alerts = AlertManager(slack_webhook=config.SLACK_WEBHOOK)

    async def record_trace(self, trace: AgentTrace):
        # 记录所有指标
        self.metrics.record("task_completion_rate", trace.completed)
        self.metrics.record("latency_ms", trace.total_time_ms)
        self.metrics.record("tool_call_count", len(trace.tool_calls))
        self.metrics.record("token_usage", trace.total_tokens)

        # 异常检测
        if trace.has_security_violation:
            self.alerts.p0("安全违规检测", trace)
        elif trace.token_usage > trace.history_avg * 3:
            self.alerts.p2("异常 Token 消耗", trace)
```

**关键设计原则：** 不要把监控做成"信息轰炸"——只设 5-8 个核心指标的告警，其他指标做趋势面板供根因分析用。太多告警会导致 alert fatigue，反而忽略真正的问题。

### Q5. 什么是 MCP（Model Context Protocol）？它对 Agent 生态有什么影响？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Anthropic 生态公司、字节跳动、所有构建 Agent 基础设的公司
**答：**

MCP（Model Context Protocol）是 Anthropic 在 2024 年底提出的开放协议，被称为"AI 应用的 USB-C 接口"，2026 年已成为 Agent 工具集成的事实标准。

**核心思想：**
MCP 定义了一个标准化的客户端-服务器协议，让 LLM 应用（MCP Client）可以统一发现和调用外部工具/数据源（MCP Server）。每个工具服务器只需要实现一套标准接口（`list_tools`, `call_tool`），任何兼容 MCP 的 Agent 系统都能直接使用。

**MCP 的三层架构：**
```
┌─────────────────────────┐
│    MCP Client (Agent)    │ ← Agent 系统（如 Claude Code、Cursor）
│  ┌───────────────────┐  │
│  │  MCP Host Layer   │  │ ← 会话管理、连接生命期
│  └───────────────────┘  │
│  ┌───────────────────┐  │
│  │  MCP Transport    │  │ ← stdio / SSE / WebSocket 传输
│  └───────────────────┘  │
└────────┬────────────────┘
         │ MCP Protocol
┌────────▼────────────────┐
│    MCP Server (工具)     │ ← 每个工具都是独立服务器
│  - 文件系统工具           │
│  - 数据库查询工具         │
│  - Web 搜索工具           │
│  - 自定义业务工具         │
└─────────────────────────┘
```

**对 Agent 生态的影响：**

1. **工具发现标准化**：以前每个 Agent 框架都必须手动定义工具 Schema，现在 MCP Server 自动暴露工具列表和参数，大幅减少集成成本。
2. **插件生态爆发**：2026 年已有超过 2000 个公开 MCP Server，从 GitHub 操作到 Figma 设计工具都有。
3. **安全隔离**：每个 MCP Server 运行在独立进程中，Agent 通过协议交互，天然实现了工具层的安全沙箱。
4. **可组合性**：用户可以组合任意 MCP Server，就像搭积木一样构建自己的 Agent 工具栈。

**面试加分：** "MCP 不是完美的。它的弱点在于：(1) 传输延迟比直接 API 调用高 5-10ms；(2) 安全性依赖每个 Server 自身实现，协议层面没有强制的安全校验；(3) 协议仍在演进中，不同版本的 MCP 之间兼容性需要关注。"

### Q6. 如何为 RAG 系统设计一个完整的离线评估流程（不使用 LLM-as-Judge 的情况下）？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 微软、腾讯、Galileo AI 生态公司
**答：**

虽然 LLM-as-Judge 是 2026 年最流行的评估方式，但面试官考察的是你能不能"不用黑盒裁判也能评估 RAG"——这代表你真正理解 RAG 的每一个环节。

**RAG 系统的三阶段评估：**

**阶段 1：检索质量评估（Retrieval Evaluation）**

这是 RAG 特有的评估，与生成无关。核心指标：
```python
def evaluate_retrieval(test_set: list[QueryWithExpectedChunks]):
    results = {"hit_rate": 0, "mrr": 0, "ndcg": 0}

    for query, expected_chunks in test_set:
        retrieved = retriever.search(query, top_k=5)
        retrieved_ids = [r.id for r in retrieved]

        # Hit Rate：期望的文档是否在前 k 个结果中
        hit = any(cid in retrieved_ids for cid in expected_chunks)
        results["hit_rate"] += 1 if hit else 0

        # MRR：第一个命中结果的排序位置
        for rank, rid in enumerate(retrieved_ids, 1):
            if rid in expected_chunks:
                results["mrr"] += 1 / rank
                break

    results["hit_rate"] /= len(test_set)
    results["mrr"] /= len(test_set)
    return results
```

**正常参考值：** Hit Rate@5 > 0.80，MRR > 0.60。

**阶段 2：上下文质量评估（Context Quality）**
即使检索命中，还需要检查检索到的 chunk 是否包含足够信息让 LLM 回答问题：
- **Context Sufficiency**：chunk 是否包含答案所需的全部事实
- **Context Relevance**：chunk 中非相关信息比例（噪声）
- **Context Position Bias**：答案信息是否在 chunk 的中前部

**阶段 3：端到端评估（End-to-End Evaluation）**
不用 LLM-as-Judge，而是用传统 NLP 指标做回归测试：
- **Answer Recall**：答案是否覆盖了黄金标准中的所有事实要点（用实体匹配或关键词匹配）
- **Answer Faithfulness**：答案是否引入了检索内容之外的信息（用字符串对齐检测）
- **Format Compliance**：输出格式是否符合预期 Schema

**面试加分：** "最被低估的评估指标是'检索失败时的 fallback 表现'。测试设置 edge case：故意让 retriever 返回空结果，看 Agent 如何应对——大部分系统在这里暴露出最多问题。"

### Q7. Agent 的 Tool Use 策略中，LLM 如何选择正确的工具？影响工具选择准确率的关键因素有哪些？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | OpenAI 生态公司、Anthropic 生态公司、LangChain 生态公司
**答：**

2026 年 Tool Use 的核心挑战不再是"能不能调用工具"，而是"在 20+ 工具中能不能选中对的"。Anthropic 的研究表明工具数量和选型准确率成反比。

**工具选择的影响因素（按影响度排序）：**

**因素 1：工具名称和描述的语义质量（最大影响因素）**
Anthropic 的核心建议："Keep the format close to what the model has seen naturally occurring in text." 工具描述要像人类自然语言写的那样清晰，而不是机器生成的 Schema 文档。

```python
# ❌ 不好的工具描述
"search_products": "Executes a parameterized SQL query against the product catalog database and returns a JSON-encoded result set"

# ✅ 好的工具描述（贴近自然语言）
"search_products": "搜索商品目录。输入你想找的商品关键词，返回匹配的商品列表，包含价格、库存和商家信息。"
```

**因素 2：工具数量与分组策略**
- 1-10 个工具：平铺即可，准确率 > 90%
- 10-20 个工具：按功能域分组（分组描述 → 组内选择），准确率可恢复到 85%+
- 20+ 个工具：必须用两层路由（Router → Worker），否则准确率 < 70%

**因素 3：工具行为的一致性**
如果同一个工具在不同上下文中表现不一致（比如 `search` 有时候返回结构化数据有时候返回文本），LLM 会困惑。
**解决方案：** 工具 Schema 要严格定义返回格式，用结构化输出（JSON Schema）强制一致性。

**因素 4：Few-shot 示例的质量**
提供 1-2 个工具使用的 Few-shot 示例，效果远好于单纯描述：
```python
# 在系统 prompt 中加入
tools_examples = [
    {"user": "帮我查昨天北京的天气",
     "action": "get_weather(city='北京', date='2026-05-08')"},
    {"user": "帮我查一下订单 ORD-20260508 的状态",
     "action": "query_order(order_id='ORD-20260508')"},
]
```

**因素 5：Agent 推理显式化**
让 LLM 在调用工具前先输出"为什么选这个工具"的推理过程，可以提升 5-10% 的选择准确率（ReAct 模式的隐性优势）。

### Q8. 你的 Agent 系统上线后一小时，发现某个工具调用突然全部返回 500 错误。作为 on-call 工程师，请描述完整的应急响应流程（含回滚、降级、恢复策略）。

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、美团、蚂蚁集团
**答：**

这是 2026 年 Agent 面试的经典场景题——考察"工具依赖故障"的完整处理能力。

**完整应急响应流程：**

**时间线 T+0（发现与评估）— 前 5 分钟**
1. 确认故障范围：是所有工具都挂了还是特定工具？如果是特定工具：
   - 检查该工具服务的状态页 / 健康端点
   - 查看最近 30 分钟的部署记录（是不是刚发了新版本？）
   - 对比该工具的最近调用成功率和延迟趋势

2. 判断影响面：
   ```python
   # 快速计算受影响请求比例
   affected_ratio = tool_failures / total_requests
   # > 30% → P0 级事故
   # 10-30% → P1 级
   # < 10% → 继续监控
   ```

**时间线 T+5（止血与降级）— 前 10 分钟**
3. 执行紧急降级方案：
   - **方案 A（推荐）**：启动该工具的降级代理（Downgrade Proxy），用缓存数据或简化版替代
   - **方案 B**：如果不依赖该工具的 Agent 任务，直接标记该工具"不可用"，让 Agent 跳过相关步骤
   - **方案 C（最保守）**：如果故障影响核心业务流程，临时关闭 Agent 系统切回人工处理

**时间线 T+15（根因定位与修复）— 前 30 分钟**
4. 三种常见根因排查：
   - **工具服务自身故障**：联系工具服务团队，确认是否有宕机/发布/限流
   - **Schema 变更**：工具最近是否改了参数格式？对比故障前后的请求日志
   - **权限/Token 过期**：检查服务间调用的凭据是否过期或被吊销

**时间线 T+30（恢复与复盘）**
5. 恢复后：
   - 先切回旧版本稳定运行 30 分钟
   - 检查积压的任务是否有需要重试的
   - 撰写事故报告（RCA）：根因/影响范围/恢复时间/改进措施

**关键原则：** "先止血再排障"——第一时间做降级/回滚让用户可用，然后再慢慢排查根因。允许调用失败但优雅告知用户，比无限重试让整个系统瘫痪要好。同时建立"工具健康看板"——每次工具调用失败自动记录，提前发现不稳定趋势。

### Q9. Anthropic 在《Building Effective Agents》中提出的 Agent 设计原则中，最核心的几条是什么？如何应用于实际系统？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic 生态公司、所有面试首选
**答：**

Anthropic 这篇 2024 年底的文章被认为是 2026 年 Agent 工程化的"圣经"。以下是泥梨精炼的**四大核心原则**，也是面试官最想听到的：

**原则 1：最简单的方案优先（Simplicity First）**
> "Never add complexity for the sake of complexity. Only use agents when simpler solutions — like direct API calls, prompt chaining, or a single LLM call with retrieval — truly cannot meet your requirements."

实际应用：在考虑 LangGraph 多 Agent 之前，先用一个 `llm.generate(prompt + context)` 试一下。超过 70% 的 Agent 需求可以用单 LLM 调用 + RAG 解决。

**原则 2：将 Agent 视为工具（Agent as a Tool）**
> "The most powerful pattern is not monolithic agents but a main agent that coordinates sub-agents as tools."

实际应用：设计时不要把"智能"分散到每个 Agent，而是有一个"聪明的主 Agent"负责拆解和路由，Worker Agent 做具体的、可预测的工作。

**原则 3：工具设计决定 Agent 质量（Tools Define Capabilities）**
> "Test how the model uses your tools. Keep tool descriptions close to natural language. The quality of your agent is bounded by the quality of your tool design."

实际应用：花 70% 的精力设计工具而不是 Prompt。每个工具写完后至少测试 10 个 edge cases，观察 LLM 是否按预期调用。

**原则 4：渐进式部署（Gradual Deployment）**
> "Start with a simple workflow, verify it works in production, then add agentic elements incrementally."

实际应用：不要第一天就上全自主 Agent。先做 Prompt Chaining → 加缓存 → 加 Routing → 加简单 Agent →再加 Multi-Agent。每一步部署到生产观察 1-2 周，确认收益大于复杂度的成本。

**面试加分：** "Anthropic 这条原则实际上体现了 2026 年 Agent 工程化的最大共识——大多数 Agent 项目失败不是因为能力不够，而是因为过度工程化（Over-engineering）。"

### Q10. 你如何为 Agent 系统做回归测试？请设计一个可自动执行的 Agent 回归测试流水线。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、所有 Agent 平台公司
**答：**

回归测试是 Agent 持续交付中最大的挑战——因为 LLM 输出非确定性，不能像传统软件那样做"预期输出完全匹配"的断言。

**Agent 回归测试流水线设计：**

**第 1 层：单元测试（工具/组件级）**
每级工具独立测试，确保工具功能本身正确：
```python
def test_get_weather_tool():
    result = get_weather("北京", "2026-05-09")
    assert result["city"] == "北京"
    assert "temperature" in result  # 不校验精确值
    assert isinstance(result["temperature"], (int, float))
```

**第 2 层：场景回归测试（Agent 行为级）**
维护 50-100 个黄金测试场景，每个场景用结构化验证：
```python
class ScenarioTest:
    def __init__(self, name: str, input: str, checks: list[Callable]):
        self.name = name
        self.input = input
        self.checks = checks

    def run(self, agent) -> TestResult:
        response = agent.run(self.input)
        failures = []
        for check in self.checks:
            if not check(response):
                failures.append(check.__name__)
        return TestResult(name=self.name, passed=len(failures) == 0, failures=failures)

# 断言示例
scenarios = [
    ScenarioTest(
        name="用户查询天气",
        input="北京今天天气怎么样？",
        checks=[
            lambda r: "北京" in r,           # 城市正确
            lambda r: any(k in r for k in ["温度", "天气", "℃"]),  # 包含天气信息
            lambda r: "上海" not in r,        # 不张冠李戴
        ]
    ),
    ScenarioTest(
        name="退单拒绝（不属于用户的订单）",
        input="帮我退掉订单 ORD-999999",
        checks=[
            lambda r: "不" in r or "无权" in r,  # 应该拒绝
            lambda r: not any(k in r for k in ["已退款", "成功退款"]),  # 不能真的退款
        ]
    )
]
```

**第 3 层：差异测试（Diff Testing）**
每次修改后，跑同样的测试集，对比新旧版本的输出差异：
- 精确格式部分（如 JSON 结构、字段类型）做严格匹配
- 语义部分（如回答内容）做模糊匹配（BLEU / ROUGE / Embedding 相似度）
- 差异超过阈值 → 人工审查

**第 4 层：持续回归流水线**
```
PR 提交 → 单元测试（< 5 min）→ 场景回归测试（< 15 min）→ 差异测试（< 30 min）
                  ↓                      ↓                       ↓
               失败退PR             失败退PR + 自动标注        差异报告发 Slack
```

**面试加分：** "最容易被忽视的是'负面场景测试'——测试 Agent 在面对恶意输入、空输入、超长输入时的行为。我见过很多 Agent 在正常场景表现完美，但用户输入一个空字符串就炸了。"
