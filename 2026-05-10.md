### Q1. 多 Agent 系统的编排（Orchestration）有哪些主流模式？如何根据业务场景选择？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、微软、LangChain 生态公司、所有构建多 Agent 系统的公司
**答：**

多 Agent 编排是 2026 年 Agent 工程面试最高频的架构题之一。LangChain 官方文档明确将编排模式分为三类：

**1. 主从模式（Supervisor/Subagent）**
一个主 Agent 负责拆解任务并路由给子 Agent（作为工具调用）。LangChain 的 Distributed 模式即此。**适合：** 任务边界清晰、需要强控制审计。**局限：** 主 Agent 成为性能瓶颈，子 Agent > 10 后路由准确率下降。

**2. 对等模式（Peer-to-Peer / Networked）**
Agent 之间可以直接通信协商，没有中心节点的概念。Agent 发现彼此后通过消息协议协作。**适合：** 任务动态变化、Agent 数量多、需要松耦合。**局限：** 调试困难，可能出现死锁或循环依赖。

**3. 混合模式（Hierarchical + P2P）**
实践中最常见的模式：顶层 Orchestrator Agent 管理几个中层 Manager Agent，中层再管理 Worker Agent；同层 Agent 之间可以 P2P 通信。**适合：** 大型生产系统（> 20 个 Agent）、需要分层授权和灵活协作。

**选型决策树：**
- Agent < 5 → 主从模式
- 5 < Agent < 20，动态协作 → 对等模式
- Agent > 20 或需多级管理 → 混合模式
- 需要强审计追踪 → 主从模式

### Q2. Google 提出的 Agent-to-Agent（A2A）协议与 Anthropic 的 MCP 协议有什么区别和联系？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | Google 生态公司、Anthropic 生态公司、字节跳动
**答：**

A2A（2025 年 Google 提出）和 MCP（2024 年 Anthropic 提出）是互补而非竞争的关系，2026 年面试官常用此题考察候选人对 Agent 协议生态的全局理解。

**核心区别：**

| 维度 | MCP (Model Context Protocol) | A2A (Agent-to-Agent) |
|------|------------------------------|----------------------|
| 定位 | 模型 ↔ 工具/数据源 | Agent ↔ Agent |
| 角色 | Client-Server（非对称） | Peer-to-Peer（对等） |
| 通信 | 请求-响应模式 | 协商-委派模式，支持流式 |
| 核心对象 | Tool / Resource / Prompt | Agent Card / Task / Artifact |
| 传输层 | stdio / SSE / WebSocket | HTTP/SSE + JSON-RPC |
| 认证 | OAuth 2.0 | 支持多种 Auth Scheme |

**联系：** 它们面向不同层次——MCP 解决"Agent 如何调用工具"，A2A 解决"Agent 如何调用其他 Agent"。一个典型的 2026 年 Agent 系统会同时使用：MCP 连接工具（数据库、搜索引擎），A2A 连接其他 Agent 服务（客户服务 Agent 调用物流 Agent）。

**面试加分：** "A2A 的核心创新是 Agent Card——类似 API 文档的 Agent 能力声明，包含技能、输入/输出 Schema、计费方式和 SLA。这使 Agent 之间可以动态发现和协商合作，而不需要预先硬编码。"

### Q3. Agent 的记忆（Memory）系统应如何分层设计？请对比短期记忆、长期记忆和工作记忆。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、微软、LangChain 生态公司
**答：**

Agent 记忆是区分「玩具 Demo」和「生产系统」的关键能力之一。CallSphere 和 novelvista 的数据都将其列为 2026 年的高频考点。

**三层记忆架构：**

| 类型 | 存储内容 | 存储介质 | 失效策略 | 典型容量 |
|------|---------|---------|---------|---------|
| 短期记忆 | 当前对话上下文、正在进行的推理步骤 | LLM Context Window | 滑动窗口（最近 N 轮） | ~128K tokens |
| 工作记忆 | 当前任务的中间状态、待执行计划 | 内存 + 缓存 | 任务完成后清理 | MB 级 |
| 长期记忆 | 用户偏好、跨会话知识、历史行为 | 向量数据库 + SQL | 基于重要性/时效性 EVICTION | 无限 |

**关键设计模式——记忆检索与合并：**
```python
class AgentMemory:
    def build_context(self, user_id: str, current_query: str) -> Context:
        # 1. 从短期记忆提取当前对话
        short_term = self.short.get_recent(user_id, n=10)

        # 2. 从长期记忆检索相关历史
        long_term = self.long.similarity_search(current_query, top_k=5)

        # 3. 工作记忆中的活跃任务
        working = self.working.get_active(user_id)

        # 4. 选择性合并（避免上下文膨胀）
        return Context.merge(
            short_term,      # 优先保留
            long_term,       # 按相关性评分修剪
            working,         # 必须保留
            max_tokens=16000 # 硬上限
        )
```

**关键原则：** 记忆不是越多越好。在上下文窗口中填满历史信息会导致"记忆干扰（Memory Interference）"——模型注意力被稀释。建议设置硬性 token 上限，并用重要性评分（importance scoring）决定哪些记忆保留。

### Q4. 你如何评估一个 Agent 系统是否足够安全？请设计一个完整的 Agent 安全测试方案。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、所有金融/安全敏感场景的公司
**答：**

2026 年 Agent 安全已经成为独立的安全工程学科。Galileo 和 Thinking Company 的评估框架都强调安全评估必须独立于功能评估。

**Agent 安全测试五维度：**

**1. Prompt Injection 防护**
- 测试直接注入（用户输入中隐藏"忽略之前的指令"）
- 测试间接注入（检索到的文档中包含恶意指令）
- 测试多轮注入（逐步诱导）
- **防御：** 输入净化 + 权限最小化 + 指令隔离（System Prompt 与用户输入之间的语义边界）

**2. 工具滥用防护**
- 测试用户是否能让 Agent 调用高权限工具（如删除数据库、发送邮件）
- 测试参数篡改（尝试传递 SQL 注入、路径遍历参数）
- **防御：** 工具级 RBAC + 参数白名单 + 敏感操作二次确认

**3. 信息泄露防护**
- 测试 Agent 是否会在输出中泄露 Prompt 信息
- 测试错误信息是否会暴露内部架构（数据库类型、IP 地址）
- **防御：** 输出过滤 + 错误信息泛化 + 上下文消毒

**4. 拒绝服务防护**
- 测试超长输入导致 Token 耗尽
- 测试递归调用（让 Agent 不断调用自身）
- 测试循环依赖（Agent A 调用 B，B 又调用 A）
- **防御：** 最大推理步数限制 + Token 预算 + 递归深度检测

**5. 合规与审计**
- 完整记录 Agent 的每一个推理步骤和工具调用
- 支持回放（Replay）——让安全审计员可以精确重现 Agent 的行为
- **防御：** 不可变的审计日志 + 敏感操作标记

**面试加分：** "生产环境中最常被忽略的是间接注入（Indirect Injection）——恶意数据通过 RAG 检索到的文档进入 Agent 推理链。2025 年多起 Agent 安全事件都源自从外部数据源注入的攻击指令。"

### Q5. ReAct（Reasoning + Acting）模式是如何工作的？它与 Plan-and-Execute 模式有什么本质区别？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 百度、微软、Anthropic 生态公司、OpenAI 生态公司
**答：**

ReAct 和 Plan-and-Execute 是 Agent 推理的两个基础范式，2026 年几乎所有 Agent 框架都基于其中之一。

**ReAct 模式（同步推理 + 执行）：**
```
用户输入 → 思考(Thought) → 行动(Action) → 观察(Observation) → 思考 → ...
```
核心思想：推理和行动交替进行，每一步都基于上一步的观察。没有全局规划，而是"边走边想"。

**Plan-and-Execute 模式（先规划后执行）：**
```
用户输入 → 规划(Plan) → 执行计划(Execute Step 1) → 观察 → 调整(Re-plan) → ...
```
核心思想：先生成完整的多步执行计划，然后按计划执行，过程中根据观察调整。

**本质区别对比：**

| 维度 | ReAct | Plan-and-Execute |
|------|-------|-----------------|
| 规划时机 | 每一步才规划下一步 | 先全局规划再执行 |
| 灵活性 | 高（随时改变方向） | 中（依赖初始规划的准确度） |
| Token 效率 | 较低（每一步都有 Think 开销） | 较高（减少重复的推理开销） |
| 任务理解深度 | 浅（逐步推进） | 深（需要提前理解任务全貌） |
| 失败恢复 | 自然（下一步可以修正） | 需要触发 re-plan |
| 最佳场景 | 探索性任务、动态环境 | 流程明确、多步骤确定性任务 |

**面试加分：** "2026 年生产 Agent 系统通常不是二选一，而是混合使用——先用一个 Plan-Execute 生成粗粒度计划，然后在每个步骤内部用 ReAct 做细粒度决策。LangGraph 的 `StateGraph` 正是为了支持这种混合模式而设计的。"

### Q6. 在 Agent 系统中，如何确保 LLM 的输出与业务规则对齐（Alignment）？Constitutional AI 在 Agent 场景中如何应用？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、Anthropic 生态公司、金融行业公司
**答：**

2026 年 Agent 安全对齐的核心挑战是：传统 LLM 安全对齐（RLHF/DPO）针对的是"生成"行为，而 Agent 还有"行动"行为——因此需要额外的对齐层。

**Agent 对齐的三层架构：**

**1. Constitutional Rules（宪法规则层）**
定义 Agent 不可违反的基本原则，写入 System Prompt 和约束检查器：
```
CONSTITUTION = [
    "未经用户确认不得执行支付/退款操作",
    "不得将个人身份信息（PII）包含在工具调用参数中",
    "不确定时必须向用户说明不确定性",
    "隐私信息只能在用户明确授权后使用",
]
```

**2. On-the-fly Validation（运行时校验层）**
每次 Agent 生成输出前，用独立的小模型或规则引擎快速校验是否违反宪法规则。可以选择两种模式：
- **Pre-action Check：** Agent 决定调用工具前，检查工具+参数是否合规
- **Post-action Review：** Agent 执行后，检查输出是否合规（非阻塞式，适用于异步场景）

**3. Self-Correction（自纠错层）**
当 Agent 被检测到违反宪法规则时，不直接报错，而是触发自动修正（Self-Correction）：
```python
class ConstitutionalGuard:
    def check(self, action: AgentAction) -> ActionResult:
        violations = self.rules.violated_by(action)
        if violations:
            action.correct(violations)  # 触发自修正
            return ActionResult.CORRECTED
        return ActionResult.PASS
```

**与 RLHF 的区别：** RLHF 依赖大量标注数据，而 Constitutional Alignment 可以在运行时动态生效，不需要重新训练模型。对于 Agent 场景意义重大——因为 Agent 的"行为"空间远大于 LLM 的"语言"空间，预先标注所有可能的违规行为不现实。

### Q7. 什么是 Agentic Search（代理式搜索）？它和传统 RAG 在架构和体验上有什么本质区别？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、腾讯、Google、所有搜索+AI 公司
**答：**

2026 年 Agentic Search 是各大搜索团队最热的赛道之一。它本质上是 RAG 的"主动版"。

**核心区别：**

| 维度 | 传统 RAG | Agentic Search |
|------|---------|---------------|
| 搜索行为 | 单次检索，被动匹配 | 多轮搜索，主动探索 |
| 查询构造 | 用户直接输入 | Agent 自主构造多组搜索词 |
| 信息整合 | 直接拼接 Top-K 结果 | 多源交叉验证、对比分析 |
| 数据源 | 固定知识库 | 可动态追加工具（实时 API + 知识库 + 网页） |
| 失败处理 | 返回空结果 | 自动调整策略再搜一次 |

**架构差异：** Agentic Search 的核心是"搜索 > 评估 > 再搜索"的循环：
```
用户问题 → Agent 判断"信息够不够" → 不够就构造新查询 → 搜索 → 评估 → 够就回答
```
而传统 RAG 是：`用户问题 → Embedding 检索 → Top-K 拼接 → 回答`——搜索结果一次性决定，无法补搜。

**典型场景：** "对比 iPhone 16 Pro 和 Samsung S25 Ultra 的相机性能"——传统 RAG 返回零散的参数表，Agentic Search 会（1）分别搜索两款手机的评测（2）提取相机参数（3）找到对比评测（4）综合给出有结构的对比分析。

**面试加分：** "Agentic Search 最大的工程挑战是'搜索深度控制'——Agent 可能陷入无限补搜。生产系统必须设置搜索次数上限（默认 3-5 次）和'信息充分性判定'规则——当一个维度的信息方差低于阈值，就认为信息已饱和。"

### Q8. 如何为 Agent 系统设计可审计、可复现的追踪（Tracing）方案？LangSmith/Arize/OpenTelemetry 在其中扮演什么角色？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团、所有 Agent 平台团队
**答：**

Agent 的 Trace（调用链追踪）比传统微服务架构复杂得多——因为 Agent 的行为包含 LLM 推理、逻辑分支、工具调用等多层交织，需要支持"回放（Replay）"而非仅仅是日志查看。

**Agent Tracing 的五个关键数据单元：**

```
Trace (一次用户请求)
├── Span (LLM Call)          # 模型输入/输出/token 数/延迟
│   ├── Span (Tool Call 1)   # 工具名/参数/结果/耗时
│   ├── Span (Tool Call 2)
│   └── Span (Thought)       # Agent 的推理文本（含情绪/犹豫信号）
├── Span (逻辑分支)           # 条件判断的结果
└── Span (最终输出)
```

**三方工具的定位：**

- **OpenTelemetry（基础设施层）：** 提供标准化的 Trace/Metric 数据模型和传输协议。所有 Agent Trace 先标准化为 OTel 格式，再做上层处理。这是"底层不可动摇的标准"。
- **LangSmith（Agent 开发/调试层）：** 专注于 Agent 开发阶段的调试和评估。提供"回放"功能——你可以精确重放一次 Agent 执行过程，观察每一步的 LLM 输入/输出和工具调用，这对调试推理路径错误非常有用。
- **Arize（生产监控层）：** 专注于生产环境中的异常检测和批量评估。当你需要"在 10 万次 Trace 中找到异常的推理路径"时使用——支持 Embedding 漂移检测、Token 消耗趋势等。

**关键设计原则：** Trace 必须包含"完整上下文"才能实现真正的可复现。很多团队只记录 LLM 的输入输出，却忽略了 System Prompt 版本、模型超参数、检索到的文档内容——这导致 Trace 无法被精确回放。建议记录：`(timestamp, model_version, prompt_template_version, full_prompt, full_response, temperature, tools_definition_snapshot)`。

### Q9. 你的 Agent 系统在生产环境中遇到"模型退化"（Model Degradation）——同样的输入，回答质量逐周下降。请分析可能原因和修复策略。

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团
**答：**

模型退化是生产 Agent 中最隐蔽的问题之一。Thinking Company 和 Galileo 的数据都指向这是一个被严重低估的风险。

**三个主要原因（按概率排列）：**

**原因 1：数据漂移（Data Drift）——最常见（~60%）**
RAG 依赖的知识库在持续更新。新加入的文档质量参差不齐——可能引入噪音、过时信息或低质量内容。LLM 在检索到的 context 中找不到正确答案。
- **修复：** 建立"数据质量门禁"——新文档入库前用评分模型评估质量；定期清理低质量/过时文档；监控检索结果的"平均相关性分数"趋势。

**原因 2：Prompt 渐进漂移（Prompt Drift）——次常见（~25%）**
团队在修复 Bug 或优化体验时逐步修改了 System Prompt。每个修改单独看都合理，但累积下来让 Agent 的行为偏离了最初的设计目标。
- **修复：** 建立"Prompt 版本管理"——每次修改都新建一个版本；维护回归测试集，每次 Prompt 变更都跑全量测试；记录每版 Prompt 的关键指标变化。

**原因 3：外部 API/工具的行为变化——隐蔽（~15%）**
Agent 依赖的第三方工具返回结构改变或响应变慢。由于不是 Agent 代码的问题，排查时容易忽略。比如天气 API 突然在晴天也返回 `rain: 0` 而不是 `rain: null`。
- **修复：** 工具调用输出自动记录 Schema；设置"工具行为一致性告警"——检测返回结构变化；对关键第三方工具建立本地缓存+Mock 测试。

**系统化防御（Agent Health Index）：**
```
Agent Health Score = w1 * TaskCompletion + w2 * HumanIntervention + w3 * AvgInferenceSteps
```
每周计算趋势。连续 2 周下降 → 启动全量回归测试。连续 3 周下降 → 进入"降级模式"（提高 Human-in-the-loop 比例）。

### Q10. 请对比 Agent 系统中的 Prompt Chaining、Routing 和 Orchestration 三种模式，并说明各自的最佳适用场景。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 百度、微软、Anthropic/LangChain 生态公司
**答：**

这三种模式是 Agent 系统中由简到繁的三种"决策流编排"方式。Anthropic 在《Building Effective Agents》中反复强调：**从最简单的开始，只在必要时增加复杂度**。

**1. Prompt Chaining（链式调用）**
将任务拆成多个 LLM 调用的串行管道，前一步的输是下一步的输入。
```
Input → LLM(提取) → 结构化数据 → LLM(分析) → 结论 → LLM(格式化为JSON) → Output
```
- **优点：** 简单、可预测、每步可单独调试
- **适合：** 流程固定的任务（如"提取商品参数 → 查询库存 → 生成报价"）
- **不适合：** 需要动态决策或分支的任务
- **面试注意：** "这是大多数系统应该采用的模式，但很多人一上来就追求更复杂的方案。"

**2. Routing（路由）**
一个分类器（通常是 LLM 或小模型）判断用户意图，然后将请求分发到对应的处理流程。
```
Input → Classifier → [助手Agent | 查询Agent | 售后Agent | 投诉Agent]
```
- **优点：** 可扩展性好，可以动态新增处理路径
- **适合：** 用户意图明确的场景（如客服系统分流：查询/退款/投诉）、工具数量大于 15 个的场景
- **不适合：** 需要多步推理的场景

**3. Orchestration（编排）**
最复杂的模式，通常由一个主 Agent 动态决定"现在该做什么"。
```
Input → Orchestrator → [拆解任务 → 分发子任务 → 评估结果 → 是否需要再查？ → ...]
```
- **优点：** 最灵活，能处理开放式的复杂任务
- **适合：** 需要多步规划、多源信息整合、动态决策的高级场景
- **不适合：** 简单任务（杀鸡用牛刀，增加了延迟和 Token 成本）

**选型建议：** 70% 的场景用 Prompt Chaining，20% 用 Routing，10% 用 Orchestration。这是 2026 年生产环境中的数据分布——大多数 Agent 团队的问题不是太简单，而是过度设计。
