### Q1. "目标驱动（Goal-Driven）"和"指令驱动（Instruction-Driven）"的 Agent 系统有什么区别？各自适合什么场景？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、美团、微软、创业公司 CTO 面
**答：**

这是 2026 年 Agent 面试中的「第一道分水岭题」。两个概念经常被人混用，但面试官想听的是你对 Agent 自主性层级的理解。

**指令驱动系统**——Agent 严格遵循用户给出的每一步指令，像一个高级宏脚本。它不做自主规划，只执行明确的步骤序列。

**目标驱动系统**——用户只给出一个目标（比如"帮我预约下周三和客户的会议"），Agent 自主分解为子任务（查日历 → 看客户时区 → 发邀请 → 确认），自主选择工具和执行路径。

**核心区别对比：**

| 维度 | 指令驱动 | 目标驱动 |
|------|---------|---------|
| 用户输入 | 详细步骤 | 高层目标 |
| Agent 自主性 | 低 | 高 |
| 规划能力 | 不需要 | 必须 |
| 容错性 | 步骤出错则失败 | 可动态调整路径 |
| 适用场景 | 流水线、合规流程 | 开放域任务、客服、个人助手 |
| 技术复杂度 | 低 | 高 |

**选型建议：** 金融合规场景必须用指令驱动（每一步可审计），客服/个人助手类场景用目标驱动（用户体验好）。大厂面试官还会追问混合模式——指令驱动作为安全兜底，在权限范围内允许目标驱动自由执行。

**面试加分：** "最好的生产系统往往是两层混合——外层是目标驱动的自由 Agent，内层是一个指令驱动的'安全内核'。自由 Agent 可以规划路径，但关键决策点（付钱、删除数据、修改权限）必须通过安全内核的指令流审批。"

### Q2. Agent 的记忆系统为什么需要分层？请解释 Episodic Memory、Semantic Memory、Procedural Memory 在 Agent 中的实现方式。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | Google、Anthropic、字节 Coze、创业公司
**答：**

人脑不会把"昨天吃了火锅"和"火锅是辣的食物"存在同一个地方——前者是 episodic（情境记忆），后者是 semantic（语义记忆）。Agent 同样需要分层记忆。

**1. Episodic Memory（情境记忆）**
记录具体的事件历史：用户 X 在时间 T 做了操作 A。
- 实现：向量数据库 + 时间戳索引，按 session 聚类
- 检索：最近 N 条 + 语义相似度召回
- 存储成本：高，每条对话都存
- 难点：去重和压缩——不能让记忆无限膨胀

**2. Semantic Memory（语义记忆）**
从 episodic 中抽象出的知识：用户 X 偏好 A 选项、用户 X 的地址是 Y。
- 实现：知识图谱或结构化 Key-Value 存储，定期从历史对话中提取
- 更新策略：新证据权重 > 旧证据；置信度达到阈值才写入
- 关键设计：需要"冲突检测"——新信息和旧记忆矛盾时怎么处理

**3. Procedural Memory（程序记忆）**
Agent 学会的做事的"技能"：处理退款请求时应该按哪几步走。
- 实现：模板库 / 工作流注册表 / 少量示例的 prompt 缓存
- 优化：成功路径被高频复用后固化为一键技能
- 和 RAG 的区别：不是"查知识"，而是"知道怎么做"

**面试加分：** "面试官如果问'记忆膨胀'问题，答案是分级 TTL——Episodic 保留 30 天自动归档，Semantic 通过定期摘要更新，Procedural 只要不失效就一直保留。每条记忆都带置信度分数和最后更新时间戳。"

### Q3. RAG 系统中，除了基础语义检索，高级检索策略有哪些？分别解决什么痛点？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 百度、阿里、知乎、所有知识库类产品
**答：**

基础的 Top-K 余弦相似度检索在 2026 年的面试中已经不够看了。面试官想听的是你对 RAG 检索策略的完整工具箱认知。

**1. 混合检索（Hybrid Search）**
- 语义检索（向量） + 关键词检索（BM25/全文索引）
- 解决痛点：语义检索对专有名词、缩写、精确匹配不敏感
- 实现：分别检索后用 Reciprocal Rank Fusion（RRF）合并排序

**2. 多轮检索（Iterative Retrieval）**
- Agent 先检索一次 → 看结果 → 根据缺失信息再检索一次
- 解决痛点：单次检索无法覆盖复杂问答的所有维度
- 实现：Agent 自主判断是否"信息充足"，不足则构造新查询

**3. 父文档检索（Parent Document Retriever）**
- 检索小的子块 → 返回大的父块
- 解决痛点：小块语义精确但上下文不足，大块上下文丰富但噪音多
- 实现：chunk 分两层（512 tokens 检索用 + 2048 tokens 上下文用）

**4. 查询重写（Query Rewriting）**
- 将用户模糊/隐晦的问题改写为检索友好的查询
- 解决痛点：用户说"上次那个事"——AI 需要理解"上次那个事"是什么
- 实现：用 LLM 把对话历史 + 当前问题改写为独立查询

**5. 假设性文档嵌入（HyDE - Hypothetical Document Embeddings）**
- 先让 LLM 生成假设的理想答案文档，再用该文档的向量去检索
- 解决痛点：查询和文档在语义空间的分布不一致（查询写得很短，文档写得很长）
- 实现：LLM 先生成一个"想象中的答案"，用答案向量检索真实文档

**面试加分：** "在实际生产中最实用的是 混合检索 + 查询重写的组合。我们线上测试显示纯语义检索命中率 78%，加 BM25 到 86%，再加查询重写到 93%。但注意——每多加一个策略就多加一次延迟，需要做延迟预算。"

### Q4. Anthropic 在《Building Effective Agents》中提到"什么时候用框架，什么时候不用"。你如何判断是否应该用 LangChain / CrewAI / 自研框架？

**类型：** 架构决策
**考察频率：** ⭐⭐⭐ | Meta、Apple、Anthropic 生态公司、所有 CTO 面
**答：**

这道题考的不是技术能力，而是工程成熟度——你是否真的 build in production，还是只在 demo 阶段。

**原则：从简单开始，复杂是拒绝出来的。**

**适合用框架的场景：**
- 团队成员 3 人以上，需要统一的工具调用抽象层
- Agent 需要复杂的状态机管理（多 step、分支、重试）
- 团队需要快速验证原型，在 2 周内上线 MVP
- 你需要现成的 Guardrails、Callbacks、Tracing

**不适合用框架的场景：**
- Agent 逻辑很简单（1 个 prompt + 1 个 tool）——可直接用 LLM API
- 需要极致的延迟优化——框架的抽象层有额外开销
- 需要非常规的 Agent 行为——框架限制了灵活性
- 安全合规要求极高——你无法审计框架内部的黑盒代码

**Anthropic 的核心建议：**
1. 优先直接调用 LLM API + 手写工具逻辑
2. 当出现重复模式时，再引入框架的特定组件
3. 不要一开始就套上整个框架全家桶

**面试加分：** "最关键的一个判断标准是：你的 Agent 是否需要复杂的长期状态管理（multi-turn with state checkpointing）？如果需要，用框架（LangGraph 的 StateGraph 非常适合）。如果只是 request-response 模式，手写就好了。框架的价值在于帮你管理复杂状态，不帮你写好 prompt。"

### Q5. 如何设计 Agent 系统的 Human-in-the-Loop（HITL）生产指标？怎样量化"人类干预"来驱动系统改进？

**类型：** 生产运维
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、京东、字节、所有金融/医疗 Agent
**答：**

HITL 不是一种功能，而是一种可量化的运维策略。面试官想听的不是"我们要有人审核"，而是"审核率每上升 1% 意味着什么"。

**核心指标：**

**1. HIR（Human Intervention Rate）**
- 定义：每 100 个 Agent 任务中，人类需要介入修改或覆盖的次数
- 正常范围：简单任务 < 5%，金融/医疗任务 10-20%（取决于合规要求）
- 意图恶化信号：HIR 突然从 5% 跳到 12% → 模型退化或上游数据变化

**2. 干预分类统计**
- 类型 A：Agent 执行正确但合规要求必须人工审核（可自动化，但政策不允许）
- 类型 B：Agent 执行错误被人工纠正（需要改进模型）
- 类型 C：人工主动介入不是因为 Agent 出错，而是因为用户要求（体验设计问题）
- 重点追踪：B 类干预率的下降趋势

**3. 干预回流率**
- 定义：人工修改后的任务，有多少比例被用户再次修改
- 信号：高回流率 → 人工审核本身也有问题，需要优化审核模板

**4. Approval Latency（审批延迟）**
- 定义：从 Agent 请求审批到人工响应的时间
- 目标：SLA 内完成率（如 90% 在 5 分钟内响应）
- 优化：自动升级——超时 5 分钟升级到主管，10 分钟降级为默认 safe 操作

**面试加分：** "线上 HITL 最容易被忽视的是 '审核疲劳'——人工审核员每小时看 50+ 次审批，注意力必然下降。解决方案是 '置信度分级'：低置信度操作随机抽审 100%，中置信度抽审 20%，高置信度自动放行。这样既保证安全又减少审核员疲劳。"

### Q6. 在多 Agent 系统中，如何设计 Agent 间的"协商"机制——当两个 Agent 对同一个问题的答案不一致时，如何解决冲突？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 创业公司、复杂工作流系统、多 Agent 团队
**答：**

这道题的考点不是"路由"——路由是给谁做的问题。这里的考点是"两个 Agent 各执一词"的时候怎么办。

**层级方案（从简单到复杂）：**

**Level 1：仲裁 Agent**
一个独立的仲裁 Agent 听取两方观点，参考预定义规则，做最终裁决。
- 优点：实现简单
- 缺点：仲裁 Agent 本身也有偏见，且增加延迟

**Level 2：辩论式协商**
两个 Agent 各自出论据，每轮交换观点，3 轮后投票或交由仲裁。
- 灵感来源：Anthropic 的 Debate 研究
- 关键设计：论据必须带证据引用（知识来源），不能是靠"我觉得"
- 优点：信息充分，双方观点都被听到
- 缺点：延迟高（3 轮 × 2 个 Agent）

**Level 3：置信度协商**
每个 Agent 给出回答时附带置信度分数 + 推理链。比较置信度，高者胜出。
- 需要：Agent 的校准能力（置信度 = 真实准确率）
- 优点：延迟低，一次交互即可
- 缺点：Agent 可能过度自信（校准不准确）

**Level 4：退化成单 Agent**
如果协商成本超过了重新执行的代价，直接"重启"——把冲突上下文交给一个新的 Agent 从头处理。
- 适用：低风险场景，协商延迟 > 业务容忍度

**面试加分：** "我们线上的实现是辩论 + 证据校验。Agent A 和 B 先各自出论点带引用，然后互相检查对方的引用是否真的支持其结论（引用验证）。最后仲裁 Agent 只看引用被验证通过的论点做决策。这个设计的核心洞察是：'谁的引用更可靠'比'谁的论点更动听'重要得多。"

### Q7. Agent 的自省（Introspection）和批判（Critique）机制是什么？如何在生产 Agent 中实现"自我纠错"？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | OpenAI 生态、Anthropic 生态、字节、Google
**答：**

自省和批判是 Agent 用来避免"一条道走到黑"的关键机制——不等到最终输出出错，在过程中就自我纠正。

**自省（Introspection）：** Agent 在执行过程中不断问自己"我在做什么？我的推理合理吗？我有没有遗漏关键信息？"
**批判（Critique）：** 对已完成的工作进行检查——"这个工具调用的输出合理吗？这个答案有没有潜在幻觉？"

**生产级实现方式：**

**1. ReAct 循环中的 Critic 步骤**
```python
# 伪代码：ReAct + Critic
while not task_complete:
    if confidence < threshold:
        reasoning = llm.reason(state)  # 推理
        action = llm.plan(reasoning)   # 行动
        observation = execute(action)  # 执行
        critique = llm.judge(action, observation, expected_outcome)  # 批判 ← 新增
        if critique["needs_revision"]:
            add_feedback_to_state(state, critique["feedback"])
            continue  # 不进入下一步，重新规划
        state.update(action, observation)
```
- 关键：批判者不决策，只做 QA——防止循环失控

**2. 双重模型（Strong Critic + Fast Actor）**
- Actor：用低成本模型（如小模型）快速执行
- Critic：用高质量模型（如大模型）审核执行结果
- 成本优化：Actor 执行 10 次，Critic 抽审 3 次

**3. 结构化反思模板**
- 每个关键步骤后，Agent 回答三个问题：
  - 我目前有没有足够的信息？
  - 我有没有忽略替代假设？
  - 如果我的推理错了，最可能错在哪里？
- 这是 Google DeepMind 在 Self-Refine 论文中的核心思想

**面试加分：** "自省机制最大的陷阱是 '过度反思'——Agent 对每个决策都反思，导致行动延迟膨胀 3-5 倍。我们的经验是：只在高风险决策点（调用写操作 API、涉及用户隐私、金额操作）开启强制自省，低风险操作只做日志不做阻断。"

### Q8. 生产环境中 Agent 的 Tool Calling 如何做"参数校验"？请设计一个让 LLM 的幻参数在到达 API 之前就被拦截的架构。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 蚂蚁集团、字节、所有金融 Agent
**答：**

这是老问题（Q6 of Day 005 提到参数幻觉），但这里聚焦的是具体**拦截架构**——不只是一个理念，而是可落地的代码级设计。

**架构：三层参数校验管道**

```
LLM 生成参数 → Layer 1: Schema 校验 → Layer 2: 语义校验 → Layer 3: 实体校验 → API 调用
                      ↓ 失败             ↓ 失败             ↓ 失败
                   retry/修复          retry/修复          上报 + 人工
```

**Layer 1：Schema 校验（毫秒级）**
- 用 Pydantic / Zod 检查参数类型、格式、必填项
- 示例：`order_id` 必须是 UUID 格式 vs 用户传了"12345"
- 修复方式：自动格式化（"12345" → "00000000-0000-0000-0000-000000012345"）
- 拦截率：约 30% 的错误参数

**Layer 2：语义校验（百毫秒级）**
- 检查参数值的"合理性"——值在合理范围内？
- 示例：`amount=-50`（金额负数）被拦截；`date=1900-01-01`（时间太旧）被拦截
- 实现：规则引擎 + 小模型推理（不是大模型，成本太高）
- 拦截率：再拦截约 40%

**Layer 3：实体存在性校验（参考延迟，取决于数据源）**
- 检查参数对应的实体（用户、订单、商品）是否真的存在
- 示例：`user_id=abc123` → 查数据库，该用户不存在 → 拦截
- 实现：对关键字段做到"查一次确认存在"再放行
- 拦截率：最后约 25%，只有约 5% 的错误参数会漏网

**面试加分：** "最关键的设计决策是——Layer 1 和 Layer 2 失败后，Agent 自动重试（重新生成参数），但 Layer 3 失败后直接上报不重试。因为实体不存在时重试也没用，需要人工判断是用户给错了 ID 还是 Agent 产生了彻底幻觉。自动重试反而会把一次错误放大为多次 API 调用。"

### Q9. 你们团队用 LLM-as-Judge 评估 Agent 时，遇到过哪些"评判者偏差"？如何缓解？

**类型：** 评估体系
**考察频率：** ⭐⭐⭐ | 字节、百度、中等规模 Agent 团队
**答：**

LLM-as-Judge 已经成了 Agent 评估的标准做法（Galileo、LangSmith、自研系统都在用），但面试官想听的是你对它**缺陷**的深刻理解。

**常见评判者偏差：**

**1. 自我偏好偏差**
- 现象：Judge 模型更喜欢和自己风格相似的输出
- 例：GPT-4 给 GPT-4 生成的长篇回答打高分，给 Claude 的简洁回答打低分
- 缓解：使用与 Actor 不同的模型做 Judge（如用 Claude 评判 GPT-4 Agent）

**2. 位置偏差（Position Bias）**
- 现象：Judge 倾向于选择 A/B comparison 中排在前面的
- 缓解：做双向对比——A vs B 和 B vs A 各跑一次，取平均

**3. 长度偏差（Verbosity Bias）**
- 现象：Judge 更喜欢更长的回答，即使更长不代表更好
- 缓解：在评分 prompt 中强调"请忽视长度差异"，或控制 Agent 回答长度一致

**4. 角色偏差（Role Bias）**
- 现象：Judge 对"专家语气"的回答有天然好感
- 缓解：在评估 rubric 中明确"正确性 > 表达自信度"

**5. 系统 prompt 泄漏**
- 现象：Judge 的系统 prompt 如果包含"你是严格的评分员"，其评分分布会整体下移
- 缓解：严格控制评估用的 prompt 模板，做 ablation 测试

**面试加分：** "最实用的一招是——不要用 LLM-as-Judge 做单点评分（1-10 分），而是做配对对比（A vs B, which is better?）。配对对比的评分者间一致性（inter-rater reliability）比单点评分高约 20%。而且 Google 的论文也证明了配对对比更接近人类标注者的判断。"

### Q10. 如何为 Agent 系统做"压力测试"——测试 Agent 在高负载、高并发、恶劣输入条件下的行为？请设计一套完整的压力测试方案。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐ | 美团、字节、滴滴、高流量场景
**答：**

面试官问这个问题是因为——demos 都在 1 个用户的理想场景跑，而生产环境要面对 1000 个用户同时捣乱。

**压力测试的 4 个维度：**

**1. 负载测试（Load Testing）：到底能承受多少并发？**
- 指标：并发用户数、每秒请求数(APS)、P50/P95/P99 延迟
- 方法：用 k6 / Locust 模拟 10→50→100→500→1000 并发
- 目标：找到"拐点"——延迟开始飙升的并发数
- Agent 特有注意：LLM API 调用是单并发还是可并行？LLM API 本身有 Rate Limit

**2. 压力测试（Stress Testing）：超载后的行为**
- 测试：超过设计极限的 2-5 倍负载
- 关注点：是否优雅降级？队列满了是丢弃还是阻塞？LLM API 超时后怎么处理？
- 必须验证：不会级联崩溃（一个 Agent 的 LLM 调用超时不会导致整个系统挂掉）

**3. 恶劣输入测试（Adversarial Input Testing）**
- 超长输入：10 万字输入（Agent 的 Context Window 能否处理）
- 空输入：用户只发了一个句号
- 多语言混合：中文 + 英文 + emoji + 乱码
- 恶意指令：Prompt Injection 尝试
- 重复指令：同一请求发 100 次

**4. 混沌测试（Chaos Testing）：组件失效场景**
- LLM API 挂了 → Agent 是否自动切换到备用模型
- 向量数据库挂了 → RAG 是否需要降级到 BM25
- 工具 API 返回 500 → 重试逻辑是否正确
- 缓存服务器挂了 → 是否影响核心功能

**面试加分：** "我最推崇的做法是 '灰度压测'——在线上用 1% 流量开启一个新的 Agent 版本，观察延迟、错误率、HIR 指标。如果 1% 没问题，升到 5%，再到 20%。这样压测本身就是生产流量，比在测试环境压 10 万次虚假请求更有说服力。唯一的要求是——新版本必须有 '一键回滚' 的能力。"
