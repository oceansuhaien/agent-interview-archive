### Q1. 如何设计 Agent 的 Tool Schema 才能最大化 LLM 的工具调用准确率？Anthropic 推荐的「自然语言优先」原则具体如何落地？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、字节跳动、所有 Agent 生产团队
**答：**

Anthropic 2026 官方指南《Building Effective Agents》明确指出：工具描述的格式越贴近自然文本，模型调用的准确率越高。这是因为 LLM 在海量的自然文本上训练，对 JSON Schema 这种机器格式的理解远不如自然语言。

**自然语言优先原则的具体落地：**

```python
# ❌ 坏的 Tool Schema —— 纯 JSON Schema
{
  "name": "search_products",
  "description": "Search products in catalog",
  "parameters": {
    "type": "object",
    "properties": {
      "q": {"type": "string", "description": "The search query"},
      "cat": {"type": "integer", "description": "Category ID"}
    }
  }
}

# ✅ 好的 Tool Schema —— 自然语言描述
{
  "name": "search_products",
  "description": """在商品目录中搜索产品。query 参数支持关键词匹配和模糊搜索。
category_id 是可选的，传入后会在指定类目内搜索。
示例：search_products(query="红色连衣裙", category_id=5) 在服装类目搜红色连衣裙。""",
  "parameters": {
    "query": {
      "description": "用户的搜索关键词，支持中文和英文",
      "type": "string"
    },
    "category_id": {
      "description": "商品类目 ID，不传则在全站搜索。类目 5=服装，3=电子产品",
      "type": "integer"
    }
  }
}
```

**面试加分：** 「Anthropic 还强调：工具描述中要包含**使用条件（When to use）**和**不使用条件（When NOT to use）**。比如在 search_products 的描述中加一句『不要用于查询订单状态，那是 track_order 工具的工作』能显著减少工具选择错误。另外，**工具数量要克制**——从 5 个工具增加到 50 个，工具选择的准确率会下降约 15-20%。如果工具真的很多，考虑用分层结构（先 Router 再 Specialist）。」

### Q2. 在 Agent 系统中，Context Window 管理有哪些成熟策略？如何设计 Token Budget 机制防止 Agent 在单次会话中 Token 爆炸？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、Anthropic、OpenAI、所有生产 Agent 团队
**答：**

Context Window 管理是 2026 年 Agent 系统最重要的工程挑战之一。CallSphere 和 novelvista 都指出：Agent 在一个复杂任务中可能消耗 50K-200K tokens，远超单次会话的合理预算。

**Context Window 的四大管理策略：**

**策略 1：滑动窗口（Sliding Window）**
```python
class SlidingWindowContext:
    def __init__(self, max_tokens=32000):
        self.max_tokens = max_tokens
        self.history = []

    def add(self, message: dict):
        self.history.append(message)
        # 当超出限制时，丢弃最早的对话
        while self.token_count(self.history) > self.max_tokens:
            self.history.pop(0)  # 丢弃最早的消息
```

**策略 2：摘要压缩（Summarization Compression）**
当 Context 接近上限时，自动将早期消息压缩为摘要：
```python
compressed_summary = llm.summarize(
    f"将以下对话压缩为 200 字摘要，保留关键决策状态和已完成的工具调用结果：{early_history}"
)
# 用摘要替换被压缩的原始消息
```
适用于对话历史较长的客服 Agent 或多步骤任务 Agent。

**策略 3：结构化 Token Budget**
```python
TOKEN_BUDGET = {
    "system_prompt": 2000,         # 系统提示
    "conversation_history": 8000,  # 对话历史
    "retrieved_documents": 12000,  # 检索到的文档
    "tool_results": 5000,          # 最近工具调用结果
    "scratchpad": 3000,            # Agent 的推理记录
    "reserved": 2000,              # 保留给输出
}
TOTAL_LIMIT = 32000  # 所有部分的和
```
每个部分独立限制，一个部分耗尽不会挤占其他部分的空间。

**策略 4：选择性遗忘（Selective Forgetting）**
- 保留所有工具调用的**结果摘要**（50 tokens/条），丢弃原始完整输出
- 只保留最后 3 轮完整对话，更早的用摘要替代
- 对检索文档只保留最相关的 top-3，其余的丢弃

**面试加分：** 「最容易被忽视的是 **Token 分配的动态性**——复杂任务在检索阶段需要多分配 Token，简单任务则可以直接把更多 Token 给对话历史。推荐实现一个『自适应分配器』：在任务执行中动态监控各部分的实际消耗，并在系统提示中注入 Token Budget 使用报告，让 LLM 自己决定如何节约使用。」

### Q3. 请整理一个完整的 Agent 失败模式分类学（Failure Taxonomy）。生产中最常见的 Top-5 失败模式是什么？各自的检测和缓解方案是什么？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、Galileo 生态公司
**答：**

Galileo 和 Thinking Company 的评估资料都提到了 Agent failure 的系统化分类，但没有给出完整的 Taxonomy。以下是基于业界实践的完整分类。

**Agent Failure Taxonomy（六类，每类含子类）：**

**1. 工具失败（Tool Failures）—— 最常见（约占 35%）**
| 子类 | 表现 | 检测 | 缓解 |
|------|------|------|------|
| 工具选择错误 | 用 search_products 查订单 | LLM-as-Judge 检查工具选择 | 工具描述加使用条件 |
| 参数幻觉 | 传不存在的 user_id | 调用前 Schema 校验 | 参数枚举校验 |
| 工具循环调用 | 反复调同一个工具 | 模式检测（N 次相同） | 熔断 + 强制换策略 |

**2. 推理失败（Reasoning Failures）—— 约占 25%**
| 子类 | 表现 | 缓解 |
|------|------|------|
| 循环推理 | Thought 反复评估但不行动 | 步骤上限 |
| 确认偏误 | 倾向支持已有结论的检索结果 | 引入反驳 Agent |
| 跳步推理 | 跳过关键步骤直接出结论 | 结构化推理模板 |

**3. 检索失败（Retrieval Failures）—— 约占 18%**
- 零结果检索、低相关性检索、过时信息检索
- **缓解**：CRAG（Corrective RAG）对检索结果做相关性校验

**4. 协调失败（Coordination Failures）—— 约占 10%**
- 死锁、任务重复、信息不一致
- **缓解**：Orchestrator 维护全局 DAG

**5. 安全失败（Safety Failures）—— 约占 8%**
- Prompt Injection 绕过、越权操作
- **缓解**：分层安全 + Guard Model

**6. 性能失败（Performance Failures）—— 约占 4%**
- 超时、高延迟、Token 超出限制
- **缓解**：Token Budget + 超时熔断

**面试加分：** 「生产中最有用的经验是：**70% 的失败可以通过更好的 Tool Schema 和 System Prompt 解决**，不需要改架构。另外，建议团队建立「失败案例库（Failure Library）」——每次生产事故都抽象为一个标准化的 test case 加入回归测试集。Galileo 强调：从生产故障中提取 test case 比从教科书上抄更有价值。」

### Q4. Google A2A（Agent-to-Agent）协议和 Anthropic MCP（Model Context Protocol）在定位上有什么本质区别？在多 Agent 系统中，它们如何协同工作？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | 微软、Google、Anthropic、字节跳动、所有 Agent 平台团队
**答：**

A2A 和 MCP 是 2026 年最重要的两个 Agent 协议，但很多面试者会混淆它们的定位。

**本质定位区别：**

```
MCP: Model ↔ Tool（模型到工具的连接协议）
A2A: Agent ↔ Agent（Agent 到 Agent 的协作协议）

MCP 的源起：Anthropic 发现「每个 LLM 都要重复接入 API」
A2A 的源起：Google 发现「不同 Agent 系统需要互相发现和协作」
```

**MCP（Model Context Protocol）—— 垂直集成**
- 连接 LLM 和「工具/数据源」
- Agent 通过 MCP Client 发现和调用外部工具
- 关键能力：动态工具发现、资源订阅、Prompt 模板
- 类比：USB-C（统一的设备连接标准）

**A2A（Agent-to-Agent）—— 水平协作**
- 连接不同的 Agent 系统
- Agent 通过 A2A 发布自身能力、协商协作契约
- 关键能力：能力发现、契约协商、流式通信、安全认证
- 类比：TCP/IP（节点间的通信协议）

**协同工作架构：**
```python
# Agent 内部用 MCP 调用工具
class CustomerAgent:
    async def handle_query(self, user_query):
        # MCP: 通过 MCP Server 调用数据库工具
        orders = await self.mcp_client.call_tool("query_orders", user_id=uid)
        return self.generate_response(orders)

# Agent 之间用 A2A 协作
class OrchestratorAgent:
    async def coordinate(self, query):
        # A2A: 发现其他 Agent 的能力
        agents = await self.a2a_discovery.discover(
            capabilities=["return", "shipping", "payment"]
        )
        # A2A: 协商契约并委托任务
        contract = await agents["shipping"].negotiate_contract(query)
        result = await agents["shipping"].execute(contract)
        return result
```

**面试加分：** 「最精辟的理解是：MCP 管『Agent 吃什么』（数据/工具），A2A 管『Agent 和谁玩』（协作）。在生产系统中，MCP 是基础设施必选项（没有 MCP 也可以用 Function Calling），A2A 是大型多 Agent 生态的可选项。Google 的 Agent Card 概念值得单独提——类似微服务的 Service Discovery，Agent 通过 Agent Card 声明名称、能力列表、认证方式、速率限制、P99 延迟等，其他 Agent 可以基于这些信息决定是否协作。」

### Q5. CallSphere 将 Agent 记忆系统分为哪三种类型？在 Multi-Agent 系统中，如何设计跨 Agent 的「共享记忆」与「私有记忆」隔离？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | CallSphere 生态公司、字节跳动、蚂蚁集团
**答：**

CallSphere 2026 面试指南将 Agent 记忆系统划分为三种类型，并指出：在 Multi-Agent 系统中，记忆隔离比记忆共享更重要。

**三种记忆类型：**

**1. Episodic Memory（情景记忆）—— 类比「昨天做了什么」**
- 存储 Agent 过去执行的完整任务记录
- 包含：输入、推理过程、工具调用、输出
- **用途：** 从过往经验中学习、审计追溯
- **存储：** 向量数据库 + 元数据索引

**2. Semantic Memory（语义记忆）—— 类比「太阳从东边升起」**
- 存储 Agent 从任务中提取的抽象知识和规则
- 包含：业务规则摘要、常见问题解决方案、工具使用技巧
- **用途：** 减少重复推理、加速同类任务
- **存储：** 知识图谱或结构化数据库

**3. Procedural Memory（程序记忆）—— 类比「怎么骑自行车」**
- 存储 Agent 对「如何完成任务」的隐式知识
- 包含：任务分解模板、工具调用序列模式
- **用途：** 提高效率、减少思考步骤
- **存储：** Prompt 模板、Few-shot 示例库

**跨 Agent 的记忆隔离设计：**
```python
class MemoryManager:
    def __init__(self):
        self.shared_memory = SharedMemory()   # 所有 Agent 可读
        self.private_memory = {}              # 每个 Agent 独享

    async def write(self, agent_id: str, memory: Memory):
        if memory.scope == "shared":
            await self.shared_memory.write(
                memory, accessible_by=["agent_a", "agent_b"]
            )
        else:
            # 每个 Agent 的私有记忆被完全隔离
            await self.private_memory[agent_id].write(memory)

    async def read(self, agent_id: str, query: str) -> list[Memory]:
        shared = await self.shared_memory.search(query)
        private = await self.private_memory[agent_id].search(query)
        return shared + private  # 先共享、后私有，覆盖顺序决定优先级
```

**面试加分：** 「最难的设计决策是『什么时候记忆应该共享，什么时候应该隔离』。经验法则：
- **事实性知识**（产品规格、业务规则）→ 共享
- **用户关联知识**（特定用户的偏好）→ 仅对话 Agent 可访问
- **推理模式**（Agent 自己总结的『最佳实践』）→ 私有
- 记忆的 TTL（生存时间）必须明确——共享记忆的 TTL 通常比私有记忆长。CallSphere 还强调：记忆需要有「重要性过滤」——不是所有对话都值得记住，系统应自动判断什么是『值得记住的』。」

### Q6. 什么是 Agent 的策略迁移（Policy Transfer）能力？如何评估一个 Agent 在新领域上的泛化能力，而不需要重新设计整个系统？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 微软、字节跳动、Google DeepMind 生态公司
**答：**

2026 年，企业希望 Agent 能从「单一场景」扩展到「多场景」，而不是为每个场景从头开发。策略迁移（Policy Transfer）衡量的是：Agent 在源领域学到的规划、工具使用和推理模式，能否有效迁移到目标领域。

**策略迁移的三种模式：**

**模式 1：零样本迁移（Zero-shot Transfer）—— 最理想**
```
源 Agent：电商客服（查订单、退货、退款）
目标 Agent：金融客服（查账户、转账、冻结）
迁移方式：更换知识库 + 修改工具列表，Agent 的「客服思维」保留
难度：低（但准确率通常从 92% 降到 75-80%）
```

**模式 2：少样本迁移（Few-shot Transfer）—— 最常用**
在目标领域提供 10-20 个示例，让 Agent 快速适配：
```python
class FewShotTransfer:
    def adapt(self, source_agent, target_domain) -> Agent:
        new_agent = clone_agent(source_agent)
        # 更换系统提示中的领域定义
        new_agent.system_prompt = self.replace_domain_context(
            new_agent.system_prompt, target_domain
        )
        # 注入目标领域的 few-shot 示例
        new_agent.few_shot_examples = self.get_domain_examples(
            target_domain, n=15
        )
        return new_agent
```

**模式 3：渐进式迁移（Progressive Transfer）—— 最高质量**
分阶段迁移：先迁移简单的子任务，逐步增加复杂度。每次只改 1-2 个工具/规则，保持其他不变。

**泛化能力评估矩阵：**
```python
EVAL_MATRIX = {
    "tool_selection_accuracy":    # 在新领域正确选择工具的比率
    "planning_coherence":         # 规划是否适应新领域的业务流程
    "fallback_rate":              # 遇到新领域陌生情况时的求助频率
    "adaptation_speed":           # 从首次使用到稳定表现需要的轮数
}
```

**面试加分：** 「策略迁移的瓶颈通常不在 LLM 的能力，而在**工具层面的适配**——新领域的工具虽然语义相似（都是『查数据』），但参数 Schema 完全不同。解决方法是设计**统一的工具抽象层（Tool Abstraction Layer）**：将『查询、写入、删除』等通用操作抽象为标准接口，不同领域的实现作为具体适配器。这样 Agent 只需要学会『什么时候查询』，不用重新学『怎么查询』。」

### Q7. 如何为 Agent 系统设计满足金融/医疗等强监管行业的日志审计与合规体系？Agent 的「决策可解释性」要求如何落地？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、微软、所有受监管行业 Agent 团队
**答：**

Thinking Company 和 Galileo 都强调：受监管行业的 Agent 日志不能只是「调试用」，而必须是**法定合规证据**。

**合规日志体系的三层设计：**

**Layer 1：完整追溯（Immutable Audit Trail）—— 不可篡改**
```python
class ComplianceLogger:
    def __init__(self):
        self.ledger = AuditLedger()  # 不可变存储

    async def record_action(self, action: AgentAction):
        entry = {
            "timestamp": datetime.utcnow().isoformat(),
            "session_id": action.session_id,
            "agent_id": action.agent_id,
            "action_type": action.type,
            "input": self.sanitize_pii(action.input),    # PII 脱敏
            "output": action.output,
            "model_used": action.model,
            "confidence": action.confidence,
            "human_review_id": action.review_id,
            "hash": self.compute_hash(action),            # 防篡改哈希链
            "previous_hash": self.last_hash,               # 区块链式链接
        }
        self.last_hash = entry["hash"]
        await self.ledger.append(entry)
```

**Layer 2：决策可解释性（Decision Explainability）**
Agent 每个关键决策必须附带「理由记录」：
- 工具调用时：记录「为什么选这个工具」（Thought 原文）和「为什么不选其他工具」
- 拒绝操作时：记录「为什么拒绝」的规则引用（如「此操作超出 Agent 权限范围，依据安全策略第 3.2 条」）
- 降级处理时：记录「什么条件触发了降级」

**Layer 3：合规报告自动生成**
每月自动生成合规报告：
```
要求：列出本月 Agent 的所有「高风险操作」
输出：每个高风险操作的完整决策链 + 人工审批记录 + 事后审计结论
```

**与可观测性的区别：**
- 可观测性 = 找问题用的（可删，可抽样）
- 合规日志 = 作为证据用的（不可删，100% 记录）
- **关键差异：** 合规日志必须支持「独立审计者仅凭日志就能完全理解 Agent 的决策过程」

**面试加分：** 「最容易被忽视的是 **PII 脱敏与合规的矛盾**——合规需要记录完整操作，数据保护法要求脱敏。解决方案是『双层存储』：第一层存脱敏后的合规日志（用于日常审计），第二层存加密后的原始数据（密钥由合规委员会管理，仅在审批流程中解密）。另一个落地要点：Agent 的决策树必须有『否决回溯点』——如果 Agent 的推理链中某一步的信息来源被后续认定为不可靠（比如检索到已废弃的文档），系统能标记该整条决策链为『待人工复核』。」

### Q8. 如何为 Agent 的 Prompt 系统设计版本管理与 A/B 测试体系？线上 Prompt 变更如何降低风险？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、所有大规模 Agent 团队
**答：**

2026 年，Agent 团队最大的痛点之一是 Prompt 变更缺乏管控。Coprep 的数据显示：超过 60% 的 Agent 生产事故由未经测试的 Prompt 变更引起。

**Prompt 版本管理四要素：**

**要素 1：版本化存储**
```python
# 每个 Prompt 像代码一样管理
PROMPT_REGISTRY = {
    "customer-agent:v1.2.3": {
        "content": "你是客服助手...",
        "author": "alice@company.com",
        "created_at": "2026-05-10",
        "parent": "customer-agent:v1.2.2",
        "change_log": "修复：处理退款超时场景的工具选择错误",
        "test_coverage": 0.95,
        "production_performance": {"task_success": 0.93, "latency_p50": "3.2s"}
    }
}
```

**要素 2：A/B 测试框架**
```python
class PromptABTest:
    async def run_experiment(self, base_prompt, variant_prompt, traffic_split=0.2):
        # A 组（80%）：现有 Prompt（对照组）
        # B 组（20%）：新 Prompt（实验组）
        async def evaluate():
            a_metrics = await collect_metrics("A", n=1000)
            b_metrics = await collect_metrics("B", n=250)
            return compare_with_statistical_significance(a_metrics, b_metrics)

        result = await evaluate()

        if result.improvement > 0.03 and result.p_value < 0.05:
            # 显著提升 → 逐步放量 50%→100%
            await gradual_rollout(variant_prompt, steps=[50, 100], hours_between=24)
        elif result.degradation > 0.02:
            # 显著退化 → 自动回滚
            await auto_rollback(variant_prompt)
        else:
            # 无显著差异 → 保持现状或继续实验
            pass
```

**要素 3：降级可观测指标**
每条 Prompt 必须定义「Guardrails指标」：

| 指标 | 下降阈值 | 动作 |
|------|---------|------|
| 任务完成率 | 下降 > 3% | 自动回滚 |
| 工具调用准确率 | 下降 > 5% | 自动回滚 |
| 用户踩率 | 上升 > 1% | 告警 + 人工判定 |
| P99 延迟 | 上升 > 50% | 告警 |

**要素 4：Prompt Impact Analysis**
变更前自动分析「哪些场景会受影响」：
- 对每个工具调用场景计算回归测试覆盖度
- 未覆盖的场景 → 自动补充测试用例
- 高风险变更 → 需要 Senior 工程师签字

**面试加分：** 「Prompt 变更和代码变更的不同在于 **非确定性** ——同样的 Prompt 今天跑 92%，明天可能变 88%（因为上游模型更新）。所以 Prompt 的 SLO 应该是**滚动平均**（7 天窗口）而不是单次固定值。另一个实战技巧：**Prompt 的回滚比代码回滚难**——因为模型缓存了用户的旧会话。当 Prompt 回滚后，正在进行的会话可能继续使用新 Prompt。解决方案是：回滚时标记所有活跃会话使用旧 Prompt，新会话使用回滚后的版本。」

### Q9. 在设计一个新的 Agent 系统时，如何解决「冷启动（Cold Start）」问题——没有历史数据、没有用户反馈、没有标注数据的情况下，如何确定初始 System Prompt 和评估基线？

**类型：** 场景应用
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、所有从零搭建 Agent 的团队
**答：**

冷启动是 2026 年 Agent 系统落地面临的最大工程挑战之一——没有数据就没办法评估，没有评估就不知道 Prompt 好不好，没有好 Prompt 就收集不到好数据。这是一个经典的「鸡和蛋」问题。

**冷启动五步法：**

**Step 1：种子数据生成（Seed Data Synthesis）**
```python
class SeedDataGenerator:
    async def generate(self, domain: str, n=100) -> list[TestCase]:
        # 用 LLM 基于领域知识生成模拟用户查询
        test_cases = await llm.generate(
            f"""你是一个{domain}领域的用户。请生成{n}个真实的用户查询，
覆盖：简单查询、复杂查询、边缘情况、恶意输入。
格式：{{"query": "...", "intent": "...", "difficulty": "easy/medium/hard"}}"""
        )
        return test_cases
```
- 让 LLM 扮演用户生成 100-200 个模拟查询
- 覆盖正常场景 + 边界场景 + 少量对抗场景

**Step 2：黄金答案准备（Golden Answer Preparation）**
- 人工标注 30-50 个核心用例的预期输出和轨迹
- 其余用例用 LLM-as-Judge 自动生成评估标准
- 这个阶段的「粗糙」是可以接受的——目标是建立基线

**Step 3：Baseline Prompt 的「三明治设计」**
```
第一层：角色定义（你是谁）
第二层：任务规则（你做什么，不做什么）
第三层：边界条件（出错怎么办）
--- 用户内容分隔线 ---
第四层：输出格式要求（怎么回答）
```
这个结构在多种场景下经过验证，作为冷启动的默认模板。

**Step 4：合成用户模拟测试（Synthetic User Simulation）**
```python
async def simulate_conversations(prompt, test_cases, simulator_llm):
    for case in test_cases:
        agent_response = await agent.run(prompt, case["query"])
        # 用另一个 LLM 模拟用户对 Agent 回复的评价
        evaluation = await simulator_llm.judge(
            user_query=case["query"],
            agent_response=agent_response,
            expected_intent=case["intent"]
        )
        store_evaluation(case, agent_response, evaluation)
```

**Step 5：渐进式迭代（Progressive Iteration）**
- 上线第一天：100% 人工审核输出，收集初始反馈
- 上线第一周：降为 50% 抽样审核，确认数据分布
- 两周后：建立初步的自动评估 Pipeline
- 一个月后：形成稳定的反馈闭环

**面试加分：** 「冷启动最大的误区是追求完美——很多团队花了两个月做人工标注，结果上线后发现用户的真实查询分布和假设完全不同。正确做法是 **两周内上线 MVP**，用真实数据修正假设。另一个关键：合成数据虽然不够真实，但能发现 80% 的 Prompt 缺陷（如工具选择错误、格式不对），对冷启动阶段已经足够。真正的『冷启动』其实不是问题——**问题是在冷启动阶段过于追求完美**。」

### Q10. Agents-as-a-Service（AaaS）架构如何设计？如果把 Agent 能力作为服务提供给内部多个业务线，需要考虑哪些关键架构决策？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、微软、所有平台型 Agent 团队
**答：**

2026 年，大型企业开始将 Agent 能力标准化为平台服务，这是继 SaaS/PaaS 之后的「AaaS」（Agents-as-a-Service）趋势。Novelvista 和 LangChain 都提到了 Agent 平台化的架构挑战。

**AaaS 平台的核心架构组件：**

```python
class AaaSPlatform:
    def __init__(self):
        self.agent_registry = AgentRegistry()       # Agent 目录
        self.capability_discovery = Discovery()     # 能力发现
        self.service_mesh = AgentServiceMesh()       # 服务网格
        self.unified_logging = UnifiedLogger()       # 统一日志
        self.cost_manager = CrossTenantCostMgr()    # 跨租户计费
```

**关键架构决策：**

**决策 1：多租户隔离（Multi-tenancy Isolation）**
- **数据隔离：** 每个业务线的 RAG 数据库、Agent 记忆完全隔离
- **模型隔离：** 支持每个业务线配置不同的模型提供商和版本
- **配置隔离：** System Prompt、工具列表、安全策略按租户管理
- **性能隔离：** 一个业务线的突发流量不影响其他业务线

**决策 2：Capability Marketplace（能力市场）**
```yaml
# 业务线 A 开发了一个 Agent 能力，可以发布到平台
capability:
  name: "inventory-check"
  owner: "business-a"
  description: "查询库存量和预计到货时间"
  api: "https://aas-platform/agents/inventory-check"
  pricing: "$0.001/call"
  sla: "p99 < 500ms"
  available_to: ["business-b", "business-c"]
```
其他业务线可以在能力市场上发现和订阅。

**决策 3：统一可观测性与成本分摊**
- 每个 Agent 调用的成本按「租户 + Agent + 工具」三个维度归因
- 月度账单自动生成（LLM API 成本 + 向量存储 + 基础设施）
- 支持预算告警和超额熔断

**决策 4：版本化与灰度发布**
- 每个 Agent 可以保持多个版本同时运行
- 业务线可以按「5% 流量走 v2，95% 走 v1」逐步升级
- 支持按租户维度做 A/B 测试

**面试加分：** 「AaaS 架构最容易被忽略的是 **注册表竞争（Registry Rivalry）**问题——多个 Agent 提供相似能力（都是『查订单』但来自不同业务线），Orchestrator 在路由时需要选择『最合适的那个』。解决方案是：Orchestrator 不仅要看功能匹配度，还要考虑：1) 数据权威性（订单数据来自哪个系统更准确） 2) 成本优先级（内部 Agent 优先于付费 Agent） 3) 历史成功率加权。另外，AaaS 平台的制胜关键是 **Agent Composition API**——让业务线不需要编程就能『组合』多个现有 Agent 能力形成新的端到端工作流，类似 Zapier for Agents。」
