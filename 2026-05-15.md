### Q1. 什么是 Adaptive RAG？它与 Corrective RAG（CRAG）和 Self-RAG 有什么区别？请从 Agent 的「元认知」能力角度来解释。

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐⭐ | 微软、字节跳动、DataCamp 生态公司、所有 RAG 进阶团队
**答：**

DataCamp 2026 面试指南列举了三种重要的 RAG 演进变体，它们在 Agent 的「元认知」层级上逐个递进。

**Adaptive RAG —— 何时检索？**
Agent 根据查询复杂度动态决定检索策略：
```python
class AdaptiveRAG:
    def retrieve(self, query: str):
        complexity = self.estimate_complexity(query)
        if complexity < LOW:       # 简单事实查询
            return None            # 不检索，直接生成
        elif complexity < MEDIUM:  # 中等复杂
            return self.single_shot_retrieve(query)
        else:                      # 高复杂度
            return self.iterative_retrieve(query)
```
- **核心能力：** Agent 的「self-aware」——知道自己是否需要外部知识
- **实现方式：** 用一个小模型（classifier）或 LLM 自身判断查询复杂度

**Corrective RAG（CRAG）—— 检索结果是否可信？**
检索后 Agent 对文档进行相关性检查：
```
Retrieve → Check Relevance
  ├── Relevant → Feed to Generator
  ├── Irrelevant → Retry with refined query OR fall back to web search
  └── Ambiguous → Retrieve more documents for cross-validation
```
- **核心能力：** Agent 的「self-correction」——检测检索噪音并纠正

**Self-RAG —— 生成结果是否正确？**
Agent 在生成后评估自身输出的质量：
```
Generate → Self-Evaluate
  ├── Accurate → Output
  ├── Partial → Refine
  └── Likely Wrong → Re-retrieve and regenerate
```
- **核心能力：** Agent 的「self-reflection」——评估输出是否与检索内容一致

**面试加分：** 「三者的关系是递进的元认知层级：Adaptive = 知道『要不要查』，CRAG = 知道『查得好不好』，Self-RAG = 知道『答得好不好』。生产中最好组合使用：Adaptive 决定是否检索，CRAG 保证检索质量，Self-RAG 保证输出质量。但要注意——元认知本身也有成本，每增加一层就多一次 LLM 调用。实际产品中可设置『只在高置信度需求场景启用全部三层』，普通场景仅用 Adaptive RAG。」

### Q2. 什么是 Late Chunking（延迟分块）？它与传统分块方法的核心区别是什么？为什么它更适合现代 RAG Agent？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 微软、DataCamp 生态公司、所有 RAG 技术团队
**答：**

DataCamp 2026 面试指南详细介绍了 Late Chunking，这是 Günther et al. (2024) 提出的一种改进方法。

**传统 Chunking 的问题——「孤立嵌入」：**
```
Document → Split → Chunk A, Chunk B, Chunk C
                      ↓          ↓          ↓
                 embed(A)    embed(B)    embed(C)
```
每个 Chunk 独立嵌入，彼此不知道其他 Chunk 的上下文。结果是「代词引用断裂」——Chunk B 中的「its price」无法知道「its」指的是什么。

**Late Chunking 的核心——「先全文编码，再分块池化」：**
```
Document → Transformer Encoder (全文处理)
                      ↓
          [tok1, tok2, ..., tokN]  (含全文上下文的 token 向量)
                      ↓
          Split into chunks → Pool within each chunk
                      ↓
          Chunk_embed(A)  Chunk_embed(B)  Chunk_embed(C)
           (含上下文)       (含上下文)       (含上下文)
```
每个 Chunk 的嵌入都「知道」全文信息，因为 Transformer 的 Attention 层已经处理了所有 token 之间的交互。

**对 Agent 的具体价值：**
- **跨段引用**：Chunk B 中「该产品于 2024 年发布」能正确引用 Chunk A 中的产品名
- **Agent 检索更准确**：Agent 检索到的 Chunk 嵌入已包含上下文，减少后续需要再次检索的几率
- **端到端一致性**：查询「总结本文第三方研究结论」能正确找到分布在全文各处的相关内容

**面试加分：** 「Late Chunking 的代价是计算量——先全文编码再分块池化比直接分块嵌入多约 20-40% 的计算量。面试中要说明『仅对长文档使用 Late Chunking』的取舍：短文本（<512 tokens）直接用传统方法，长文本才升级到 Late Chunking。另外 Late Chunking 与 Agent 的 Context Window 管理天然契合——Agent 在做摘要任务时需要理解整个文档的全局上下文才做分块摘要。」

### Q3. 什么是 Agent Deadlock（死锁）？在 Multi-Agent 系统中，死锁是如何产生的？列举三种检测和预防策略。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、novelvista 生态公司
**答：**

novelvista 2026 面试指南定义了 Agent Deadlock——当多个 Agent 互相等待对方完成某个动作或释放资源，导致系统无法继续推进的状态。

**死锁的四种典型场景：**

1. **循环等待（Circular Wait）：** Agent A 等待 Agent B 的输出，Agent B 等待 Agent C 的输出，Agent C 等待 Agent A 的输出
2. **资源争抢（Resource Contention）：** 两个 Agent 都需要调用同一个外部 API，但 API 有并发限制，互相阻塞
3. **确认循环（Confirmation Loop）：** Agent A 说「我不确定，请 B 确认」，B 说「我无法确认，转 A 再确认」
4. **隐含依赖（Implicit Dependency）：** Agent A 修改了数据库状态，Agent B 在没有感知到状态变化的情况下继续使用旧数据

**检测策略：**

```python
class DeadlockDetector:
    def __init__(self):
        self.wait_for_graph = {}  # 有向图：Agent → 等待的 Agent

    def detect(self, timeout: int = 30) -> bool:
        # 策略 1: 循环依赖检测（DFS 环检测）
        if self.has_cycle(self.wait_for_graph):
            return True

        # 策略 2: 超时检测
        for agent, waited_since in self.wait_start_times.items():
            if time.now() - waited_since > timeout:
                return True

        # 策略 3: 状态停滞检测
        if self.no_progress_global(window=10):
            return True

        return False
```

**预防策略：**
1. **超时机制**：每个 Agent 调用设置明确超时，超时后降级/跳过
2. **任务所有权**：每项任务有且仅有一个 Owner，避免两个 Agent 认为「对方会做」
3. **中心化协调器**：所有 Agent 通信经由 Orchestrator，Orchestrator 维护全局 DAG 并检测环

**面试加分：** 「最隐蔽的死锁是『逻辑死锁』——技术上没有环，但两个 Agent 的决策相互抵消（Agent A 添加了一个筛选条件，Agent B 在数据处理时去掉该条件）。这种死锁的检测需要基于语义分析而非拓扑分析。推荐方案：在 Agent 加入任务时注册『输入假设』和『输出承诺』，协调器交叉检查是否存在『A 的输入假设被 B 的输出破坏』的情况。」

### Q4. 如何衡量 Agent 系统的 ROI？从「Token 成本管理」「小模型路由」「结果缓存」三个维度设计成本优化方案。

**类型：** 产品思维
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、novelvista 生态公司、所有 PM 面试
**答：**

novelvista 2026 指南指出：Agent 系统的价值不再是提不提效的问题，而是「多快能收回 Agent 的建设成本」。

**ROI 衡量公式：**
```
Agent ROI = (人工节省成本 + 质量提升收益 - Agent 运行成本) / Agent 建设成本
```
- **人工节省成本** = 原人工处理时间 × 时薪 × 替代比例
- **质量提升收益** = 错误率降低造成的损失减少 + 响应速度提升带来的客户留存
- **Agent 运行成本** = LLM API 费用 + 向量检索成本 + 基础设施费用

**成本优化三维方案：**

**维度 1：Token 成本管理**
```python
class TokenBudgetManager:
    def __init__(self, budgets: dict):
        self.budgets = budgets  # per-task token budgets

    def should_continue(self, task_id: str, tokens_used: int) -> bool:
        if tokens_used > self.budgets[task_id] * 1.5:
            return False  # 触发熔断
        # 每多消耗 budget 的 20%，确认一次
        if tokens_used % (self.budgets[task_id] // 5) == 0:
            return self.confirm_progress(task_id)
        return True
```
- 设置每任务的 Token 预算，超预算自动熔断或降级
- 监控 Token 浪费模式（如 Agent 反复检索同一文档）

**维度 2：小模型路由（Model Routing）**
```
查询复杂度分类:
  ├── 简单: gpt-4o-mini (成本降低 95%)
  ├── 中等: claude-sonnet-4 (成本降低 60%)
  └── 困难: claude-opus-4 / gpt-4o (原始成本)
```
- CallSphere 和 novelvista 都提到：简单子任务路由给小模型
- 对 Agent 的规划层用大模型，执行层用小模型

**维度 3：结果缓存（Caching）**
- **精确缓存**：完全相同的查询命中缓存，零成本
- **语义缓存**：相似查询（余弦相似度 > 0.95）返回缓存结果
- **结果复用**：Agent 的常见子任务结果可跨会话复用
- DataCamp 的 CAG 概念：缓存过的摘要结果可降低 60-70% Token 开销

**面试加分：** 「最容易被忽略的是『决策成本』——Agent 花大量 Token 和延迟在做『是否要检索』『是否要终止』的决策。一个聪明的做法是：**在 Agent 层用规则取代 LLM 决策**。例如『检索必要性判断』用一个几十行代码的规则引擎代替 LLM 调用，能节省约 30% 的总运行成本。另一个重点：Agent 的成本优化不能只看单次调用，要看完整任务链——有时多花 2 次 LLM 调用避免一次错误的 API 调用反而更省钱。」

### Q5. CallSphere 将 Agent 设计模式总结为 ReAct、Plan-and-Execute 和 Multi-Agent 三种。请深度对比这三种模式，并给出生产环境中的「模式组合」方案。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、微软、字节跳动
**答：**

CallSphere 2026 面试指南明确指出：这三种模式不是互斥的——生产系统中它们通常**组合使用**。

**三种模式深度对比：**

| 维度 | ReAct | Plan-and-Execute | Multi-Agent |
|------|-------|-------------------|-------------|
| **核心机制** | 推理与行动交错 | 先规划后执行，失败重规划 | 多领域专家协作 |
| **最佳步骤数** | 1-5 步（简单顺序任务） | 5-20 步（多步骤工作流） | 复杂跨域任务 |
| **失败模式** | 复杂任务迷失、循环 | 规划可能过时，执行与规划脱节 | 协调开销、错误传播 |
| **模型调用次数** | 每步 1 次 | 规划 1 次 + 执行每步 1 次 | N×Agent 调用数 |
| **上下文管理** | 简单（上下文积累） | 规划与执行分离 | 每个 Agent 独立上下文 |
| **调试难度** | 容易（线性轨迹） | 中等（计划 vs 实际偏差） | 困难（跨 Agent 追踪） |

**生产中的组合方案——分层架构：**

```
用户请求
    │
    ▼
[Orchestrator Agent]  ← Plan-and-Execute（规划层：大模型）
    │  ┌──────────────────────────────────────┐
    │  │ 1. 分析请求复杂度                    │
    │  │ 2. 分解为 DAG 依赖图                │
    │  │ 3. 为每个子任务指派 Specialist      │
    │  └──────────────────────────────────────┘
    │
    ├── [Research Agent]  ← ReAct（检索层：中模型）
    │       └── Thought→Search→Observe→Thought...
    │
    ├── [Code Agent]  ← ReAct（执行层：大模型）
    │       └── Thought→Write→Test→Observe→Thought...
    │
    └── [Data Agent]  ← ReAct（分析层：小模型）
            └── Thought→Query→Analyze→Thought...
```

**面试加分：** 「CallSphere 最有价值的 insights：1) **Model-native tool use** 趋势——2026 年无需显式 ReAct Prompt 工程，Claude 和 GPT-4 的原生工具调用能力比 Prompt 层面的 ReAct 更稳健。2) 三种模式的本质区别在于**控制权转移的粒度**——ReAct 每步都转，Plan-and-Execute 只在失败时转，Multi-Agent 通过消息转。3) 生产系统应遵循『从简到繁』原则：先从 ReAct 开始，当任务复杂到超出其能力范围时升级到 Plan-and-Execute，当需要领域专家时升级到 Multi-Agent 模式。」

### Q6. 什么是 Model Routing（模型路由）？在 Multi-Agent 系统中，如何为不同子任务选择最合适的模型以平衡成本与质量？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | Anthropic、OpenAI、字节跳动、CallSphere 生态公司
**答：**

CallSphere 2026 指南特别强调：Multi-Agent 系统可以是单 Agent 系统成本的 5-10 倍——Model Routing 是控制成本的关键技术。

**Model Routing 的核心原则：**
将简单推理任务分配给小/快/便宜的模型，将复杂推理任务分配给大/慢/贵/能力强的模型。

**三级路由架构：**
```python
class ModelRouter:
    MODELS = {
        "fast": "claude-haiku-4",     # $0.25/M tokens, 用于简单任务
        "balanced": "claude-sonnet-4",  # $3/M tokens, 用于中等任务
        "powerful": "claude-opus-4",    # $15/M tokens, 用于复杂任务
    }

    def route(self, task: Task) -> str:
        if task.complexity == "simple":
            return self.MODELS["fast"]
        elif task.complexity == "medium":
            return self.MODELS["balanced"]
        else:
            return self.MODELS["powerful"]
```

**复杂度判断策略：**

1. **启发式规则**：任务类型 → 模型选择
   - 文本分类 → 小模型
   - 情感分析 → 小模型
   - 多步推理 → 大模型
   - 代码生成 → 大模型

2. **LLM Classifier**：用一个快模型评估任务的「复杂度分数」
   ```python
   def estimate_complexity(task_description: str) -> float:
       # 用 Haiku/Sonnet 快速分类
       response = claude_haiku.estimate(
           f"请评估此任务需要{0简单～1困难}: {task_description}"
       )
       return float(response.score)
   ```
   
3. **自适应回退**：先尝试小模型，如果结果置信度低于阈值，升级到大模型重新执行
   ```python
   def execute_with_fallback(task: Task):
       result = execute_with_model(task, "sonnet")
       if result.confidence < 0.6:
           result = execute_with_model(task, "opus")
       return result
   ```

**面试加分：** 「最巧妙的技巧是『分阶段路由』——同一个任务的不同阶段用不同的模型。比如 Plan 阶段用 Opus（规划需要全局推理），但 Execute 阶段的每个子任务可以根据内容选择 Haiku（简单子任务）或 Sonnet（复杂子任务）。另一关键点：**Model Routing 也需要监控和评估**——如果某类简单任务反复触发fallback到大模型，说明复杂度判断不准，需要调整路由规则。」

### Q7. 设计一个健壮的 Tool Calling 错误恢复系统。当 API 返回 500、限流、参数错误等情况时，LLM 应如何自适应恢复？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | Anthropic、OpenAI、所有 Agent 生产团队
**答：**

CallSphere 2026 面试指南提供了一个完整的错误恢复模式，强调「让 LLM 根据错误信息自适应」。

**分层错误恢复架构：**

**Layer 1: 内置重试（自动）**
```python
async def execute_with_retry(tool_name, params, max_retries=3):
    for attempt in range(max_retries):
        try:
            return await call_tool(tool_name, params)
        except RateLimitError:
            await asyncio.sleep(2 ** attempt)  # 指数退避
        except TimeoutError:
            continue  # 直接重试
        except Exception as e:
            # 非重试可恢复的错误，立即转为 Layer 2
            return handle_error(tool_name, params, e)
```

**Layer 2: 自定义错误恢复（LLM 参与）**
关键设计——让错误信息对 LLM 可消费：
```python
# 错误信息不是纯技术文本，而是结构化的、包含恢复建议
error_response = {
    "status": "error",
    "error": "数据库连接超时",
    "recovery_hint": "数据库暂时不可用。请尝试使用缓存数据工具，或告知用户稍后重试。",
    "alternative_tools": [
        "get_cached_result(query, ttl=3600)",
        "search_local_index(keyword)"
    ]
}

# 将此错误+恢复建议传给 LLM
messages.append({
    "role": "tool",
    "content": json.dumps(error_response)
})
# LLM 现在可以根据恢复建议自适应地选择下一步
```

**Layer 3: 参数校验与 Schema 一致性**
```python
class ToolValidator:
    def validate(self, tool_name, params):
        schema = self.tool_schemas[tool_name]
        errors = []

        for field, rules in schema.items():
            if field not in params and rules.get("required"):
                errors.append(f"缺失必填参数: {field}")
            elif field in params and not self.match_type(params[field], rules):
                errors.append(f"参数 {field} 类型错误，期望 {rules['type']}")
        
        if errors:
            return {
                "status": "error",
                "error": "; ".join(errors),
                "fixes": self.suggest_fixes(tool_name, params, errors)
            }
        return None
```

**避免重复执行（幂等性）：**
```python
# 给每个工具调用分配唯一 ID
tool_call_id = f"{task_id}:{tool_name}:{hash(params)}"
if idempotency_cache.exists(tool_call_id):
    return idempotency_cache.get(tool_call_id)
```

**面试加分：** 「生产中最常见的坑是『假成功』——工具返回 200 但数据是错的（比如数据库返回了旧版本数据）。因此不能只看 HTTP Status Code，还要做**结果语义验证**：用一个小模型快速检查结果的合理性。另一个关键实践：**工具调用模式监控**——如果 LLM 对同一个工具用完全相同参数反复调用，说明 Agent 卡住了，需要强制中断并在新轮次中注入『你似乎在重复同一个动作，请换一种方法』的提示。CallSphere 强调：monitor tool call patterns — if the agent is calling the same tool in a loop with the same parameters, it's stuck。」

### Q8. 如何为 Agent 构建 Continuous Evaluation CI/CD 体系？Commit 触发、Schedule 触发和 Event 触发三种模式分别适用于什么场景？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | Galileo 生态公司、字节跳动、蚂蚁集团、所有生产 Agent 团队
**答：**

### Q9. 如何为实时系统设计 Agent（Real-Time Agent）？当延迟要求 < 500ms 时，传统 Agent 的哪些设计模式不再适用？

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、novelvista 生态公司
**答：**

novelvista 2026 指南提出了实时 Agent 的设计挑战：传统 Agent 的「规划-推理-执行」循环在实时场景中会超出延迟预算。

**传统 Agent 在实时场景的三大瓶颈：**

1. **串行 ReAct 循环延迟累积**：每步 Thought→Action→Observe 都是 1-3 次 LLM 调用，5 步就是 5-15 次 → 无可接受
2. **全量 Context 加载**：每次 LLM 调用都加载全部上下文 → Token 处理时间随上下文线性增长
3. **外部 API 同步调用**：等待外部服务返回结果会阻塞整个控制流

**实时 Agent 的改造方案：**

**方案 1：预规划 + 流式执行**
```python
class RealTimeAgent:
    async def process(self, query: str):
        # 离线阶段：预规划可能的路径
        plan = await self.precompute_plan(query)  # 延迟预算: 2s (允许)
        
        # 在线阶段：流式执行
        async for action in self.stream_execute(plan):  # < 500ms 首次响应
            yield action  # 边执行边输出
```
- 关键：将「规划」和「执行」解耦，规划允许较长延迟，执行必须低延迟

**方案 2：Speculative Execution（推测执行）**
```python
class SpeculativeAgent:
    async def process(self, query: str):
        # Agent 预测多个可能的下一步并并行执行
        candidates = self.speculate_next_steps(query, n=3)
        results = await asyncio.gather(*[
            self.execute(c) for c in candidates
        ])
        # 根据实际需要选择最匹配的结果
        return self.choose_best_match(results, query)
```
- 代价：多消耗资源但降低延迟
- 典型应用：语音助手场景（需要 < 300ms 响应）
- 适合：并行执行多个可能性，丢弃不需要的

**方案 3：模型级优化**
- 使用专门的小模型做实时推理（如 streaming mode）
- 使用 speculative decoding 降低推理延迟
- 缓存常用推理路径（如常见意图的回复模板）

**面试加分：** 「实时 Agent 最关键的设计理念是『**提前做能做的事情**』——把『规划』和『知识检索』移到用户请求之前。例如在对话 Agent 场景中，用户还没说完前就开始推测意图并预加载相关数据。另一个关键点：**Response Priority**——Agent 先输出一个『快速占位响应』告知用户『正在处理』，然后继续完整推理。这种做法在语音 Agent 中非常常见，用户感知延迟从 3s 降到 300ms。」

### Q10. 什么是 Agent 的 Explainability（可解释性）？如何在保持 Agent 性能的前提下实现「可解释的推理轨迹」？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 微软、蚂蚁集团、金融合规场景、novelvista 生态公司
**答：**

novelvista 2026 指南指出：可解释性在金融、医疗、法律等合规性强的场景中是「硬需求」——监管机构不能接受一个「黑箱决策」。

**Explainability 的三个层次：**

```
Level 1：Actions Log → 「Agent 做了什么」
  ├── 记录所有工具调用、输入输出
  ├── 适用：简单审计需求
  └── 局限：无法解释「为什么」

Level 2：Reasoning Trail → 「Agent 为什么这么想」
  ├── 保留 Agent 在每个决策点的推理过程
  ├── 适用：调试和性能优化
  └── 局限：可能包含敏感信息或噪声

Level 3：Human-Readable Summaries → 「用人类能理解的语言解释」
  ├── Agent 自动生成决策摘要
  ├── 适用：合规报告、客户沟通
  └── 局限：摘要可能丢失细节或引入偏差
```

**实现可解释性而不牺牲性能：**

```python
class ExplainableAgent:
    def __init__(self):
        self.reasoning_log = []      # 在线记录
        self.decision_attributions = {}  # 决策归因

    async def decide(self, input_data) -> tuple[Action, Explanation]:
        # Step 1: 内部推理（不额外调用 LLM）
        reasoning = self.generate_reasoning(input_data)
        
        # Step 2: 基于推理选择行动
        action = self.choose_action(reasoning)
        
        # Step 3: 轻量级归因记录（非 LLM 调用，仅元数据操作）
        attribution = {
            "input_summary": input_data[:200],
            "key_factors": self.extract_key_factors(input_data, reasoning),
            "alternative_actions_considered": self.alternatives,
            "decision_rationale": reasoning[:500],
            "retrieved_documents": self.used_documents,
            "confidence": self.current_confidence,
        }
        self.reasoning_log.append(attribution)
        
        # Step 4: 需要时生成人类可读摘要（额外一次 LLM 调用，可控）
        if self.needs_human_readable:
            explanation = await self.generate_explanation(attribution)
        else:
            explanation = attribution  # 原始归因即可

        return action, explanation

    def extract_key_factors(self, input_data, reasoning):
        # 不调用 LLM，用规则从推理文本中提取关键因素
        # 如检测 "因为" "基于" "考虑到" 等关键词后的内容
        return [factor for factor in self.reasoning_patterns.findall(reasoning)]
```

**关键性能优化：**
- **不做额外 LLM 调用做解释**：大多数解释（Level 1-2）可以从已有数据中提取
- **按需解释**：只在需要时才生成 Level 3 的人类可读摘要
- **预定义模板**：常见决策类型使用解释模板填充变量，避免每次生成
- **异步归因**：解释生成可以异步进行，不阻塞主推理流程

**面试加分：** 「金融合规场景最关键的概念是『**负向归因（Negative Attribution）**』——不仅解释 Agent 做了什么，还要解释『什么因素导致 Agent 没有选择其他行动』。比如贷款审批 Agent 不仅要解释『批准了贷款是因为收入达标』，还要解释『拒绝其他方案是因为负债率超过阈值』。另一个进阶话题：**Counterfactual Explanation**——『如果将用户的年收入从 35K 改为 45K，结果会变成批准』，这在欧盟 AI Act 下可能是强需求。」
