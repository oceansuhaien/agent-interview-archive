### Q1. RAG 系统的检索召回率（Recall）上不去，你会从哪些维度排查和优化？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、微软、腾讯、所有搜索+AI 公司
**答：**

RAG 检索召回率是 2026 年面试中几乎必问的工程题。DataCamp 的 30 道 RAG 题中，关于「检索失败原因分析」类题目占了近 1/3。

**排查四维度：**

**1. 分块策略（Chunking）—— 最常见的原因（~50%）**
- 块太小 → 每个块缺乏完整语义，关键信息被截断
- 块太大 → 块内噪声过多，LLM 注意力被稀释
- **优化：** 采用「语义分块」而非固定 token 长度——用 NLP 检测段落/主题边界；或使用「递归分块」：先按段落拆，段落过长再按句拆

**2. Embedding 模型 —— 次常见（~25%）**
- 通用 Embedding（如 text-embedding-ada-002）在垂直领域效果差
- **优化：** 用领域微调的 Embedding 模型（如 BGE-Large、E5-Mistral）；或在同一领域做领域适配

**3. 检索策略单一 —— 隐蔽（~15%）**
- 只用向量相似度搜索，忽略 BM25 关键词匹配的价值
- **优化：** 混合检索（Hybrid Search）：向量检索 + BM25 加权融合；多路召回（Multi-Route Retrieval）：同一问题同时走多个检索路径

**4. 查询质量问题 —— 隐蔽（~10%）**
- 用户输入的短查询（如「怎么部署」）包含的信息量不足以做语义匹配
- **优化：** 查询重写（Query Rewrite）：用小模型将短查询扩展为完整句；查询分解（Query Decomposition）：复杂问题拆成多个子查询分别检索

**面试加分：** 「不要只盯着检索策略调参——先去分析『没召回』的数据是『语义相近但 chunk 边界截断』还是『压根语义不匹配』。这两种情况需要完全不同的优化方向。」

### Q2. 什么是混合检索（Hybrid Search）？向量检索和关键词检索如何有效融合？

**类型：** 概念辨析
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、腾讯、所有 RAG 团队
**答：**

混合检索是 2026 年生产 RAG 系统的标配，面试官用此题考察你对「检索不是越先进越好，而是越适合越好」的理解。

**为什么需要混合检索？**

| 场景 | 向量检索 | BM25 关键词检索 |
|------|---------|----------------|
| 同义不同文（"如何购买"↔"购买流程"） | ✅ 强 | ❌ 弱 |
| 精确匹配（"Python 3.12.0"、"Bug #4521"） | ❌ 弱 | ✅ 强 |
| 跨语言匹配 | ✅ 强 | ❌ 弱 |
| 稀有术语/缩写 | ❌ 弱 | ✅ 强 |

**三种融合策略（从简单到工程化）：**

**策略 1：RRF（Reciprocal Rank Fusion）—— 推荐入门**
```python
def rrf(scores_vec, scores_bm25, k=60):
    combined = {}
    for rank, doc_id in enumerate(sorted(scores_vec, key=scores_vec.get, reverse=True)):
        combined[doc_id] = combined.get(doc_id, 0) + 1 / (k + rank + 1)
    for rank, doc_id in enumerate(sorted(scores_bm25, key=scores_bm25.get, reverse=True)):
        combined[doc_id] = combined.get(doc_id, 0) + 1 / (k + rank + 1)
    return sorted(combined, key=combined.get, reverse=True)
```
无需标准化分数，对分数分布不敏感，工程上最可靠。

**策略 2：加权融合（Weighted Sum）—— 可调性强**
- `score = α * vec_score_normalized + (1-α) * bm25_score_normalized`
- α 通过验证集搜索确定（一般 0.5-0.8）

**策略 3：Learn-to-Rank（LTR）—— 重量级方案**
- 用 XGBoost/LambdaRank 训练一个排序模型，输入为向量相似度 + BM25 分数 + 文档特征
- 适合检索 Top-1 精度要求极高的场景

**面试加分：** 「不要盲目上 LTR。对于 90% 的业务场景，RRF 已经足够好且训练成本为零。只有在排序精度对业务指标有可量化的显著影响时（如医疗搜索），才值得上 LTR。」

### Q3. 请设计一个完整的 Agent 生产评估体系——包括离线评估、在线评估和持续监控。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团、Galileo/Thinking Company 框架相关团队
**答：**

2026 年 Agent 评估已经从「单指标打分」演进为「三层评估体系」。Galileo 和 Thinking Company 的数据都一致强调这一趋势。

**第一层：离线评估（Pre-Production）**

| 评估项 | 方法 | 通过标准 |
|-------|------|---------|
| 功能正确性 | 自动化回归测试（50-200 条 Eval Cases） | > 90% |
| 抗干扰能力 | Prompt Injection 攻击 | > 85% 未被攻破 |
| 工具调用准确率 | 正确的工具+正确的参数 | > 95% |
| Token 效率 | 平均推理步数和 Token 消耗 | 建立基线 |
| 幻觉率 | 输出事实性验证 | < 5% |

**第二层：在线评估（Staged Rollout）**

- **Canary 阶段（1% 流量）：** 监控 Human Intervention Rate（人工干预率）和 Task Success Rate
- **Shadow 阶段（10% 流量）：** 与已有系统并行运行，对比输出质量
- **Production 阶段（100% 流量）：** 持续采样评估

**第三层：持续监控（Production）**

Galileo 强调的 **Outcome Metrics vs. Trajectory Metrics** 是核心：
```python
# Outcome Metrics —— 结果好不好（外部视角）
task_success_rate      # Agent 完成用户目标了吗？
user_satisfaction      # 用户评分/点赞/投诉比例
human_intervention     # 人工介入率（关键指标！）

# Trajectory Metrics —— 过程好不好（内部视角）
avg_inference_steps    # 平均推理步数
tool_call_success      # 工具调用成功率
context_window_usage   # 上下文使用率（是否接近上限）
self_correction_count  # 自我纠错频率（过高说明不稳定）
```

**关键原则：** Thinking Company 指出——「Outcome metrics tell you if your agent works; trajectory metrics tell you why. Production needs both perspectives.」

**面试加分：** 「最容易被忽视的是『Human Intervention Rate』。这不是 KPI，这是 SOS 信号——当它超过 10-15% 时，说明 Agent 需要架构级的改进，而不是 prompt 微调。」

### Q4. Anthropic 在《Building Effective Agents》中提出的 Agent 设计核心原则是什么？如何应用到实际项目中？

**类型：** 技术理解
**考察频率：** ⭐⭐⭐⭐ | 百度、微软、Anthropic 生态公司、所有 Agent 平台团队
**答：**

Anthropic 的这篇文章是 2026 年 Agent 工程领域的「圣经级参考」，面试官用此题考察你对工程哲学而非框架调用的理解。

**五个核心原则：**

**原则 1：保持简单（Keep it simple）**
> "Start with the simplest possible solution and only add complexity when you have evidence it's needed."
- 大多数场景只需要一个 Workflow（Chain/Router/Parallel），不需要 Agent
- Agent（LLM 自主决策工具调用）引入了不可预测性，只在必要时才用

**原则 2：从 Workflow 开始，只在需要时才升级到 Agent**
| 场景 | 建议模式 |
|------|---------|
| 流程固定的任务 | Prompt Chaining |
| 意图明确的分流 | Routing |
| 需要并行处理的 | Parallelization |
| 需要自主决策的 | Agent |

**原则 3：工具设计决定 Agent 质量**
- "Test how the model uses your tools: Run many example inputs to see what mistakes the model makes."
- 工具描述要写清楚（什么参数、什么时候用、返回什么）
- 返回格式要接近模型在互联网文本中见到的自然格式（JSON vs Markdown vs 自然语言选择有讲究）

**原则 4：在可控环境中充分测试**
- 「不要把生产流量当作你的测试数据」
- 建议跑 50-100 个 Edge Case 看工具调用的常见错误类型

**原则 5：衡量（Measure）而不是假设（Assume）**
- 每次修改都量化评估——不用「感觉变好了」，而是「Human Intervention Rate 从 8% 降到 4%」
- 建立 A/B 测试基础设施，支持并行版本对比

**面试加分：** 「大多数人犯的错误是过早引入 Agent 的自主性。Anthropic 的建议是：先用确定的 Workflow 跑通，再用 Agent 替代那些『确实需要灵活决策』的步骤。这不仅是 engineering 选择，也是 debugging 哲学——Workflow 好调试，Agent 难定位。」

### Q5. 在 Agent 的 Tool Calling 中，工具描述（Tool Description）如何编写才能最大化模型的准确率？

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、百度、Anthropic/LangChain 生态公司
**答：**

2026 年面试中，「工具描述是 Agent 系统的关键组成部分」已经成为共识。Anthropic 官方数据表明：工具描述的质量直接决定工具调用准确率，影响幅度可达 20-30%。

**工具编写的六个黄金法则：**

**1. 描述清晰「什么时候用」**
- ❌ `get_user_info(id)` → "获取用户信息"
- ✅ `get_user_info(id)` → "根据用户 ID 获取用户的注册信息、账户状态和会员等级。当用户问'查一下我的账户'或'看看我的会员'时使用。"

**2. 参数描述要指明格式和约束**
- ❌ `param: "用户 ID"`
- ✅ `param: "用户 8 位数字 ID，如 12345678。纯数字字符串，不需要前导零。"`

**3. 返回格式靠近自然语言**
- 模型在互联网文本上训练，对 Markdown 表格、自然语言描述的理解远好于纯 JSON 数组
- 如果工具返回的是复杂结构化数据，考虑用自然语言摘要 + 原始 JSON 的混合格式

**4. 参数校验要在工具端做，不要在描述里期望模型「记住」规则**
- 不要期待模型每次调用都传入正确的参数格式
- 工具侧做容忍度高的解析和校验

**5. 避免工具过多**
- 工具超过 15-20 个时，模型的工具选择准确率显著下降
- **策略：** 超过数量就引入 Router——先用一个「工具选择器 Agent」路由到各自的工具子集

**6. 错误信息的返回要有指导性**
- ❌ 返回 `{"error": "Invalid input"}`
- ✅ 返回 `{"error": "用户 ID 格式不正确，应传入 8 位数字字符串。你传入了: ABC123"}` —— 帮助模型自我纠正

**面试加分：** 「有一个常被忽略的 trick——在工具返回中主动给出 next step 建议。比如订单查询工具返回时附带『用户最近的 5 笔订单，如果你想查看更多历史，可以调用 get_order_history 工具』。这会让 Agent 的自主决策质量明显提升。」

### Q6. Eval-Driven Development（评估驱动开发）在 Agent 系统中如何落地？请设计一个完整的 Eval 流程。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、美团、Galileo/Thinking Company 框架团队
**答：**

Eval-Driven Development（EDD）是 2026 年 Agent 工程的最佳实践范式。Thinking Company 明确将其作为 Agent 生产化的前置条件。

**EDD 的五步闭环流程：**

```
收集真实用例 → 构建评估集 → 跑基线评估 → 迭代改进 → 回归比较
```

**步骤 1：收集真实用例**
- 从生产日志中采样真实用户请求
- 覆盖：核心功能（60%）+ 边缘情况（20%）+ 异常输入（10%）+ 安全攻击（10%）

**步骤 2：构建评估集（Eval Dataset）**
每条用例包含三个要素：
```json
{
  "query": "把这条消息发给张三",
  "expected_tools": ["search_contact", "send_message"],
  "expected_params": {"contact_name": "张三"},
  "success_criteria": "消息发送成功且收件人为张三"
}
```

**步骤 3：跑基线评估**
- 用当前 Agent 版本跑满评估集
- 记录：工具调用准确率、推理步数、Token 消耗
- **关键：** 建立「评估集不该被 Agent 渐进「记住」—— 如果 Agent 通过提示工程记住了 eval 集中的查询，指标会虚高。需要定期轮换。

**步骤 4：迭代改进**
- 分析失败用例的共同模式（常见错误类型）
- 针对性修改：工具描述、System Prompt、业务逻辑、评估标准

**步骤 5：回归比较**
- 每次修改都重跑全量评估集，对比指标
- 如果 A 指标提升了但 B 指标下降了 → 需要 trade-off 决策

**面试加分：** 「大多数人只做『正向用例』评估，但真正能让 Agent 生产环境稳定的，是『负向用例』（Negative Tests）——测试 Agent 在遇到无法完成的任务时是否会优雅地说明能力边界，而不是编造答案或尝试违规操作。」

### Q7. LangChain 最新文档中列出了 5 种多 Agent 编排模式（Subagents / Handoffs / Skills / Router / Custom Workflow），请深度对比它们的性能特征和适用场景。

**类型：** 架构设计
**考察频率：** ⭐⭐⭐⭐⭐ | 字节跳动、微软、LangChain 生态公司
**答：**

LangChain 2026 年更新的多 Agent 文档提供了详尽的性能对比数据。这是面试官考察候选人「是否跟进行业前沿」的经典题目。

**五种模式速览：**

| 模式 | 核心机制 | 适用场景 |
|------|---------|---------|
| **Subagents** | 主 Agent 将子 Agent 作为工具调用 | 需要分布式开发和强审计 |
| **Handoffs** | Agent 通过 tool call 互相移交控制权 | 需要多轮用户交互 |
| **Skills** | 单一 Agent 按需加载专用知识 | 需要简单易维护 |
| **Router** | 路由层分类后分发任务 | 需要并行执行多个子任务 |
| **Custom Workflow** | LangGraph 自定义执行流 | 需要自定义编排逻辑 |

**性能对比（LangChain 官方数据）：**

**场景 1：单次请求（"买咖啡"）**
| 模式 | 模型调用次数 | 推荐度 |
|------|------------|--------|
| Subagents | 4 | ❌ |
| Handoffs | 3 | ✅ |
| Skills | 3 | ✅ |
| Router | 3 | ✅ |

**场景 2：重复请求（"再买一次咖啡"）**
| 模式 | 第二次调用次数 | 总次数 | 推荐度 |
|------|-------------|-------|--------|
| Subagents | 4 | 8 | ❌ |
| Handoffs | 2 | 5 | ✅ |
| Skills | 2 | 5 | ✅ |
| Router | 3 | 6 | ❌ |

**场景 3：多领域（比较 Python/JS/Rust 的 Web 开发能力，每领域 ~2000 tokens）**
| 模式 | 模型调用 | 总 Token | 推荐度 |
|------|---------|---------|--------|
| Subagents | 5 | ~9K | ✅ |
| Handoffs | 7+ | ~14K+ | ❌ |
| Skills | 3 | ~15K | ❌ |
| Router | 5 | ~9K | ✅ |

**关键洞察：**
- **Subagents 是「最强隔离」的代价**：每次独立调用但有集中控制，适合大规模团队协作
- **Handoffs / Skills 利用状态粘性**：重复请求下省 40-50% 的模型调用
- **Router 和 Subagents 在并行场景最优**：可以用并行工具调用同时处理多个子任务
- **Handoffs 在多领域并行处理时最差**：顺序执行无法利用并行能力

**选型决策：**
- 单次请求多 → Handoffs / Skills / Router
- 重复请求多 → Handoffs / Skills
- 并行执行需求 → Subagents / Router
- 大规模上下文隔离 → Subagents / Router
- 简单聚焦任务 → Skills

### Q8. 在 Agent 系统中，如何防止「工具过度使用」或「工具误调用」？请设计防御策略。

**类型：** 工程实践
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、蚂蚁集团、所有有工具调用场景的公司
**答：**

工具误调用是 Agent 生产环境中最常见的问题之一。coprep 在面试题中明确指出了「Tool Failure — API is down or returns a 500 error」的场景，Anthropic 也强调「测试工具调用中的常见错误」。

**四层防御体系：**

**第一层：工具定义层的防护（Design-time）**
- 每个工具都要有清晰的「能力边界」描述
- 易混淆的工具加「排除条件」：`"此工具不适用于 XX 场景"`
- 高权限工具加警告提示：`"注意：此操作不可撤销，调用前请确认"`

**第二层：预调用校验层（Pre-call）**
```python
class ToolValidator:
    def validate(self, tool_name: str, params: dict) -> ValidationResult:
        # 参数格式校验
        if not check_types(params, TOOL_SCHEMA[tool_name]):
            return Result.REJECT

        # 风险等级检查
        if tool_name in HIGH_RISK_TOOLS:
            return Result.REQUIRE_CONFIRMATION

        # 调用链检测（防止循环调用）
        if detects_cycle(tool_name):
            return Result.REJECT

        return Result.ALLOW
```

**第三层：错误恢复层（Runtime）**
工具调用失败时的优雅处理：
```python
class ToolCallHandler:
    def execute_with_fallback(self, tool_name, params, max_retries=2):
        for attempt in range(max_retries):
            result = self.call_tool(tool_name, params)
            if result.success:
                return result
            # 记录错误供 Agent 推理
            result.retry_hint = self.generate_hint(tool_name, result.error)
        # 全部重试失败后返回「降级」响应
        return FallbackResponse(
            original_request=params,
            failure_reason=result.error,
            suggestion="可以尝试手动操作或联系人工客服"
        )
```

**第四层：审计与告警层（Monitoring）**
- 统计「非预期工具调用」的占比
- 设置「单次请求工具调用上限」（例如同类型工具最多调用 3 次）
- 跟踪「工具调用失败率」的趋势

**面试加分：** 「最容易被忽略的是『无意义重复调用』——Agent 第一次调用失败了，然后换了一个几乎等价的参数又调用一次。这类行为在生产日志中很难被注意到，但会浪费大量 Token 和 API 费用。建议设置『相同工具短时间内重复调用的频率限制』。」

### Q9. 在 RAG 系统中，Embedding 模型选择的核心考量是什么？dense 和 sparse embedding 在 Agent 场景下如何选择？

**类型：** 技术理解
**考察频率：** ⭐⭐⭐⭐ | 字节跳动、微软、腾讯、所有搜索+AI 公司
**答：**

Embedding 模型选择是 DataCamp RAG 题集中的高频考点。2026 年 Agent 系统中，Embedding 不仅仅是检索工具——它还在记忆系统、工具路由等场景中发挥作用。

**五大核心考量维度：**

| 维度 | 影响 | 推荐度优先 |
|------|------|-----------|
| **语义理解能力** | 能否处理同义词、抽象概念 | ✅ 最重要 |
| **维度大小** | 检索速度和存储成本 | 768 维是平衡点 |
| **领域适应性** | 通用 vs 垂直领域（医疗/代码/金融） | 领域优先 |
| **上下文长度** | 能否编码长文本片段 | 至少 512 tokens |
| **多语言支持** | 是否需要跨语言检索 | 按业务需求 |

**Dense vs Sparse Embedding：**

| 对比维度 | Dense Embedding | Sparse Embedding |
|---------|----------------|-----------------|
| 原理 | 神经网络学习密集向量 | 统计特征（TF-IDF/BM25）并保留原始 token 信号 |
| 语义匹配 | ✅ 强（理解同义词） | ❌ 弱（依赖精确匹配） |
| 精确匹配 | ❌ 弱（容易丢失稀有术语） | ✅ 强（保留精确 token 信息） |
| 存储 | 高（每文档一个向量） | 低（稀疏表示） |
| OOV（越界词） | ❌ 有（预训练词表有限） | ✅ 无（任何 token 都可表示） |
| 典型模型 | text-embedding-3-large, BGE | SPLADE, uniCOIL |

**Agent 场景下的选型建议：**

- **RAG 检索：** 用 Dense Embedding（BGE-M3 / E5-Mistral）做主要检索 + Hybrid 搭配 BM25
- **记忆检索：** 用 Dense Embedding，因为记忆通常是语义相似的（「之前聊过类似的订单问题」）
- **工具路由/分类：** 小模型的 Sparse Embedding 或简单 TF-IDF 即可——因为工具分类的候选集很小（10-20 个）
- **跨语言场景：** 优先考虑多语言模型如 BGE-M3 或 multilingual-e5-large

**面试加分：** 「一个常见的误解是『Embedding 模型越新越好』。实际上在垂直领域（如医疗、法律），微调过的旧版 BGE 往往比最新的通用模型效果更好。关键是评估集要反映真实场景的检索分布。」

### Q10. 你的 Agent 有机会接触到用户敏感数据（PII），请设计一个最小权限原则（Least Privilege）下的数据访问策略。

**类型：** 安全设计
**考察频率：** ⭐⭐⭐⭐ | 蚂蚁集团、字节跳动、所有安全敏感场景的公司
**答：**

2026 年，Agent 权限管理已经从「全有或全无」演进为「分层授权」。CallSphere 和 novelvista 的数据都强调 Agent 安全是 2026 年面试的「保底高频题」。

**四层数据访问控制模型：**

**第一层：身份验证（Who）**
- 确定当前用户身份和角色（普通用户 / 管理员 / 审计员）
- 不同角色有不同的数据可见范围

**第二层：数据分层（What）**

| 级别 | 数据类型 | 访问条件 |
|------|---------|---------|
| L0（公开） | 产品信息、FAQ、公告 | 无需认证 |
| L1（用户级） | 用户订单、账户信息 | 仅本人可查 |
| L2（商户级） | 商家后台数据 | 仅商家管理员 |
| L3（系统级） | 用户全量数据、系统配置 | 仅审计/系统管理员 |

**第三层：场景感知（When / Where）**
- 敏感操作需要「场景验证」：Agent 访问用户邮箱时，需要确认是「用户本人要求查看」
- 跨越上下文的访问需要告警：Agent 在查询订单上下文后突然访问用户手机号

**第四层：数据脱敏（How）**
```python
class DataAccessLayer:
    def get_user_info(self, user_id, requesting_agent):
        if requesting_agent.level == "customer_service":
            return {  # 脱敏返回
                "name": "张**",
                "phone": "138****5678",
                "email": "zh***@example.com"
            }
        elif requesting_agent.level == "audit":
            return full_data  # 审计员可查看完整信息
        else:
            return None  # 无权限
```

**关键设计模式——Permission Propagation（权限传播）：**
Agent 从用户处获取的权限不能无限传播给其他 Agent。当一个 Agent 调用另一个 Agent 时，权限有衰减规则：
- 同层调用：保留原权限
- 向下调用（用户 Agent → 数据 Agent）：子 Agent 只有执行权限，没有读取权限
- 向上调用（数据 Agent → 管理 Agent）：禁止，必须经过用户显式授权

**面试加分：** 「2026 年生产中一个经典教训是——不要相信 Agent 的『意图』。Agent 说『我需要用户手机号来发送验证码』，但它实际上可能把手机号传给了另一个不受信的工具。正确的做法是：在工具调用层面做数据范围控制，而不是在 Agent 推理层面做意图判断。」
